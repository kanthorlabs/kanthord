# How Epic 026 wires to the 019.4 provider engine

Recorded 2026-07-11 after Epic 019.4 shipped (multi-account provider engine).
Owning epic for the logic: **019.4**. Epic 026 **exposes these core operations
over the control-plane API — wiring only, no logic.** Do not reimplement CRUD,
login, resolution, or binding here; call the functions below and map their
inputs/outputs to your gRPC/RPC schema.

## The core functions to call (all in `src/agent/**` + `src/cli/**`)

| Operation | Core function | File |
| --- | --- | --- |
| Account CRUD | `createProviderAccountRegistry(...)` → `.add / .get / .list / .update / .remove` | `src/agent/provider-account-registry.ts` |
| Credential custody | `createProviderCredentialStore(...)` (pi-ai `CredentialStore`; `modify()` write path) | `src/agent/provider-credential-store.ts` |
| Observable login | `startLoginOperation(opts)` → `{ getState(), result }` (state: `device-code`→`pending`→`complete`/`failed`) | `src/agent/login-operation.ts` |
| Resolve a session | `buildProviderSession({ accountId, modelId }, deps)` → `{ model, streamFn }` | `src/agent/provider-session.ts` |
| Durable binding | `createAccountBindingStore(...)`, `resolveOrBindAccount(...)` | `src/agent/account-binding.ts` |
| OpenAI-compatible config | `createOpenAICompatibleConfigStore(...)` | `src/agent/openai-compatible.ts` |
| CLI reference impl | `runLoginCommand(...)` — mirror its arg→core mapping | `src/cli/login.ts` |

The CLI end-to-end test (`src/cli/provider-e2e.test.ts`) is a working template
of the full management surface (add×2 same-kind → list → resolve+bind → respawn
persistence → remove) with no network — copy its wiring shape.

## Mapping each RPC

- **AddAccount / GetAccount / ListAccounts / UpdateAccount / RemoveAccount** —
  thin pass-through to the registry methods. `ProviderAccount` is the response
  DTO; never expose `credentialKey`'s value, only its presence. `list({ kind })`
  supports the kind filter. Unknown id → the typed error the registry throws;
  map it to `NOT_FOUND`.
- **StartLogin (streaming/observable)** — call `startLoginOperation`; stream the
  observable state to the client (`device-code` carries the user code + URL;
  terminal states are `complete`/`failed`). On `complete` the `{type:"oauth"}`
  credential is already written to the store keyed by account id — you do not
  write it. Cancel/timeout writes nothing. Unknown kind → typed error →
  `INVALID_ARGUMENT`. This op is designed for exactly this render-and-poll use.
- **ResolveSession / RunAgainstAccount** — do **not** expose `streamFn` over the
  wire (it is an in-process function). Instead the daemon calls
  `resolveOrBindAccount` + `buildProviderSession` internally at spawn; the API
  only needs to accept the account/model selection and report the resulting
  binding. See "What 026/027 still owns" below.
- **Custody errors** — a store file wider than 0600 or wrong-owner throws
  `IdentityLoadError` (`code: "insecure-file-mode" | "wrong-owner"`). Surface the
  `code`, never the token. Logs already redact tokens to type tags.

## What 026/027 still owns (the ONE remaining wire)

019.4 threaded `model` + `streamFn` **end-to-end** through the spawn seam:
`PiSpawnOpts`/`PiRespawnOpts`/`FakePiSurface.spawnAgent` carry them,
`spawnPiSession`/`respawnPiSession` forward them, and `run-deps.ts`
`PiSurface.spawnAgent` passes them to `makeAgentOpts` (Agent gets `model` +
`streamFn`, **no** `getApiKey`). So the plumbing is done.

What is **not** wired (and is legitimately 026/027 + scheduler work, not logic to
invent): **choosing** which account a task runs on at the scheduler tick
(`src/daemon/run-loop.ts`), then calling `resolveOrBindAccount` +
`buildProviderSession` and passing the resulting `{model, streamFn}` into the
`spawnPiSession` opts. That is a **call-site wire**, not new mechanism: the
resolver picks the account (precedence: existing binding > slot > default), the
binding persists per task, and the session flows through the seam above.

## Boundaries

- **Switching** a running task between accounts is **Epic 043** (updates the
  binding 019.4 writes). 026 must not build switch logic.
- **Rich account-assignment precedence** (the 5-level chain) is **Epic 024**.
  019.4 gives only basic repo/slot→account selection.
- **Claude** is deferred (pi-ai `anthropic` vs Claude Agent SDK — open decision).
