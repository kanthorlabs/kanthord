# How Epic 027 (dashboard) wires to the 019.4 provider engine

Recorded 2026-07-11 after Epic 019.4 shipped. Owning epic for the logic:
**019.4**. Epic 027 **wires the dashboard UI to the core operations Epic 026
exposes over the API ÔÇö wiring only, no logic.** The dashboard reads/writes the
same account, login, resolution, and binding operations; it invents none of them.

## What the UI renders (all backed by 019.4 core via the 026 API)

- **Accounts list & CRUD** ÔÇö from the registry (`add/get/list/update/remove`,
  `src/agent/provider-account-registry.ts`). A `ProviderAccount` is
  `{ id, providerKind, label, defaultModel? }`. **Multiple accounts of the same
  kind are normal** (ten `openai-codex` accounts with distinct labels) ÔÇö render
  them by `label`, key rows by `id`, never by kind. Never render the credential
  value ÔÇö only "connected / not connected".
- **Login flow (device-code)** ÔÇö the observable login operation
  (`startLoginOperation`, `src/agent/login-operation.ts`) is built for a UI:
  its state moves `device-code` Ôćĺ `pending` Ôćĺ `complete` / `failed`. Render the
  `device-code` state's **user code + verification URL** (a copy button + an
  "open" link), poll while `pending`, show success/failure at the terminal
  state. On `complete` the credential is already stored server-side; the UI just
  reflects it. Cancel = the UI stops polling; nothing is written.
- **Account selection per run** ÔÇö the UI picks an account (and model) for a
  task; the daemon resolves + durably binds it. Show the **bound account** on the
  task so the operator sees which account a long-running task is pinned to
  across respawn/restart.
- **Custody / error states** ÔÇö surface the typed `IdentityLoadError` `code`
  (`insecure-file-mode`, `wrong-owner`) as an actionable custody warning; never
  show a token.

## Framing to keep visible in the UI copy

- The dashboard **does not switch** a running task's account ÔÇö that is **Epic
  043** (it updates the binding). If you show a "switch" affordance, it is a 043
  feature, gated there; 027 must not implement switch logic.
- Basic repo/slotÔćĺaccount selection only; the rich precedence chain is **Epic
  024**.
- **Claude** is deferred ÔÇö do not offer it as a provider kind yet.

## Do not reimplement

There is no UI-side logic to add for CRUD, login state machine, resolution, or
binding ÔÇö they are core (019.4) reached through the 026 API. The dashboard is a
**visualization over one schema** (per AGENTS.md). If a rendering need reveals a
missing API field, that is an Epic 026 wire, not new logic in 027.

See `.agent/plan/feedback/026-control-plane-api/019.4-provider-engine-wiring.md`
(the API mapping + the one remaining scheduler-tick wire) and
`docs/md/ai-providers.md` (the operator-facing description of every surface the
UI renders).
