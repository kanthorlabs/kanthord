# 019.17 Committer Identity Configuration

## Outcome

The daemon commits the agent's work with an explicitly configured git committer
identity, so delivery never fails with "Author identity unknown" and every commit
is attributed to the operator's chosen identity. The identity is resolved from a
**global default** (name + email, stored under the data root), optionally
**overridden per repo slot**. The `git.commit` broker verb applies it explicitly
(`git -c user.name=… -c user.email=… commit`), so it does not depend on ambient
container/host git config.

This is the **2A logic** for identity. The interactive "show the default identity
when a user adds a repo, let them confirm / change / keep it" experience is a
**2B/3** concern (control-plane UI, Epic 027 line) and is a non-goal here.

## Decision Anchors

- **Live gap (LP-A1, 2026-07-13):** the container had no git committer identity, so
  the daemon's commit failed `Author identity unknown` and delivery stalled. The
  `git.commit` adapter (`src/broker/verbs/git-local.ts` `makeCommitAdapter`) runs
  `git commit -m <msg>` with no author, relying on ambient config that a fresh
  checkout / sandbox does not have. Memory `lp-a1-delivery-gap` (B1).
- **Ulrich decision (2026-07-13):** apply the identity via **explicit `-c
  user.name=… -c user.email=…` on `git.commit`** (not by writing the checkout's
  local git config at bootstrap) — robust in any fresh checkout or container.
- **Ulrich decision (2026-07-13):** identity resolves **global default → per-slot
  override**; 2A builds the config + resolution + apply logic only, the confirm-on-add
  UI is 2B/3.
- Data-root convention: config lives under `KANTHORD_DATA` as JSON
  (`accounts.json`, `credentials.json` already there); `src/foundations/data-root.ts`
  owns the root. Memory `kanthord-data-root-rule`.
- `RepoSlot` (`src/slots/repo-slot.ts`) currently has `repo / strategy /
  maxConcurrentTasks / workflowsAllowed / identity` — it gains an optional
  `committer`.

## Stories

- `001-global-identity-store.md` - a loader/store for the global committer identity
  (name + email) under the data root; a missing config is a typed absence, not a
  crash. (foundation)
- `002-resolve-identity-slot-override.md` - the repo slot may carry an optional
  `committer` that overrides the global default; a resolver returns the effective
  identity (slot override → global default → typed "unconfigured"). (resolution)
- `003-commit-with-resolved-identity.md` - `git.commit` commits with the resolved
  identity via explicit `-c user.name/-c user.email`, so a commit is authored by it
  even with empty ambient config; an unconfigured identity does not silent-fail — it
  escalates. (delivery fix)

## Verification Gate

- `npm run typecheck` exits 0; `npm test` green — a test proves `git.commit` with an
  explicit resolved identity produces a commit authored/committed by that identity in
  a repo whose **ambient git config is empty**; a test proves resolution precedence
  (slot `committer` overrides the global default); a test proves an unconfigured
  identity yields an escalation (inbox item) and no bad/anonymous commit; existing
  `git-local` / `run-loop` / slot tests pass; guard green.
- **Live proof (LP-A1):** with a global identity configured and no ambient git
  config in the container, the daemon commits the agent's work authored by that
  identity, and delivery proceeds (supersedes the manual `git config` patch used in
  the 2026-07-13 run).

## Dependencies

- Epic 019.16 (delivery: the daemon commits + pushes + opens the PR). This epic
  hardens the commit step's identity so 019.16's manual environment patch is no
  longer needed.

## Non-Goals

- **No confirm-on-add-repo UI** — showing the default identity and letting the user
  confirm/change/keep is 2B/3 (Epic 027 line).
- **No per-commit author≠committer split**, no GPG/commit signing, no multiple named
  identities beyond global + per-slot override.
- **No change to the PAT / auth identity** (`slot.identity` keyring) — this is the
  git *committer* identity (name/email for `git commit`), a separate concern.
