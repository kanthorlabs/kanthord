# 019.5 Task Audit Timeline (narrow observability spine)

## Outcome

When a task fails, an operator can see **which step was the root cause** — a technical
error (tool/broker), an infra signal (rate-limit/quota/auth), a budget halt, a ring-1
block, a gate failure — and, now that tasks run on **multiple accounts**, **which
account/model served the failing step**. kanthord gains a **narrow** observability spine
(debate-hardened — not a grand distributed-tracing substrate): one durable, append-only
**task timeline** anchored on `task_id + attempt`, built by threading a shared
`correlation_id` through the records that already exist, plus the one missing data source
— pi session events — surfaced so per-model-call/account detail can be recorded.

Five pieces, in a deliberate order:

1. **Session-event exposure (the unlock).** pi's `Agent` already emits tool-call /
   message / usage / stop-reason / error events via `agent.subscribe`, but the daemon
   never sees them — `PiSessionHandle` exposes only `abort/waitForIdle/reset/
   contextTokens/stopReason`. This is the recorded "should be extensible but can't" gap
   (`.agent/plan/feedback/026-control-plane-api/session-events-and-signal-coverage.md`).
   Expose them as **one outbound, read-only** session-event stream through the
   `PiSessionHandle`/`PiSurface` seam.
2. **Durable task timeline.** An append-only `task_timeline_event` log anchored on
   `task_id + attempt` (session ids are child events — the anchor survives respawn), fed
   by a shared `correlation_id` threaded through attempt-evidence, interaction-capture,
   the broker ledger (already carries `correlation`), gate results, and spawn/respawn
   journal events. Normalizes what exists; it is **not** a new universal trace hierarchy.
3. **Per-model-call / account record.** On top of (1): each model call records
   `{task_id, attempt, session_id, account_id, model, tokens_in/out, cost, latency,
   stop_reason, typed_error}` — the multi-account "which account/step failed" answer.
4. **Two-tier failure attribution (honest about precision).** A machine-derived
   `observed_failure_signal` (factual, narrow) + an optional human/reviewer-confirmed
   `suspected_root_cause` + `root_cause_confidence`. Prompt-issue / model-behaved-weird
   are **not** auto-classified (PRD: categories are "coarse, approximate, human-
   confirmed"). Includes completing the escalation `SIGNAL_MAP` (026 Input 2) so every
   known signal carries a proposed type.
5. **Query interface (core) + CLI.** A core `queryTaskTimeline(taskId)` function plus a
   minimal CLI to read a task's timeline and filter by failure signal — the seam Epic 026
   exposes over the API (`ListTaskTimeline`) and Epic 027 renders; **026/027 wire only,
   no logic** (per the 019.4 precedent).

The automated gate is **hermetic** (fakes/doubles, no network). The typed provider-error
taxonomy defined here (`rate_limited | quota_exhausted | auth_failed | transient |
fatal`) is the **shared prerequisite** Epic 043's switch also needs.

## Decision Anchors

- **Ulrich, this conversation (2026-07-11)** — multi-account makes tracing important;
  the audit must let us know **which step is the root cause** of a task failure; it must
  support a **UI-integration interface later**; a **small dedicated epic** is approved to
  keep it clear.
- **Audit + debate finding (this conversation)** — the current audit is **siloed,
  purpose-built stores** (`attempt_ledger` = counter; `attempt_evidence` = bounded
  free-text; `interaction-capture` = typed interactions + cost; `budget_ledger` =
  model-independent charge; `op_ledger` = broker ops with a `correlation` field), with
  **no correlated timeline, no per-model-call/account record, and no read/query surface**
  (`GetStatus/ListInboxItems/Respond*` only). The debate-corrected cure is **narrow**:
  correlation-id + one events table over existing records, anchored on `task_id +
  attempt`; per-call logging gated on session-event exposure; failure attribution split
  into machine-fact vs human-confirmed.
- **Recorded 026 design input** (`.agent/plan/feedback/026-control-plane-api/session-
  events-and-signal-coverage.md`, 2026-07-10) — pi's Agent emits events via
  `agent.subscribe`; the daemon does not surface them; this blocks the UI timeline +
  audit depth. Input 2: `SIGNAL_MAP` covers 2 of ~6 architecture signals. This epic
  builds both; 026 exposes the read stream/API.
- **PRD (categories are coarse/approximate/human-confirmed)** — machine attribution must
  not pretend to auto-classify prompt/model-behavior failures (debate finding). Hence the
  `observed_failure_signal` vs `suspected_root_cause` split.
- **Epic 019.2 seam** — `PiSurface.spawnAgent` → `spawnPiSession`
  (`src/agent/pi-session.ts`, which already journals `session_spawned`/`session_respawned`)
  → the daemon. The session-event stream is added at this seam.
- **Epic 019.4** — provides the `account_id` per run (durable binding) that the per-call
  record and timeline attribute to. Hard order: 019.4's account identity exists first.
- **Epic 019.3** — `attempt_evidence` / `attempt_ledger` keyed by `task_id + attempt`;
  the timeline reuses that anchor and threads the correlation id through the evidence.
- **Epic 005 / 017** — broker `op_ledger` (has `correlation`) and `interaction-capture`
  (`SIGNAL_MAP`, cost) are threaded, not replaced.
- **Epic 043 (to be authored)** — shares the typed provider-error taxonomy defined here;
  a switch must be auditable (this epic makes it so).
- **Authoring Spike Gate** — pi's `agent.subscribe` event surface (what fields/usage/
  account are actually emitted per call) is a **pinned-dependency real-surface** unknown;
  Story 001 is spike-gated to confirm it before the timeline depends on it.

## Stories

- `001-session-event-exposure.md` — expose pi's `agent.subscribe` events as one
  outbound, read-only stream through `PiSessionHandle`/`PiSurface` (tool-call, message,
  usage, stop-reason, error). Spike-gated on the real pi event surface. The unlock for
  per-call detail; inbound hooks stay daemon-owned (no ring-1 extension points).
- `002-task-timeline-log.md` — the durable append-only `task_timeline_event` log
  anchored on `task_id + attempt`; a shared `correlation_id` threaded through
  attempt-evidence, interaction-capture, broker ledger, gate results, and spawn/respawn
  events, so one ordered timeline reconstructs from durable records.
- `003-per-model-call-record.md` — a per-model-call record `{task_id, attempt,
  session_id, account_id, model, tokens, cost, latency, stop_reason, typed_error}` fed by
  Story 001; defines the typed provider-error taxonomy (shared with Epic 043).
- `004-failure-attribution.md` — `observed_failure_signal` (machine, factual, from
  events) + optional human-confirmed `suspected_root_cause` + `root_cause_confidence`;
  complete the escalation `SIGNAL_MAP` so every known signal carries a proposed type.
- `005-timeline-query-and-references.md` — core `queryTaskTimeline(taskId)` +
  minimal CLI (read a task's timeline, filter by signal); operator doc; reference notes
  that Epic 026 exposes `ListTaskTimeline` over the API and Epic 027 renders it (wiring
  only).

## Verification Gate

- `npm run typecheck` exits 0; `npm test` green — new session-event, timeline, per-call,
  attribution, and query suites plus all pre-existing suites, no regression. Zero-network
  guard stays green (session events + model calls run on doubles).
- **Session events:** a spawned session (double) emitting tool-call/message/usage/error
  events surfaces them through `PiSessionHandle` as one ordered read-only stream; the
  daemon records them; no inbound extension point is exposed.
- **Timeline reconstructs:** given a scripted task run (dispatch → spawn → tool call →
  broker op → gate fail → escalation → respawn), `queryTaskTimeline(taskId)` returns one
  ordered timeline whose events all share the task's `correlation_id`, spanning attempt,
  session (child), broker op, and gate/interaction records — reconstructed from **durable
  state only** (the LP7 "audit reconstruction" drill, now automated).
- **Root-cause step is identifiable:** for each of the machine signals
  (`rate_limited / quota_exhausted / auth_failed / tool_blocked / budget_breach /
  broker_failed / gate_failed`), a scripted failure produces a timeline whose failing
  event carries that `observed_failure_signal` and, for a model-call failure, the
  `account_id` + `model` that served it.
- **Honest attribution:** a prompt/model-behavior failure is **not** auto-classified —
  it records a factual signal (e.g. `gate_failed`) with `suspected_root_cause` left for
  human confirmation; a test asserts no machine code writes `suspected_root_cause`.
- **Multi-account:** two attempts of one task on two different accounts produce timeline
  events each attributed to the correct `account_id`.
- **CLI end-to-end (hermetic):** the minimal CLI reads a task timeline and filters to the
  failing step, without network.
- **Gate closes when** the above are green and the reconstruction drill passes on durable
  records alone.

## Non-Goals

- **No new control-plane API or UI.** This epic owns the observability **logic** (event
  exposure, timeline, per-call record, attribution, query function + CLI). Epic 026
  exposes `ListTaskTimeline` / the session-event stream over the API; Epic 027 renders
  the timeline — **wiring only, no logic** (019.4 precedent).
- **No auto-classification of prompt/model-behavior root cause.** Machine records factual
  signals only; interpretive causes are human/reviewer-confirmed (PRD posture).
- **No grand distributed-tracing substrate.** Narrow: correlation-id + one events table
  over existing records, anchored on `task_id + attempt`; session ids are child events,
  not the anchor (debate finding — a session-centered spine fragments across respawn).
- **No inbound extension points.** The session-event stream is **outbound, read-only**;
  ring-1 hooks stay daemon-owned (026 feedback).
- **No switch orchestration.** Epic 043 owns switching; this epic only makes it
  auditable and defines the shared typed provider-error taxonomy.
- **No real network/model call in the automated gate.** Hermetic doubles only; the real
  per-call usage/error surface is spike-confirmed (Story 001) and re-verified by the
  019.4 maintainer live proof.
