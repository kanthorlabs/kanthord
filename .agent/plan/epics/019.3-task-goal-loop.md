# 019.3 Task Goal Loop — run until a terminating condition is met

## Outcome

The corrective outer loop that makes a task **goal-seeking instead of
dispatch-once**: after a session completes, the run-loop evaluates the task's
exit gate; on `fail` the task returns to a dispatchable state with the gate's
failure evidence durably recorded and injected into the next spawn brief; the
cycle repeats until one terminating condition is met — **goal reached** (gate
`pass`), **needs-human** (gate escalation), **max attempts** (durable,
respawn-proof counter), **budget ceiling** (already built, precedence
asserted), or **operator halt** (already built, precedence asserted). Without
this loop an agent gets exactly one session-lifetime per task and every
imperfect run costs a human touch; with it, escalation returns to meaning
"a real problem", and the only cost-bound brake (budget) stops being the
accidental retry limit.

The loop is built **hermetically against the existing seams**: the Epic 006
fake workflow produces the gate outcomes, the pi-session double produces the
sessions, the Epic 019.2 run-loop tick is the caller. Epic 024's real
`tdd@1` gates later plug into the same contract without changes here. The
retry loop is a **new, separate path from lifecycle respawn**: threshold /
task-boundary / crash respawns continue mid-attempt and must satisfy
respawn-equivalence unchanged; a retry starts a NEW attempt from a fresh
brief. One attempt = one dispatch → session → gate-check cycle; **the
attempt ledger increments when an attempt is dispatched** (debate finding
2026-07-10 — counting only failures made "attempts" lie to the operator: a
first-try pass read 0; per-dispatch counting reads naturally and still never
counts a lifecycle respawn, which continues the same attempt).

This epic carries **one named contract change** (debate finding 2026-07-10 —
the bare `GateOutcome` enum cannot produce the evidence the retry depends
on): the Epic 006 workflow seam's `gateCheck` returns a structured
`GateResult { outcome, summary? }` — outcome as today, plus an optional
bounded failure summary — and the gate-result sink records it. Story 001
owns the change; the fake workflow scripts summaries; Epic 024's real gates
later fill `summary` from the parsed test-runner result (its epic already
mandates parsing the runner output). Two lifecycle rules complete the loop's
honesty (debate findings 2026-07-10): (a) **session-end classification
precedes gate-checking** — only a cleanly completed session is gate-checked;
an aborted or errored session (pi exposes `stopReason: "aborted" | "error"`
and `state.errorMessage` — verified in pi-agent-core `agent.ts`) routes to
the existing lifecycle/crash path, never to the gate; (b) **post-session
facts are always recorded before any verdict** — gate result, evidence, and
ledger land durably first; budget and operator halt are enforced at the
existing PRE-SPAWN gates (Epic 019.2), so a budget park can never lose the
reason the last attempt failed.

## Decision Anchors

- `.agent/plan/feedback/019.3-task-goal-loop/goal-loop-and-termination-review.md`
  (2026-07-10, B1/B2/S1) — the source findings: L3 is dispatch-once; the
  attempt loop is unowned in plan and code; budget must not be the accidental
  retry limit.
- **PRD §10 workflow interface** — `gateCheck(phase) → pass|fail|needs_human`
  is the goal signal; gates are per-shape handlers. This epic consumes the
  outcome; it does not add gate vocabulary.
- **PRD §3.2 + Epic 006 respawn-equivalence** — lifecycle respawns restore
  *equivalent* state mid-attempt; they never iterate toward a goal. The
  attempt counter therefore increments per dispatched attempt, never per
  respawn, and the ledger survives respawn/crash (same discipline as the
  PRD §4 budget ledger: "the per-task ceiling accumulates across
  compaction/crash respawns — a respawn cannot reset the breaker").
  **The brake on repeated pre-gate session death** (a session that crashes
  before its gate can ever be checked, so the attempt never ends) is the
  PRD §4 budget breaker — its cost AND wall-clock ceilings accumulate
  across the respawns (debate finding 2026-07-10 — named explicitly so
  `max_attempts` is not mistaken for a general crash-loop brake; if 019.2's
  respawn path turns out to lack the wall-clock wiring, that is a Findings
  Out entry, not silent scope growth here).
- **PRD §4 ring-1 precedence** — budget reservation gates BEFORE each spawn
  and operator halt gates dispatch; both already exist (Epics 007/013,
  017, 019.2) and outrank the retry decision. This epic asserts the order,
  it re-implements nothing.
- **PRD §8 model-policy resolution chain** — the hierarchy pattern for
  condition knobs. MVP scope here: `max_attempts` = task frontmatter
  override → system default. The feature/repo-slot tiers join when Epic 024
  Story 003 builds the real chain (B2 — stated join point, not pre-built).
- **MVP goal signal (S1, sharpened per debate 2026-07-10):** deterministic
  gates ARE the MVP goal signal (fake here, Epic 024's real test gates
  later); human review (`escalate_all_diffs`, `needs_human`) is the
  judgment **fallback** on top of them, not itself a gate. An LLM
  `gate-check` handler (the PRD §8 `gate-check` role) is the Phase-3
  automation of that judgment and is NOT built here.
- **Epic 019.2 run-loop** — the tick is the loop's caller; `spawnPiSession`
  assembles the brief in documented order (task body, epic body, RUNBOOK,
  STATE, AGENTS.md) — the failure evidence becomes one more documented brief
  element, not a mutation of workflow-owned STATE.

## Stories

- `001-gate-outcome-consumption.md` — the named `GateResult` contract change
  on the Epic 006 workflow seam (outcome + bounded failure summary), plus:
  the tick classifies session end (clean / aborted / errored) and, only on
  clean completion, evaluates the exit gate and routes the outcome: `pass` →
  exit gate marked passed (existing completion path unchanged);
  `needs_human` → task parked + escalation inbox item; `fail` → the
  corrective path (Story 002/003).
- `002-fail-evidence-and-redispatch.md` — on `fail`, the gate's failure
  evidence is durably recorded (bounded), the task returns to a
  dispatchable state, and the NEXT spawn brief carries the evidence as a
  documented brief element; a task with no evidence spawns with an
  unchanged brief.
- `003-attempt-ledger-and-termination.md` — the durable attempt ledger
  (increments per dispatched attempt, survives respawn and daemon restart)
  and the ordered termination decision: attempts exhausted → park + typed
  `attempts-exhausted` escalation item with two distinct operator actions
  (`retry-once` vs `re-arm`); budget and operator halt outrank retry
  (asserted, not re-built).
- `004-max-attempts-resolution.md` — `max_attempts` resolves task
  frontmatter override → system default (fixed default: **3**); invalid
  values fail with a typed planner-vocabulary diagnostic at compile/lint,
  never silently at runtime.

## Verification Gate

- `npm run typecheck` exits 0; `npm test` green — new suites plus all
  pre-existing suites (Epic 019.2 baseline), no regression.
- **Loop end-to-end (hermetic):** a task whose fake exit gate fails twice
  then passes reaches `complete` in three attempts; each retry's spawn brief
  contains the prior failure evidence (the `GateResult` summary); the
  attempt ledger reads 3 dispatched attempts at completion.
- **Termination end-to-end:** with `max_attempts: 2` and an always-failing
  gate, the task parks after attempt 2 with an `attempts-exhausted` inbox
  item and NO third spawn; a `needs_human` outcome parks with its own item
  and no retry; a mid-attempt lifecycle respawn does not increment the
  ledger (respawn-equivalence suite stays green); an aborted/errored
  session is NOT gate-checked and routes to the lifecycle path.
- **Operator actions:** on an `attempts-exhausted` item, `retry-once`
  grants exactly one more attempt (next fail re-parks immediately) and
  `re-arm` resets the counter — both recorded as interactions, asserted
  distinct.
- **Precedence:** with the budget ceiling breached, no retry spawn happens
  regardless of remaining attempts (budget outranks retry); a halted task is
  not re-dispatched by the loop.
- **Resolution:** a task frontmatter `max_attempts` overrides the system
  default; absent frontmatter uses 3; an invalid value produces the typed
  diagnostic.
