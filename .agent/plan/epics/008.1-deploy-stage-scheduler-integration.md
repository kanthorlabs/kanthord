# 008.1 Deploy-Stage Scheduler Integration (A-lite)

## Outcome

Deploy-stage DAG nodes become **first-class dispatchable scheduler nodes** that flow
through the **existing** dispatch lifecycle (`pollOnce`: generation/gate filter →
lease acquire → `pending`→`running` → generation pin → execute), so the scheduler
continues past PR-open into them instead of the deploy executor being called
standalone. A deploy stage is dispatchable **only** when every predecessor node's
exit gate has passed (the PR-open boundary); when the scheduler dispatches a
deploy-stage node it invokes the deploy executor for **that one stage**; a passing
stage marks the node's exit gate so the next deploy stage becomes dispatchable; a
failing stage resolves `halt_and_escalate` and does **not** pass its gate, so
downstream deploy stages never run. This closes Epic 008 review blocker **B2**
(`scheduler-driven-invocation-not-asserted`) by making the invocation a
scheduler-owned transition through the real dispatch lifecycle — asserted against
scheduler semantics, not a bare call. Merge / deploy / rollback stay **human**
verbs; the scheduler advances only the observe/gate DAG.

## Decision Anchors

- **Debate finding (2026-07-05, opencode/plan)** — chose **A-lite** (minimal
  first-class integration: deploy-stage nodes enter the scheduler's dispatchable set
  and execute through the existing `pollOnce` lifecycle) over Solution B (a helper
  that scans + calls the executor near the scheduler — a semantic shortcut that
  leaves the nodes un-dispatched) and full Solution A (per-stage lease/parking
  rewrite). The node must be **dispatched through the real lifecycle**, asserted via
  scheduler semantics.
- **Debate finding (2026-07-05) — story→deploy gating model.** The scheduler gates
  on `kind='task'` predecessors, but the compiler wires the first deploy stage from
  a `kind='story'` node. Rather than hide an aggregate "story is done when its tasks
  are done" rule inside readiness SQL (the fragile option), the **compiler emits
  `terminal-task → first-deploy` edges** so deploy gating uses the *identical*
  task-predecessor rule; the scheduler predecessor join is generalized only to
  treat a `deploy-stage` predecessor the same way (for stage→stage chaining). One
  uniform gate rule, no structural-node aggregate in SQL.
- **Epic 008 review blocker B2** (`.agent/tdd/history/2026-07-04-008-deploy-chain-executor.md`)
  — scheduler-driven invocation of the deploy executor is asserted nowhere.
- **Epic 008 Story 001 AC + Verification Gate** — "the executor is invoked by the
  scheduler continuing past PR-open into those DAG nodes … a scheduler-owned
  transition — asserted, not assumed."
- **PRD §7.4** — the DAG continues past "PR open" into per-repo deploy stages;
  `on_pass: notify_human` (merge stays human); `on_fail: halt_and_escalate` with
  evidence; cross-repo rollback is human.

## Stories

- `001-deploy-stages-are-schedulable.md` — the compiler wires the first deploy
  stage from the last-major story's terminal task node(s), and the scheduler loads
  and gates deploy-stage nodes as first-class DAG nodes under one uniform
  task/deploy predecessor-gate rule (including the generation/dirty guards); a
  deploy stage does not become ready before PR-open; task dispatch is unchanged.
- `002-scheduler-invokes-deploy-executor.md` — the existing `pollOnce` dispatch
  lifecycle owns deploy-stage execution: dispatching a deploy-stage node invokes the
  deploy executor for that one stage (`runChain` delegates to the same per-node
  primitive); a passing stage marks its exit gate and unblocks the next stage; a
  failing stage (handler **or** soak) halts, leaves the gate unpassed, and
  downstream stages never run; the last stage's passed gate is the chain's
  completion; no merge/deploy/rollback verb is ever called.

## Verification Gate

- `npm run typecheck` exits 0; `npm test` green for all Story suites.
- **Closes B2:** a compiled plan's deploy-stage nodes are dispatched by the
  scheduler continuing past PR-open — asserted against the **real `pollOnce`
  dispatch lifecycle** (a deploy-stage node enters the dispatchable set, is leased,
  goes `pending`→`running`, and is executed through the scheduler), not a bare
  `runChain`/`runStage` call.
- **Uniform gating, no structural-node aggregate:** the first deploy stage is
  gated by task predecessors (compiler-emitted terminal-task→deploy edges) and each
  later stage by its predecessor deploy stage, under one generalized predecessor-gate
  rule; a deploy stage is not ready before all predecessors have `exit_gate_passed=1`.
- **Regression:** existing task-node dispatch, gating, leases, and generation
  behavior are unchanged (a deploy-free plan schedules identically).
- **Generation isolation:** deploy-stage dispatch respects the same
  generation/dirty-plan guards as tasks — a stale-generation deploy node is not
  dispatched, and its dispatched generation is pinned.
- **Pass advances / fail blocks:** a passing stage marks its exit gate and only then
  does the next deploy stage become dispatchable; a failing stage resolves
  `halt_and_escalate` with evidence, does not pass its gate, and downstream deploy
  nodes are never dispatched or executed. A passed stage and a running stage are
  never re-executed within a poll cycle.
- **Completion:** the last deploy stage's passed exit gate is the observable
  completion of the deploy chain — no further deploy node dispatches after it.
- **No auto-merge (non-vacuous):** across the whole run the fake broker's recorded
  command log shows no merge/deploy/rollback verb, and a passing stage still emits a
  `notify_human` event with the stage context (asserted against the recorded log,
  not a vacuous absence) (PRD §7.4, §9).

## Non-Goals

- **No durable parking / resume beyond the existing scheduler lease semantics**
  (full Solution A). Deploy-stage nodes **do** participate in the existing
  lease/acquire/status/generation lifecycle (that is what "first-class" means);
  what is deferred is durable soak-state parking and crash-resume across respawn.
- **No real observers, no network, no auto-merge/deploy/rollback** — unchanged from
  Epic 008; those terminal verbs stay human (PRD §7.4, §9; Trade-off #14).
- **No durable soak/dispatch state across crash/respawn** — Epic 009's concern.
  This epic tests **synchronous `pollOnce` passes** only; it must not grow into
  run-loop, parking, or crash-recovery semantics (debate finding — keep narrow).
- **No rename of `scheduler_task`** even though it now also holds deploy-stage rows
  (naming debt acknowledged; a rename is a separate follow-up).
