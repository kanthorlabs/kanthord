## Group 1 тАФ Landing-candidate observability & recovery ergonomics

Root cause: the landing-candidate lifecycle (`pending | conflict | landed`) is
first-class internal state, but the driving CLI barely exposes it, and the only
recovery entry point (`retry`) is gated behind a state you must provoke by hand.

### F1 тАФ no CLI surface for a task's candidate lifecycle state тАФ **major**

`get task --id <id> --json` returns
`id,title,status,agent,objectiveId,dependencies,note,instructions,ac,verification,result,dependencyStatus,context`
тАФ **no** candidate/landing field. `get conflict` shows state only while a task
is in conflict. So from the CLI alone you cannot tell whether a `completed`
task's candidate actually landed, or whether an `awaiting_confirmation` task's
candidate is built on a stale base. The whole approve/retry loop in this run was
only navigable via the `e2e-status.sh` helper, which reads the
`landing_candidates` table directly.

- Repro: `node src/main.ts get task --id <landed-task> --json` тЖТ no `landed`
  anywhere; compare with `e2e-status.sh <init>` which shows
  `candidate:landed (тАж->тАж)`.
- Suggestion: add the candidate state + base/candidate SHAs to `get task`
  output, and/or a `get candidate --task <id>` (or `list candidate
--initiative <id>`) command. The operator should never need raw SQL to run
  the documented approve/retry loop.

### F2 тАФ `retry` is rejected from `awaiting_confirmation`; you must force a doomed `approve` first тАФ **major** (UX)

A sibling whose candidate is built on a now-stale base sits at
`awaiting_confirmation` with `candidate:pending`. `retry task --id <id>` refuses
it: `error: task тАж is not retryable (status: awaiting_confirmation)`. To rebuild
it you must first run an `approve` that is _known_ to fail, purely to move the
candidate into `conflict`, after which `retry` is accepted.

- Repro (from this run): List Tasks candidate on base `7160c85` while tip was
  `094f70a`; `retry` тЖТ "not retryable"; `approve` тЖТ conflict; `retry` тЖТ accepted.
- Suggestion: allow `retry` from `awaiting_confirmation` (treat a stale-base
  pending candidate the same as a conflicted one тАФ requeue for rebuild), or add
  an explicit `rebuild task --id <id>` that re-runs on the current base without
  the manufactured conflict.

### F3 тАФ N siblings editing one file serialize into N full real-model rebuilds тАФ **minor** (known cost)

The 4 read/update/delete tasks each edit `src/todo.mjs`. Each of the last three
required approveтЖТconflictтЖТ`retry`тЖТ**rebuild (real model)**тЖТapprove тАФ 3 extra
model rebuilds for non-overlapping edits. Compounds F2. This is the documented
"sibling serialization" cost, not a correctness bug, but it multiplies real
tokens/latency for a common shape (many endpoints, one file).

- Suggestion (design, low priority): attempt an automatic non-model rebase for
  trivially non-overlapping hunks before demanding a model rebuild; or
  auto-`retry` on approve-time conflict so the operator issues one command.

## Group 2 тАФ CLI interface consistency (scripting-hostile surface)

Root cause: output/help conventions are per-command rather than uniform, so a
script that works against one subcommand breaks against its sibling.

### F4 тАФ `list event --json` emits JSONL, but `list task --json` emits a JSON array тАФ **major**

`list task --json` тЖТ a single `[ {...}, {...} ]` array (parseable with one
`JSON.parse`). `list event --json` тЖТ newline-delimited objects
(`{...}\n{...}\nтАж`), which throws `SyntaxError: Unexpected non-whitespace
character after JSON` on `JSON.parse`. Same flag name, two contracts.

- Repro: `node src/main.ts list event --after 0 --limit 5 --json | head -c2`
  prints `{"` (object, not `[`); piping the whole output to `JSON.parse` throws.
- Suggestion: make `--json` mean one parseable JSON document everywhere (emit an
  array for `list event` too), OR keep the event feed as a stream but expose it
  under an explicit `--jsonl` / `--stream` flag and document it. Whatever the
  choice, `--json` must be uniform across `list *`.

### F5 тАФ nested-subcommand `--help` prints ROOT help for most groups тАФ **minor**

`create ai-provider --help`, `get conflict --help`, `run daemon --help`, and
`retry task --help` all print `Usage: kanthord [options] [command]` (root help)
instead of the subcommand's help. The `<group> help <sub>` form works
(`get help conflict`). Notably `login provider --help` DOES print correct sub
help тАФ so the fix exists in one place and is just not applied to the other
command groups.

- Repro: `node src/main.ts get conflict --help` тЖТ root help;
  `node src/main.ts login provider --help` тЖТ correct `kanthord login provider`
  help.
- Suggestion: wire `--help` on nested subcommands the same way `login provider`
  does, across all command groups.

### F6 тАФ `login provider` prints a bare id, not the unified `<kind> created: <id>` line тАФ **minor**

Every `create тАж` command now prints `"<kind> created: <id>"` (007.9 Story C),
but `login provider` (which also creates a credential resource) prints only the
bare ULID on stdout. Minor output-convention drift for a resource-creating
command.

- Repro: `login.log` tail is `01KY4YC8DQQM6W12N90Z4MYH06` with no
  `credential created:` prefix.
- Suggestion: emit `credential created: <id>` (id still on stdout) from
  `login provider`, matching `create credential`.
