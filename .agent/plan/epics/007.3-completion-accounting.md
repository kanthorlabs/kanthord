# EPIC 007.3 — completion accounting & real landing (E2E re-run fixes)

> **Coupled bug-fix epic (Ulrich, 2026-07-20).** Two defects surfaced by the
> post-007.2 TODO-API E2E (full findings record in the Appendix below): **F2**
> (token/cost accounting is a hardcoded `0`) and **F3** (a _changed_ `generic@1`
> task `completed`s without ever landing to the canonical branch). F1 (browser
> OAuth) was dismissed as a machine-side non-defect. **F4** (`login` /
> `create project` reshape) was pulled OUT of 007.3 (Ulrich, 2026-07-20): it is
> not CLI-only — it needs a resource-management refactor and gets its own epic,
> authored later. Design directions are debate-hardened (opencode, read-only,
> 2026-07-20).
>
> **Load-bearing principle (F3):** for a same-repo dependency, `completed` MUST
> mean the accepted change is on the canonical branch the dependent clones.
> Flipping the status is not enough — the change must actually land. The
> existing manual `land repository` command + the `approve-task` landing branch
> are NOT production-ready (11 cited defects); this epic makes C2's invariant
> true, reusing the landing port/adapter/tables and the C2 per-repo lock.
>
> **Reuse-pi-first (F2):** real token counts come from pi's own `Usage` buckets
> and pi's session-statistics aggregation — not a hand-rolled interpretation.

## Goal

After 007.3, a real task run through the CLI reports **truthful accounting** and
reaches **truthful completion**. (1) **Accounting** — `agent.finished` and a
dedicated daemon stdout line carry real `turns`, `tokensIn`, and `tokensOut`
summed from pi's per-turn usage (including failed/escalated runs), so the owner
can see quota/cost; the safe-facts export already downstream of these fields now
carries real numbers. (2) **Landing** — a _changed_ task produces a durable
**candidate** and stops at `awaiting_confirmation`; `approve task` **lands the
candidate onto the repository's configured canonical branch** (correct home,
correct target, shared per-repo lock, durable SHA + crash-idempotent recovery),
and only a successful/reconciled land transitions the task to `completed` and
releases dependents — so a same-repo dependent clones the canonical branch and
**sees the prior work** (the C2 non-merge bug, closed for `generic@1`).

## Verification Gate

Gates: `npm run verify` (typecheck + test + verify:handoff + lint + db status).
Hermetic — no network. The suite proves each fix with fakes/real-git-in-temp:

- **F2:** a fake session drives a **multi-turn** usage fixture with non-zero
  `input`, `cacheRead`, `cacheWrite`, `output`, and `reasoning`; tests assert
  `tokensIn = Σ(input+cacheRead+cacheWrite)`, `tokensOut = Σ(output)`, each
  bucket counted exactly once, `reasoning` NOT double-counted, and that usage is
  still aggregated for a run that fails/escalates mid-way. The daemon emits a
  separate `agent finished: turns/tokensIn/tokensOut` line (not only on
  completion).
- **F3:** the runner result contract (`completed`=no-change / `candidate` /
  `failed` / `escalated`) is exercised against a fake agent; `RunNextTask`
  persists candidate metadata + `awaiting_confirmation` **atomically** (a
  simulated crash between transition and candidate-write cannot leave a
  candidate-less `awaiting_confirmation`); `ApproveTask` lands via
  `RepositoryLanding.land` against **real git in temp dirs** onto the
  repository's configured branch (ff + merge + typed conflict → `task.conflict`
  with the task still awaiting, + crash-idempotent re-approve, + cross-process
  lock contention); candidate ids identify the execution attempt and reject an
  immutable mismatch; a no-change task `completed`s directly. Composition wires
  `RepositoryLanding` into the real `ApproveTask` and the shared landing lock
  into `LocalWorkspaceManager` (guarded by an architecture/wiring test).

Proof: two parts. **Part A** is deterministic — pure CLI + real git in temp
dirs, NO model — and proves the full landing lifecycle end to end. **Part B** is
a guarded live-provider smoke that runs one real task and asserts real token
accounting + a landed canonical branch. `set -euo pipefail` makes every step
assert; expected failures use `if ! …; then`.

```bash
set -euo pipefail
export KANTHORD_DB="$(mktemp -d)/kanthord.db"
node src/main.ts db migrate
PROJECT=$(node src/main.ts create project --name demo)

# ===================================================================
# Part A — deterministic: candidate lifecycle + real landing (no model)
# ===================================================================
# A local bare "home" repo with one commit on the configured branch.
HOME_REMOTE="$(mktemp -d)/home.git"; git init -q --bare -b main "$HOME_REMOTE"
SEED="$(mktemp -d)/seed"; git clone -q "$HOME_REMOTE" "$SEED"
git -C "$SEED" -c user.email=a@b.c -c user.name=t commit -q --allow-empty -m init
git -C "$SEED" push -q origin main
BASE_SHA=$(git -C "$SEED" rev-parse HEAD)

REPO=$(node src/main.ts create repository --project "$PROJECT" --name home \
        --remote-url "file://$HOME_REMOTE" --branch main --auth ambient)

# Import a single-task graph bound to the repo (fake/echo executor — no model),
# author a task whose changed output is deterministic, then run the daemon.
#   (kanthord-verify-style graph package with bindings: source=$REPO)
GRAPH="$(mktemp -d)/g"; scripts/e2e/make-landing-graph.sh "$GRAPH"   # helper writes the package
INIT=$(node src/main.ts import graph "$GRAPH" --create --project "$PROJECT" --bind source="$REPO")
TASK=$(node src/main.ts find task --initiative "$INIT" --name land-me)

node src/main.ts run daemon --until-idle --poll-interval 200

# The CHANGED task must NOT be completed yet — it must await confirmation with a candidate.
test "$(node src/main.ts get task --id "$TASK" --json | node -e 'process.stdin.once("data",d=>console.log(JSON.parse(d).status))')" = "awaiting_confirmation"
# main must still be at BASE_SHA (nothing landed before approval).
git -C "$SEED" fetch -q origin; test "$(git -C "$SEED" rev-parse origin/main)" = "$BASE_SHA"

# Approve -> land onto the configured branch; task becomes completed.
node src/main.ts approve task --id "$TASK"
test "$(node src/main.ts get task --id "$TASK" --json | node -e 'process.stdin.once("data",d=>console.log(JSON.parse(d).status))')" = "completed"
# The canonical branch has ADVANCED and now contains the task output.
git -C "$SEED" fetch -q origin
NEW_SHA=$(git -C "$SEED" rev-parse origin/main); test "$NEW_SHA" != "$BASE_SHA"
git -C "$SEED" cat-file -e "origin/main:$(cat "$GRAPH"/.expected-output-path)"   # landed file exists on main
# base_commit is now recorded (A7): get task --result shows the base SHA, not null.
node src/main.ts get task --id "$TASK" --result | grep -qi "base"

# ===================================================================
# Part B — guarded live smoke: real tokens + landed branch (real model)
# ===================================================================
if [ "${KANTHORD_E2E_LIVE:-0}" = "1" ]; then
  # (resources: openai-codex / gpt-5.6-terra credential + ai-provider, private repo)
  # After one real run to completion+approval:
  node src/main.ts list event --after 0 --json \
    | grep '"agent.finished"' \
    | node -e 'let s="";process.stdin.on("data",d=>s+=d).on("end",()=>{const e=JSON.parse(s.trim().split("\n").pop());const p=e.payload;if(+p.tokensIn<=0||+p.tokensOut<=0){console.error("FAIL: tokens still zero",p);process.exit(1)}})'
  # daemon stdout carried an `agent finished: turns=… tokensIn=… tokensOut=…` line (captured to a log).
  grep -qE 'agent finished: turns=[0-9]+ tokensIn=[1-9][0-9]* tokensOut=[1-9][0-9]*' "$DAEMON_LOG"
fi
echo "007.3 PROOF OK"
```

## Stories

1. **Real token accounting in the pi runner** — aggregate `Usage` per assistant
   turn (`input+cacheRead+cacheWrite` → tokensIn, `output` → tokensOut, no
   reasoning double-count), mirroring pi's session-statistics summation; emit
   real values on `agent.finished` for completed/failed/escalated runs. Replace
   the hardcoded `"0"`. Multi-turn exact-arithmetic tests. (F2)
2. **Daemon accounting stdout line** — emit a dedicated
   `agent finished: turns/tokensIn/tokensOut` line from the emit callback
   (decoupled from the task-completion line). (F2)
3. **Executor-neutral candidate result contract** — add the `candidate` outcome
   (`{workspace, branch, baseCommit, candidateCommit, summary, evidence}`);
   `generic@1` changed → `candidate`, verified no-change → `completed`; no
   inference from `task.agent`; adjust `genericProfile.verify()` no-change
   contract + tests. (F3)
4. **Atomic candidate persistence in RunNextTask** — persist candidate id +
   metadata + result + `awaiting_confirmation` + queue-finish + event in ONE
   transaction; simulated-crash test proves no candidate-less
   `awaiting_confirmation`; candidate id identifies the execution attempt and
   rejects an immutable mismatch. (F3)
5. **Correct ApproveTask landing** — wire `RepositoryLanding` into the real
   `ApproveTask` (composition); resolve the repository resource's canonical
   **home** (not the task workspace) and **configured branch** (not hardcoded
   `main`); land the named target under compare-and-swap; record
   `canonicalSHA` as `baseCommit` (A7). Real-git tests: ff, merge. (F3)
6. **Shared per-repo lock + durable candidate storage + crash recovery** — wire
   the C2 landing lock into `LocalWorkspaceManager`; retain unresolved
   workspaces (or a durable object ref) so an approve after retry still finds
   the candidate object; crash-idempotent re-approve reconciles candidate /
   integration rows against the current canonical SHA; cross-process
   lock-contention test. (F3)
7. **Conflict lifecycle** — define + test typed-conflict behavior:
   `task.conflict` leaves the task `awaiting_confirmation`; specify retry,
   rejection, and repeated-approval-after-conflict. (F3)
8. **Behavior-change disclosure sweep** — update the now-wrong "generic means no
   gate" tests, CLI output, daemon-outcome, and filesystem-backed task
   expectations to the C2 gated behavior; keep an explicit changelog note. (F3)

## OPEN findings → this epic

### F2 [major] — token / cost accounting is a hardcoded stub

`src/agent-runner/pi.ts:329-330` emits `agent.finished` with
`tokensIn: "0", tokensOut: "0"` literally on every run — never read from the
model's usage (`turns` is real). The 007.1 A6 safe-facts fields
(`tokensIn`/`tokensOut`) and the diagnostics-export tokens path exist but always
report ZERO. A confident `0` is worse than absent — the run looks free; Ulrich
loses all quota/cost visibility.

### F3 [critical] — `generic@1` completion never lands; no candidate, no gate

A _changed_ `generic@1` task transitions straight to `completed`
(`run-next-task.ts:128-159`) with `baseCommit: null, proposalCommit: null`. No
landing candidate is created (`landing_candidates` stays empty), `main` is left
untouched, and the work lives only on `kanthord/<taskId>`. `land repository`
exists but is a fully MANUAL command, disconnected from completion. This
contradicts the 007.1 C2 design ("changed → accepted-candidate, human approval
before landing by default") and defeats the north-star goal: any same-repo
dependent task clones `main` and never sees the completed work. Marking
`completed` while `main` is untouched is misleading. Also leaves A7
(`base_commit` null) unaddressed on this path.

## Debate-hardened design directions (2026-07-20)

Debate engine (`opencode`, read-only) run on the two fix approaches. It
confirmed the direction but found the naïve versions under-specified; below is
the merged, hardened design the Stories above implement.

### F2 — real token accounting

- **Exact metric (per pi's own semantics):**
  `tokensIn = Σ(usage.input + usage.cacheRead + usage.cacheWrite)` and
  `tokensOut = Σ(usage.output)` summed across **all** assistant turns.
  - `usage.reasoning` is already inside `usage.output` — never add both.
  - Do NOT use `getLastAssistantUsage()` (last context snapshot only, not run
    totals) or `calculateContextTokens()` (context size, includes output — not
    `tokensIn`).
  - Mirror pi's established session-statistics aggregation
    (`@earendil-works/pi-coding-agent` agent-session summation of
    input/output/cacheRead/cacheWrite across assistant messages) rather than
    inventing an interpretation (reuse-pi-first).
- **Aggregate as turns finish**, including turns before a verification failure,
  escalation, or budget abort — `agent.finished` describes agent work, not
  successful task completion; failed/escalated runs still carry real usage.
- **Surface it as its own line**, not coupled to the daemon's `task … completed`
  line: emit `agent finished: turns=… tokensIn=… tokensOut=…` from the existing
  emit callback (`composition.ts` emit wiring), so changed tasks that stop at
  `awaiting_confirmation` and failed runs also report usage.
- **Tests:** multi-turn usage fixture with non-zero input, cacheRead, cacheWrite,
  output, and reasoning; assert each bucket counted exactly once and reasoning
  is not double-counted (current test permits the zero placeholder and cannot
  detect wrong arithmetic).

### F3 — completion means landed (executor-neutral candidate lifecycle)

Rerouting generic completion into the existing escalation path is **too weak** —
the existing approve→land path is not production-ready. This epic fixes the
lifecycle properly:

- **Executor-neutral result contract** (do NOT infer lifecycle from
  `task.agent === "generic@1"`). Runner outcomes:
  - `completed` — verified **no-change** only.
  - `candidate` — `{workspace, branch, baseCommit, candidateCommit, summary,
evidence}` (changed work).
  - `failed` — `{reason}`.
  - `escalated` — explicit agent-requested human intervention.
- **`RunNextTask` persists the candidate atomically** in the same transaction as
  the task transition: fresh candidate id + candidate metadata + task result +
  `awaiting_confirmation` + queue-finish + event. (Today candidate persistence
  happens later inside the git adapter → a crash leaves `awaiting_confirmation`
  with no candidate.)
- **`ApproveTask` must land correctly** — the debate found these real defects to
  fix:
  1. **Landing is not composed.** The real `ApproveTask` is built WITHOUT a
     `RepositoryLanding` (`composition.ts:233-239`; adapter created later
     `:353-359`), so the daemon approval path never lands today. Wire it.
  2. **Wrong repository path.** `ApproveTask` passes `result.workspace` as
     `homeDir` (`approve-task.ts:119-131`) → mutates the task clone, not the
     repository resource's canonical home. Resolve the repo home from the
     repository resource.
  3. **Hardcoded target.** Approval hardcodes `"main"` (`approve-task.ts:126`);
     use the repository resource's configured branch.
  4. **Wrong branch may move.** `git merge` advances the checked-out HEAD
     (`landing/git.ts:145-182`), which may differ from `candidate.target`.
     Update the named target under compare-and-swap, or verify the checked-out
     branch first.
  5. **Candidate identity unsafe.** `${taskId}-lc` (`approve-task.ts:120`) +
     persistence ignoring conflicting ids (`storage/sqlite/landing.ts:18-35`) →
     a rejected+retried task reuses the id. Candidate ids must identify the
     execution attempt; immutable mismatches must fail.
  6. **Workspace is not durable candidate storage.** Approval reads the object
     from the recorded workspace, but retry wipes that workspace
     (`workspace/local.ts:455-461`). Retain unresolved workspaces, or store the
     candidate object under a durable internal ref/object store.
  7. **Crash recovery incomplete.** Landing advances git before recording final
     DB state (`landing/git.ts:109-127, 214-224`); on retry the already-landed
     branch returns without reconciling candidate/integration rows.
  8. **Shared lock absent.** `LocalWorkspaceManager` is created without the
     landing lock directory (`composition.ts:270-283`) → prepare and landing do
     not share C2's single per-repo lock. Wire the shared lock.
  9. **Conflict behavior must be defined.** Approval emits `task.conflict` and
     returns with the task still `awaiting_confirmation`
     (`approve-task.ts:133-137`); this epic must define retry / rejection /
     repeated approval after a conflict.
- **No-change contract.** `genericProfile.verify()` currently rejects no-change
  as `NO_CHANGES` (`pi-profile.ts:63-72`); direct no-change → `completed` needs
  an intentional contract + test change.
- **Behavior-change disclosure.** C2 makes changed `generic@1` require a human
  gate by default — an intentional PUBLIC behavior change. Existing
  "generic means no gate" tests, CLI output, daemon outcomes, and
  filesystem-backed task expectations must be updated explicitly (not silently).
- **A7 fix falls out:** the landed `canonicalSHA` is recorded as `baseCommit`.
