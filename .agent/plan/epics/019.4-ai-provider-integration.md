# 019.4 AI Provider Integration (pi-ai) — Multi-Account Provider Engine

## Outcome

kanthord gains a **multi-account AI-provider engine**: it can hold many accounts
(even many of the **same** provider kind — e.g. ten Codex subscriptions), pick one per
run, and **durably bind** that account to the task so every spawn/respawn/restart keeps
running on it — the foundation that makes long-running tasks viable. (The *act of
switching* a running task from one account to another — triggers, tier rules,
notification — is **Epic 043**, which updates the durable binding this epic writes; see
Non-Goals.) All provider work goes through `@earendil-works/pi-ai`, following the pi
coding-agent gold standard, and the whole **management logic is core** (built here,
exercised via the existing minimal CLI so it is proven before Phase 2B); later epics
only *wire* to it (Epic 026 exposes these core functions over the control-plane API;
Epic 027 wires the dashboard — neither adds logic).

Three provider kinds ship:

1. **OpenAI via OpenAI OAuth** — the `openai-codex` provider, **device-code** flow
   (`loginOpenAICodexDeviceCode`), headless-friendly.
2. **OpenAI-compatible endpoint** — a custom `baseUrl` + `api` type + model, api-key in
   custody (local proxy, vLLM, Azure-style).
3. **GitHub Copilot via GitHub OAuth** — `github-copilot`, device-code, including the
   **enterprise host** case (base URL derived from the token's `proxy-ep`).

The domain model separates **provider kind** from **provider account** (a debate
finding — do not collapse them into a slug id):

- **`ProviderKind`** — `openai-codex` | `github-copilot` | `openai-compatible`.
- **`ProviderAccount`** — `{ id, providerKind, label, credentialKey, defaultModel? }`;
  a stable opaque account id (e.g. `acct_…`), a human label (`work`, `repo-a-1`), and a
  credential held under kanthord custody. Multiple accounts of one kind are ordinary.
  The account id is mapped to a pi-ai **provider-instance id at the pi-ai boundary
  only** (pi-ai keys auth by `model.provider`, so the runtime instance is registered
  under the account id — the domain never exposes `openai-codex/work`-style slugs).

Five core pieces, assembled in a documented order:

- a **ProviderAccount registry + account-keyed credential store** (custody per
  `src/git/keyring.ts`) with full **CRUD** (add/get/list/update/remove) so the CLI now,
  and Epic 026/027 later, drive the *same* core operations;
- an **observable device-code login operation** (state: user-code + URL → pending →
  complete/failed) so a terminal shows it now and a UI can render+poll it later,
  writing the credential to the store;
- an **account-scoped resolver** `buildProviderSession({ accountId, modelId })` →
  `{ model, streamFn }` from pi-ai `createModels`, threaded through the existing
  `PiSurface`/`spawnPiSession` seam into the real `Agent`;
- a **durable per-task account binding** — a daemon-owned record `task → {accountId,
  modelId}` (in run/task metadata, **not** `STATE.md`) written at account selection and
  read by the resolver on every spawn, so a task keeps running on the same account across
  respawn + daemon restart. This binding is the authoritative source-of-truth Epic 043's
  switch will later update;
- the **minimal CLI** to exercise all of the above end to end.

The session wiring uses the **proven spike recipe** (`src/agent/copilot-provider.ts`,
`test/live/copilot-smoke.ts`, verified in Podman 2026-07-11): a **Models-backed
`streamFn`** owning auth + headers + token-derived base URL, and **no `getApiKey`** on
the Agent (setting it makes pi-ai take the api-key branch and skip OAuth base-URL
derivation → `421 Misdirected Request`).

The automated gate is **hermetic** (fakes for OAuth, a pi-ai `Models` double, no
network). One real credentialed call per provider kind is a **maintainer-run live
proof** inside Podman, mirroring Epic 019.2.

## Decision Anchors

- **Ulrich, this conversation (2026-07-11)** — 019.x does the heavy provider work;
  three kinds (OpenAI OAuth, OpenAI-compatible, Copilot OAuth); **Claude deferred**;
  credentials in **kanthord keyring custody**. **Multi-account per provider kind is
  required** ("10 Codex subscriptions to different repos"). **The management CRUD logic
  is 019.x core**, tested via the minimal CLI and ready before Phase 2B; **Epic 026/027
  do wiring only, no logic.** The *act of switching* a running task between accounts is
  **trimmed out of 019.4 into Epic 043** (Ulrich, B6 2026-07-11) — 019.4 delivers only
  account **resolution + a durable binding** that 043 will update.
- **pi's context mechanism (informs Epic 043, not built here)** — pi keeps context
  across a model change because `state.messages` is a provider-neutral `AgentMessage[]`
  re-serialized per turn via `convertToLlm` (`packages/agent/src/agent-loop.ts:290-295`);
  coding-agent's `setModel()` swaps `agent.state.model` in place
  (`packages/coding-agent/src/core/agent-session.ts:1544`). This is real but **in-memory
  within one live Agent** and manual-only. kanthord's long-task continuity is instead
  **durable (STATE.md checkpoint + respawn)**, so the *switch* needs a durable,
  policy-driven design — **Epic 043**. 019.4 only writes the durable account **binding**
  the resolver reads on each spawn; it does not mutate a live Agent's model (debate
  finding: a raw swap would desync budget ledger, per-account audit, and checkpoint
  metadata).
- **Kind ≠ account (debate finding)** — collapsing account into the provider id
  (`openai-codex/work`) is leaky (breaks catalogs, OAuth-handler lookup, pricing,
  YAML/metrics/URL routing). First-class `ProviderAccount`; account→pi-ai-instance
  mapping happens only at the pi-ai boundary.
- **Gold standard: pi coding-agent** — `createAgentSession` streamFn resolving
  `getApiKeyAndHeaders(model)` then `streamSimple` (`src/core/sdk.ts`); locked file
  credential store (`src/core/auth-storage.ts`); custom endpoints via `models.json`
  (`src/core/model-registry.ts`); device-code logins in `packages/ai/src/utils/oauth/*`.
- **pi-ai seams (0.80.3, verified):** `createModels` + `Models.streamSimple`/`getAuth`;
  provider factories `@earendil-works/pi-ai/providers/{openai-codex,openai,github-copilot}`;
  `/oauth` logins `loginOpenAICodexDeviceCode`, `loginGitHubCopilot`, `getOAuthProvider`,
  `registerOAuthProvider`, `refreshOAuthToken`, `pollOAuthDeviceCodeFlow`; the
  `CredentialStore` interface with `modify()` as the sole write path pi-ai runs OAuth
  refresh inside.
- **Spike (2026-07-11):** `src/agent/copilot-provider.ts` + `test/live/copilot-smoke.ts`,
  proven against **enterprise** Copilot in Podman; recorded gotchas
  ([[copilot-provider-wiring]]). Story 003 generalizes it.
- **`src/git/keyring.ts` custody invariants (Epic 011 Story 000)** — 0600 file/env,
  owner+mode check, value-redacted logs, typed errors; Story 001 mirrors + adds a write
  path for OAuth refresh.
- **Epic 019.2 seam** — `PiSurface.spawnAgent` → `spawnPiSession` → `makeAgentOpts` →
  `new Agent(...)` in `buildRealDeps` (today passes neither model nor streamFn). This
  epic threads `model` + `streamFn` and reads the durable account binding on each spawn.
- **Epic 043 (to be authored)** — owns the *switch orchestration*: user-declared
  consent boundaries + built-in strategies, boundary-only (respawn/retry) switching,
  tier guards (same-model-account safest; cross-provider = manual approval), operator
  notification of implications, and the trigger set (gated on a typed provider-error
  taxonomy). 043 **updates** the durable binding 019.4 writes; 019.4 must not promise
  automatic or mid-session switching.
- **Epic 024 (`003-model-policy-chain`, `004-provider-registry`)** — keeps the *policy*:
  the 5-level precedence and the yaml registry that names account candidates per
  repo/slot/task. This epic delivers the *runtime engine* those reference; a reference
  note is added to 024 (Story 005). 019.4 supports only **basic** repo/slot→account
  resolution needed to pick an account per run; rich precedence stays in 024.
- **AGENTS.md / [[pi-sessions-in-process]]** — in-process `Agent` objects, no
  subprocesses; why the Claude Agent SDK (subprocess) is deferred (Non-Goals).

## Stories

- `001-provider-account-registry-and-store.md` — the `ProviderAccount` domain +
  account-keyed credential store under kanthord custody, with CRUD
  (add/get/list/update/remove) and pi-ai-`CredentialStore` compliance so locked OAuth
  refresh persists rotated tokens. Multiple accounts of one kind coexist.
- `002-observable-login-operation.md` — an observable device-code login **operation**
  (user-code + URL → pending → complete/failed) plus the `kanthord login <kind>
  --account <label>` CLI that drives it and writes the credential to the store via an
  injected login seam. Same core op a UI later renders+polls.
- `003-account-resolver-and-binding.md` — `buildProviderSession({ accountId, modelId })`
  → `{ model, streamFn }` threaded through the spawn seam; and a **durable per-task
  account binding** (daemon-owned metadata, not `STATE.md`) written at selection and read
  by the resolver on every spawn — so a task keeps running on its account across respawn
  + restart. Basic repo/slot→account resolution for spawn-time selection. **No switch**
  (Epic 043 updates this binding).
- `004-openai-compatible-provider.md` — an OpenAI-compatible `ProviderAccount` kind:
  config `baseUrl` + `api` + model, api-key in the store, resolvable through Story 003.
- `005-provider-docs-and-024-reference.md` — operator doc (`docs/`) for the CLI
  account/login/selection flows; a note that Epic 026 exposes these core ops over the API
  and Epic 027 wires the UI (wiring only); and the reference note in Epic 024's
  `004-provider-registry`.

## Verification Gate

- `npm run typecheck` exits 0; `npm test` green — new registry/store, login-op,
  resolver+binding, and openai-compatible suites plus all pre-existing suites (806
  baseline), no regression. Zero-network guard stays green: no automated test opens a
  non-loopback socket (OAuth and model calls run on fakes / a pi-ai `Models` double).
- **Registry + store (multi-account):** two accounts of the **same** kind
  (`openai-codex`) with distinct labels both persist and list; CRUD round-trips
  (add → get → update label → remove); `modify()` persists a rotated oauth token; a
  store file broader than 0600 (or wrong owner) is a typed custody error; no raw token
  appears in captured logs.
- **Observable login:** `kanthord login openai --account work` (and `github-copilot`)
  drive a **fake** device-code seam, expose the user-code + URL through the operation's
  observable state, and on completion write a `{type:"oauth"}` credential keyed by the
  account id; cancel/timeout writes nothing; unknown kind → typed error.
- **Resolver + binding:** `buildProviderSession({accountId, modelId})` returns a `model`
  whose runtime provider maps to that account and a `streamFn`; a spawned session hands
  the real Agent that `model` + `streamFn` and **no** `getApiKey` (captured via
  `agentFactory`); the **durable per-task binding** makes a spawn, a respawn, and a spawn
  after a simulated daemon restart all resolve the **same** account (asserted on durable
  state — no real call); the enterprise-Copilot base URL derives from the token.
- **OpenAI-compatible:** a configured custom account resolves to a model carrying the
  custom base URL; an unregistered account/model or a missing key is a typed error
  naming the failing entry.
- **CLI end-to-end (hermetic):** the minimal CLI can add two accounts of the same kind
  (fake login), list them, resolve a session bound to one, confirm the binding persists
  across a simulated respawn, and remove an account — the whole management surface
  exercised without network. This is the "test via minimal CLI" gate that must pass
  before Phase 2B. (Switching the bound account is Epic 043, not exercised here.)
- **Maintainer live proof (not automated):** after `kanthord login <kind> --account`,
  a generalized live smoke (`test/live/provider-smoke.ts`) makes **one real call per
  kind** (OpenAI OAuth, an OpenAI-compatible endpoint, Copilot OAuth) returning a marker
  with cost observed, **inside Podman** against an isolated credential copy. Recorded in
  the Findings-Out runbook.
- **Gate closes when** the hermetic checks (incl. the CLI end-to-end) are green AND the
  live proof records one real call per shipped kind.

## Non-Goals

- **No Claude / Claude Agent SDK.** Deferred (Ulrich). Recorded: the SDK is a subprocess
  engine conflicting with the in-process invariant + per-call budget; Claude via the
  pi-ai `anthropic` provider (API key + Claude Pro/Max OAuth) would fit this pattern —
  a later, separate decision. Not authored here.
- **No control-plane API or dashboard logic.** 019.4 owns the management **logic**
  (core functions + CLI). Epic 026 only exposes those functions over the API; Epic 027
  only wires the dashboard UI to them. **Neither adds logic** (Ulrich correction) —
  019.4 must expose the operations in a shape those thin layers can call (observable
  login op, CRUD, resolution, binding), and must not build the API transport or UI here.
- **No switch orchestration (Epic 043).** 019.4 delivers account **resolution + a durable
  binding** only. Switching a running task between accounts — the trigger set (gated on a
  typed provider-error taxonomy), the tier guards (same-model-account safest;
  cross-provider = manual approval), boundary-only (respawn/retry) application, and
  operator notification of implications — is **Epic 043**, which updates the binding
  019.4 writes. 019.4 must **not** promise automatic or mid-session switching (Ulrich B6,
  debate findings).
- **No rich account-assignment precedence.** 019.4 does basic repo/slot→account
  resolution for spawn selection; the 5-level precedence chain stays in Epic 024.
- **No real network/model call in the automated gate.** Hermetic fakes only.
- **No new session/agent mechanism.** Provider wiring + the durable binding onto the
  existing pi-agent-core `Agent` + `PiSurface` seam; a missing seam is a `Findings Out`
  entry, not new mechanism invented here.
