# 019.7 Broker Live Delivery Wiring

## Outcome

`kanthord run` turns a completed session's **local commits** into a **real PR**
on the sandbox repo **through the broker** — `git.push` then
`github.create_pr` — authenticated by the per-identity PAT, with the outbound
secret-scan armed on push and reconcile-on-boot for restart safety. This is the
load-bearing gap behind LP-A1: today the daemon spawns real pi sessions (Epic
019.6) but never delivers their commits — `deliverSession` exists on the run-loop
handle yet nothing calls it, `buildRealDeps` constructs no broker adapters, and
no live git token is loaded. This epic **assembles existing broker pieces**
(`src/broker/verbs/*`, run-loop Story 003 `deliverSession`/`reconcileHeldOps`)
into `buildRealDeps` + `tick()`, and adds the one missing component: a real
GitHub HTTP seam. It adds no new broker mechanism.

## Decision Anchors

- Phase-2 E2E testsuite `.agent/plan/e2e/phase2-e2e-testsuite.md` LP-A1 pass
  condition: "the PR came **through the broker** (audit ledger shows the op
  chain), not a manual push"; LP-A4 reconcile-on-restart; the cost-breaker-active
  and escalation conditions.
- Existing broker adapters: `github-create-pr.ts` (`makeCreatePrAdapter`,
  `GithubHttpSeam`), `git-push.ts` (`makePushAdapter`, `diffScanGuard`),
  `git-local.ts` (branch/commit/clone/fetch), `submit.ts`/`poller.ts`/
  `reconcile.ts`/`registry.ts`.
- Run-loop delivery seam (Story 003): `deliverSession`, `submitBrokerVerb`,
  `reconcileHeldOps` — present as handle methods, **uncalled** in the live path.
- Git identity custody: `src/git/keyring.ts loadIdentity` (file mode `0600` or
  `KANTHOR_IDENTITY_<NAME>_TOKEN`); the `kanthordverify` PAT is custodied at
  `.data/kanthord/credentials` (Ulrich, fine-grained, Contents+PR write).
- 2A hermetic blueprint: `src/harness/scenarios/2a-golden.ts` wires the same
  adapters + `deliverSession` on doubles — the live wiring must reproduce that
  op chain against real GitHub.
- Git-platform-integration memory: CLI-first, REST only where a structured,
  deterministic response is required — `create_pr`/`getPr`/`listByHead` are that
  case, so the HTTP seam is a REST/fetch client (Constraint in Story 001).

## Stories

- `001-github-http-seam.md` - a real fetch/REST `GithubHttpSeam`
  (`createPr`/`getPr`/`listByHead`) against `api.github.com` with Bearer auth,
  status→response-shape mapping (201 created, 422 duplicate→listByHead, PR
  states for poll/reconcile). Hermetic against a loopback mock; no real network
  under `npm test`.
- `002-identity-pat-custody.md` - `buildRealDeps` loads the `kanthordverify`
  identity token via `loadIdentity` (file `.data/kanthord/credentials`, `0600`,
  owner-checked) and exposes it for `create_pr` (Bearer) and `git.push` (HTTPS
  `http.extraHeader`); fail-closed with a typed error when the identity file is
  absent or insecure.
- `003-broker-registry-in-realdeps.md` - `buildRealDeps` constructs the verb
  registry + real adapters (`git.commit`/`git.branch` local, `git.push` with the
  secret-scan `diffScanGuard` from `patternRegistry`, `github.create_pr` with the
  real HTTP seam + token) and threads them onto `RunDaemonDeps` so `tick()` can
  deliver.
- `004-tick-delivers-session.md` - after a cleanly completed session with ≥1
  local commit on the task branch, `tick()` builds the push + create_pr inputs
  (branch, base `main`, title from the task) and calls `deliverSession`; the op
  chain is recorded in the ledger; the task reaches **complete** only after the
  PR is merged (create_pr poll observes merged) — the daemon never merges.
- `005-reconcile-and-scan.md` - `runDaemon` calls `reconcileHeldOps` at start
  (LP-A4: head-branch lookup, no duplicate PR after a kill mid-`create_pr`); the
  outbound secret-scan blocks a push whose branch diff matches a pattern and
  escalates (fail-closed when the registry is absent).

## Verification Gate

- `npm run typecheck` exits 0; `npm test` green — new hermetic tests pass (the
  HTTP seam against a loopback mock; delivery orchestration against doubles), the
  full suite shows no regression, the zero-network guard stays green.
- `node src/cli/run.ts --help` exits 0.
- **Live proof (LP-A1, recorded, not automated):** with the `codex` provider
  account and the `kanthordverify` PAT, booting the daemon on a worktree with a
  signed-off feature drives session → local commits → broker `git.push` →
  `github.create_pr` → a real PR on `kanthord-verify`, the ledger op chain
  visible; a kill mid-`create_pr` + restart reconciles to exactly one PR (LP-A4).

## Dependencies

- Epic 019.6 (provider session wired; daemon spawns real sessions).
- Epic 019.2 (`buildRealDeps`, `runDaemon`).
- **Preconditions handled operationally for the LP-A1 run (NOT built here —
  flagged for Ulrich; may need a sibling epic 019.8):**
  1. **Worktree lifecycle** — `run.ts` today only *validates* the slot; it does
     not clone the sandbox repo, configure the identity token in the worktree's
     git, or create the task branch. For LP-A1 the operator pre-clones
     `kanthord-verify` into a local path, configures `http.extraHeader` with the
     PAT, and points the slot at that path.
  2. **Feature ingestion / sign-off** — how a signed-off `tdd@1` feature reaches
     the daemon's `featureDir` is assumed pre-authored on the worktree.
  3. **Session local commit** — the pi session is expected to `git commit` its
     work via its shell tool inside the worktree; this epic delivers *existing*
     commits and does not add a commit step.

## Non-Goals

- **No worktree clone / identity git-config automation** — precondition above,
  operator-provided for LP-A1 (candidate for Epic 019.8).
- **No merge** — the human merges; the daemon only polls create_pr to terminal.
- **No new broker mechanism** — assembles `src/broker/*` + run-loop Story 003.
- **No GitLab/Jira** — Phase 2B (Epic 022).
