# EPIC 007.14 — Initiative-branch git workflow (brokered isolated clones)

> Authored 2026-07-23 as the appendix to EPIC 007, from the real-model E2E run
> `e2e-0711` (`007.11-e2e-findings.md`) and two adversarial debates
> (opencode/plan — "Debate deltas" at the bottom). It **replaces** kanthord's
> per-task clone + candidate-CAS-onto-`main` landing model with the git workflow
> a developer uses daily. Prerequisite: **EPIC 007.11 (bare managed repository
> storage)** must land first.
>
> **Working-directory strategy: Option I — Brokered Isolated Initiative Clone**
> (chosen over git-worktree on a benefit/cost basis). Agents run arbitrary
> shell, so each initiative works in a **disposable, fully isolated clone** and
> **only the daemon writes to the bare home** — the authority boundary is real,
> not conventional.

## Goal

After 007.14, kanthord builds a feature the way a developer would, with this
hierarchy:

- **Initiative → a dedicated branch** `kanthord/init/<initId>` in the bare home,
  worked on inside an **isolated clone** the agent owns.
- **Objective → exactly one commit** on that branch.
- **Task → a sequential working unit** that contributes to its objective.

Concretely, per initiative:

1. **Provision (daemon):** create `kanthord/init/<initId>` in home from the
   integration tip (`create repository --branch`, default `main`), then clone it
   into a disposable working dir with `git clone --no-hardlinks --single-branch`
   and **remove the clone's `origin`** so the agent has no write path back to
   home. Home stays outside the agent's writable view.
2. **Build (agent, sequential):** the objective's tasks run one at a time, in
   dependency order, mutating the clone. At the objective boundary the
   accumulated work is squashed into **one commit** for that objective.
3. **Broker (daemon only):** the daemon fetches that single objective commit's
   objects **from the clone into home**, validates it is **exactly one commit
   after the expected parent**, and CAS-advances `kanthord/init/<initId>` in
   home. The agent never writes home; the daemon never runs agent code.
4. Repeat 2–3 per objective, so the initiative branch in home gains **one commit
   per objective**, in order — a clean linear history.
5. **Deliver:** when all objectives are integrated, the initiative is ready for
   `pr@1` (a future agent that pushes the branch + opens a PR — deferred) or a
   manual push+PR (EPIC 007.12). The branch already lives complete in home, so
   delivery needs no reconciliation.

This is a smaller, familiar model — initiative = branch, objective = commit,
"done" = PR — and it folds in findings F2 (bare home, structurally gone), F3
(sequential build removes the intra-file rebuild storm), F4/F5 (messages adopt
the new states). It **explicitly trades** intra-initiative parallelism and adds
per-objective brokering for isolation and a clean history.

## Verification Gate

Gates: `npm run verify` (typecheck + test + verify:handoff + lint + db status)

Proof: hermetic, copy-paste-runnable against a **local bare remote** with the
fake agent. One initiative, two objectives (two sequential tasks each); proves
one-commit-per-objective, daemon-only home writes, and the isolated clone has no
write path to home.

```sh
set -euo pipefail
D=".data/e2e-0714-proof"; rm -rf "$D"; mkdir -p "$D"
git init -q --bare "$D/remote.git"; git clone -q "$D/remote.git" "$D/seed"
mkdir -p "$D/seed/src"; printf 'export const base=1;\n' > "$D/seed/src/todo.mjs"
git -C "$D/seed" add -A
git -C "$D/seed" -c user.email=e@e -c user.name=e commit -qm init
git -C "$D/seed" push -q origin HEAD:main
export KANTHORD_DB="$PWD/$D/k.db"
node src/main.ts db migrate >/dev/null
PROJECT=$(node src/main.ts create project --name init-wf | grep -oE '01[0-9A-HJKMNP-TV-Z]{24}')
REPO=$(node src/main.ts create repository --project "$PROJECT" --name verify \
  --remote-url "file://$PWD/$D/remote.git" --branch main --auth ambient \
  --path "$PWD/$D/home" | grep -oE '01[0-9A-HJKMNP-TV-Z]{24}')
printf dummy > "$D/cred.txt"
CRED=$(node src/main.ts create credential --project "$PROJECT" --name c1 --provider openai-codex --value-file "$D/cred.txt" | grep -oE '01[0-9A-HJKMNP-TV-Z]{24}')
PROV=$(node src/main.ts create ai-provider --project "$PROJECT" --name p1 --provider openai-codex --model gpt-5.6-terra | grep -oE '01[0-9A-HJKMNP-TV-Z]{24}')
# 1 initiative, 2 objectives (obj-A then obj-B), each with 2 sequential tasks.
scripts/e2e/make-initiative-graph.sh "$D/graph" >/dev/null   # NEW helper (Story F)
node src/main.ts import graph "$D/graph" --create --project "$PROJECT" --bind source="$REPO" --bind provider="$PROV" --bind cred="$CRED" >/dev/null
INIT=$(node -e 'console.log(JSON.parse(require("fs").readFileSync(process.argv[1]+"/.kanthord-export.json","utf8")).initiativeId)' "$D/graph")
OBJS=$(node -e 'const g=JSON.parse(require("fs").readFileSync(process.argv[1]+"/.kanthord-export.json","utf8"));process.stdout.write((g.objectiveIds||[]).join(" "))' "$D/graph")
set -- $OBJS; OBJ_A=$1; OBJ_B=$2
export KANTHORD_FAKE_AGENT="$PWD/$D/graph/.fake-agent.json"
HOME_GIT="$PWD/$D/home.git"; [ -d "$PWD/$D/home/.git" ] && HOME_GIT="$PWD/$D/home/.git"
INITREF="refs/heads/kanthord/init/$INIT"
count () { git --git-dir="$HOME_GIT" rev-list --count refs/heads/main.."$INITREF" 2>/dev/null || echo -1; }

# ── Story A — initiative branch created in home; isolated clone has NO write path to home ──
node src/main.ts run daemon --until-idle --poll-interval 200 >/dev/null   # builds obj-A's tasks in the clone
git --git-dir="$HOME_GIT" rev-parse --verify "$INITREF" >/dev/null \
  && echo "PASS A: initiative branch exists in home" || { echo "FAIL A"; exit 1; }
CLONE=$(node src/main.ts get initiative --id "$INIT" --json | node -e "let s='';process.stdin.on('data',d=>s+=d).on('end',()=>{process.stdout.write(JSON.parse(s).workspace||'')})")
[ -n "$CLONE" ] && [ -z "$(git -C "$CLONE" config --get remote.origin.url || true)" ] \
  && echo "PASS A2: initiative clone has no origin remote (no write path to home)" \
  || { echo "FAIL A2: clone can reach home (origin=$(git -C "$CLONE" config --get remote.origin.url))"; exit 1; }

node src/main.ts approve objective --id "$OBJ_A" >/dev/null   # broker: fetch + validate 1-commit + CAS
[ "$(count)" = "1" ] && echo "PASS B: after obj-A, initiative branch is exactly 1 commit ahead of main" || { echo "FAIL B: count=$(count)"; exit 1; }
node src/main.ts get objective --id "$OBJ_A" --json | node -e "let s='';process.stdin.on('data',d=>s+=d).on('end',()=>{const o=JSON.parse(s);const r=(o.integrations||[]).find(i=>i.repository==='$REPO');process.exit(r&&r.state==='integrated'?0:1)})" \
  && echo "PASS B2: objective A integration state = integrated" || { echo "FAIL B2"; exit 1; }

# ── Story B — objective B builds on A and brokers as a SECOND single commit ──
node src/main.ts run daemon --until-idle --poll-interval 200 >/dev/null
node src/main.ts approve objective --id "$OBJ_B" >/dev/null
[ "$(count)" = "2" ] && echo "PASS C: after obj-B, initiative branch is exactly 2 commits ahead (one per objective)" || { echo "FAIL C: count=$(count)"; exit 1; }
# history is linear (each objective is one commit)

  && echo "PASS C2: initiative history is 2 linear commits" || { echo "FAIL C2"; exit 1; }

# ── Story D — a non-tip objective commit is NOT rewritable in place ──
node src/main.ts retry objective --id "$OBJ_A" 2>&1 | grep -qiE 'non-tip|corrective|restart|not rewritable|already integrated' \
  && echo "PASS D: non-tip objective refuses in-place rewrite (guided to corrective objective / restart)" || { echo "FAIL D"; exit 1; }

# ── Story E — initiative ready for delivery; branch complete in home ──
node src/main.ts get initiative --id "$INIT" --json | node -e "let s='';process.stdin.on('data',d=>s+=d).on('end',()=>{process.exit(/awaiting_pr|ready|complete/.test(JSON.parse(s).status)?0:1)})" \
  && echo "PASS E: initiative reports ready for PR (branch complete in home)" || { echo "FAIL E"; exit 1; }

echo "007.14 PROOF OK"
```

## Stories

- **A — Initiative branch + brokered isolated clone (Option I provisioning).**
  At initiative start the daemon creates `kanthord/init/<initId>` in the bare
  home from the integration tip, then provisions the working dir with
  `git clone --no-hardlinks --single-branch --branch kanthord/init/<initId>
<home> <dir>` and **removes the clone's `origin`** (no agent write path to
  home). `--no-hardlinks` is mandatory: hardlinked objects share a mutable inode,
  so an agent's in-place write could corrupt home — the copy must be real.
  Record the clone path on the initiative (surfaced by `get initiative`).
- **B — Sequential tasks → one objective commit.** The objective's tasks run in
  a **stable serial order** (topological over the task DAG, tie-broken by
  explicit task order then task id), one at a time, in the initiative clone.
  When all of an objective's tasks complete, the daemon squashes the accumulated
  work into **exactly one commit** on the initiative branch **inside the clone**
  and moves the objective to `awaiting_confirmation`. No home write yet.
- **C — The broker: daemon-only home integration.** `approve objective --id
<objId>` (per-objective human gate) triggers the broker, run **only by the
  daemon**: (1) `git --git-dir=<home> fetch <clone> <objectiveCommitOID>`;
  (2) validate it is **exactly one commit** after the recorded parent
  (`rev-list --count <parent>..<oid>` == 1) and passes commit/tree policy;
  (3) CAS `update-ref refs/heads/kanthord/init/<initId> <oid> <parentOID>`;
  (4) record `integrated` in kanthord state. CAS is required despite serial
  scheduling (concurrent daemons, crashes, manual edits; git + SQLite are not
  atomic). The agent is never involved in a home write.
- **D — New state machines + non-tip immutability.** Do **not** overload the
  007.8 `landed` state. Add: objective integration
  `building → awaiting_confirmation → conflict | integrated`; initiative
  `building → awaiting_pr → delivered`; keep task working-unit status for the
  sequential build. An **integrated non-tip objective** commit is immutable
  (rewriting it changes every later commit's SHA): allowed recovery is a
  **corrective objective** appended at the tip, or **restart the initiative
  branch generation** (bump a generation number, re-provision from base) — the
  old generation stays auditable. `retry objective` on a non-tip objective is
  refused with that guidance.
- **E — Objective integration conflict + freshness.** When main (or the
  initiative base) moved such that the objective commit is not a clean single
  commit onto the current initiative tip, run an objective-level resolution: an
  agent resolves in the isolated clone with conflict context, re-squashes to one
  commit, the full verification gate re-runs, and the daemon re-brokers under
  CAS after a human approves. Cross-initiative conflicts surface at
  delivery/PR time (coarsest unit), so keep initiatives small and freshness-check
  the base before asking for approval.
- **F — Delivery hook + ergonomics + Proof harness.** When all objectives are
  `integrated`, the initiative goes `awaiting_pr`; a `pr@1` agent stub is
  registered (implementation deferred — it will push the initiative branch +
  open a PR via EPIC 007.12), and a manual push+PR path is documented meanwhile.
  `approve objective` prints a confirmation line; `run daemon --until-idle`
  reports both objectives awaiting brokering and initiatives awaiting PR (F4/F5).
  Add `scripts/e2e/make-initiative-graph.sh` (1 initiative, 2 objectives, 2
  sequential tasks each) for the Proof.

## Non-goals

- **No git worktrees.** Rejected on the benefit/cost comparison: worktrees share
  a mutable object/ref store with home, so an agent running arbitrary commands
  could corrupt state every initiative depends on. Option I's disposable,
  origin-less, `--no-hardlinks` clone makes the isolation real. (The disk cost of
  duplicating base history per active initiative is the accepted price;
  `--single-branch`/shallow trims it.)
- **No intra-initiative parallelism.** Tasks and objectives run sequentially
  within an initiative; **different initiatives may run concurrently** (each in
  its own isolated clone). Topological-parallel task execution + deterministic
  transplant is the deferred remnant of EPIC 007.13, not this epic.
- **No `pr@1` implementation** and **no remote push** here — delivery is
  EPIC 007.12; `pr@1` is a deferred agent. This epic proves the branch is
  complete and correct **in home**, ready to hand off.
- **No bare-home migration** — EPIC 007.11 owns it and is a hard prerequisite.
- **No removal of the human gate** — per-objective approval (the broker trigger)
  stays; measuring the burden before relaxing it is future work.

## Debate deltas (opencode/plan, 2026-07-23)

- **Option I over worktrees and over plain hardlinked clone.** Hardlinks share a
  mutable inode, so they stop `unlink`/`gc` from deleting home's objects but do
  **not** stop an in-place write (`truncate`, a bad tool) from corrupting home
  through the shared inode; and a clone whose `origin` is the writable home path
  is itself a `git push --force` route into home. Real isolation requires
  `--no-hardlinks` **and** removing the write path (`origin`) **and**
  daemon-only home mutation. Plain hardlinked clone is the "awkward middle" —
  the sync cost of a clone without the isolation it claims.
- **Authority separation is the decisive property:** agents mutate only
  disposable initiative storage; only the daemon updates the persistent bare
  home (fetch exact OID + validate + CAS). Story C encodes it.
- **Cost is real and accepted:** Option I duplicates reachable base history per
  active initiative and adds the broker step's failure states (fetch-ok-but-
  CAS-failed, clone-gone-after-commit, committed-twice) — Story C/D own them.
  This buys out the shared-store blast radius, which is unacceptable when agents
  run arbitrary code.
- **New state machines, non-tip immutability, objective-level conflict resolver,
  CAS-still-required** — carried from the prior workflow debate; the integration
  unit is now the single objective commit on the initiative branch.
