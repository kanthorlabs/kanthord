# EPIC 008.2 — Static TDD executor (`tdd@1`)

> Phase 2 of the former EPIC 008 (split 2026-07-18). Orchestrates EPIC 008.1's
> three role agents into a runnable `tdd@1` cycle on repos whose tests need no
> app-up ("static" witnesses), with the FULL confirmation protocol SCHEMA (the
> performable subset wired here). Capacity is DETERMINISTIC (default + explicit
> declaration) — the agentic onboarding loop moved to EPIC 008.4 (debate
> 2026-07-18: onboarding is a separable system; 008.2 was too large). Phase 1 (role
> agents) is `008.1-role-agents.md`; Phase 3 (runtime witnesses) is
> `008.3-runtime-verification.md`; Phase 4 (agentic onboarding) is
> `008.4-agentic-onboarding.md`.

## Goal

A Task can name `tdd@1` as its `executor` and the daemon runs it as a **generic,
multi-agent TDD cycle** — 008.1's TestEngineer / SoftwareEngineer /
ReviewerEngineer, coordinated by kanthord's own logic through 008.1's `RoleRunner`
(never a second agent loop). It is **generic across static project stacks**: lanes
come from 008.1; witness + acceptance commands come from a resolved **static
`ProjectCapacity`** — resolved DETERMINISTICALLY from `Project.capacity` (declared
explicitly, no agent) or a conservative default when repo evidence supports it,
never from an LLM. The cycle runs **one or more RED → GREEN rounds**: the suite is
proven green once at the start, each round must FAIL at RED and PASS at GREEN under
an engine-run **witness** (the resolved static test command), guarded against a
GREEN that games the witness by editing command-defining files. A reviewer gate
whose findings cite a resolvable `source` and whose verdict is machine-enforced
routes `action:yes` findings back through the loop; then a **human confirmation
gate** parks the run. The run makes **one proposal commit at the end**; crash
safety is re-run-on-crash from the task's base commit, anchored by a durable
**attempt identity** minted before the first role call (no durable ledger).

A repo whose resolved witness needs a running app is **not silently failed** and
**not mis-routed**: `tdd@1` parks it under a durable, non-approvable `blocked`
confirmation cause (`reason: runtime-unsupported`) that EPIC 008.3 resumes when
runtime capacity lands — never `retry`/`fail` (which cannot succeed / would
destroy the resumable state).

`generic@1` (008.1's `GenericExecutor`) and `tdd@1` (the `TddExecutor` added here)
implement the same task-executor seam, so registering `tdd@1` needs no queue/claim
behavior change.

## Verification Gate

Gates: `npm run verify` (hermetic — the full static `tdd@1` cycle runs against
008.1's role agents on EPIC 006's `FakeSessionFactory` with scripted turns for all
three roles: real pi `Agent`, real tools, real git in temp dirs, no network,
no app-up. The suite covers: the pure `tdd@1` transition function state-by-state
(incl. the debate-locked fixes — blocker-input preserved across RED retries; a
bounded per-cycle RED counter + `red-cycle-limit`; a non-approvable `blocked`
terminal; a VERIFY failure synthesizing a bounded REVIEW re-route → `review-
cycles-exhausted`); the `SETTLE` step naming (the engine round-continuation step,
renamed from CONFIRM so it can never be confused with the human `Confirmation`
gate); deterministic static `ProjectCapacity` resolution precedence; the durable
attempt identity created before the first role call and referenced by every run
row; suite-green-at-start (per static `TestKind`) plus per-round RED-fail /
GREEN-pass witness evidence bound to the task+attempt; a RED resolving to exactly
ONE `TestKind` (zero-match / multi-kind fails the step); witness-integrity (a
GREEN that edits a command-defining file invalidates the witness); the
`TddExecutor` driving 008.1 roles via `RoleRunner` with the per-step lane check;
`failureCount`→`escalated` at the threshold; the full confirmation protocol schema
with every terminal mapped to a durable `Confirmation` cause (`ready` /
`unverified` / `escalated` / `blocked`) resolved atomically by `confirmationId`,
each wired verb (approve / reject / retry / fail / cancel) with its legal cause,
optional notes feeding the loop, stale-`confirmationId` rejection, and the
`auto|manual` policy; a re-run-after-crash reaching the same single parked
proposal via the same attempt authority; and a `requiresRuntime` witness parking
`blocked{runtime-unsupported}`.)

Proof: two parts. **Part A** is the correctness gate — deterministic: capacity is
declared from a file, roles run through 008.1's `replay` provider, and the witness
runs REAL `node --test` on the files the replay wrote, so the whole cycle is
reproducible and every cause + one legal resolution is exercised through the real
CLI. **Part B** is a non-gating live-provider smoke.

  "verifyCommands": [] }

## Stories

Depends on EPIC 008.1 (role agents, `RoleRunner`, role-lane policy + workspace
service, `submit_review`, the `replay` provider) and the shared prerequisites
(executor rename, core graph codec).

1. **Pure `tdd@1` definition (domain).** `src/domain/tdd.ts`: the zero-I/O typed
   transition function — RED → GREEN → **SETTLE** (round-continuation: loop while
   the TE opens further REDs within the ACs; renamed from CONFIRM to avoid
   clashing with the human `Confirmation` gate — debate B18) → REVIEW → route
   `action:yes` findings back as a new RED exactly once per cycle, bounded review
   cycles (default 2), per-step attempt limit (default 3) → typed terminals.
   **Debate-locked fixes:** blocker-input lives ON the RED step, preserved across
   retries; a per-cycle RED counter + `red-cycle-limit`; a non-approvable
   `blocked{kind;reason}` terminal; a VERIFY failure synthesizes a REVIEW outcome
   and re-calls the transition, bounded by `maxReviewCycles` → `review-cycles-
exhausted`. All terminals as DATA; hermetically tested. `ReviewFinding` carries
   `severity`, `action`, `file`, `note`, `source`.

2. **Durable attempt identity (single authority).** A durable attempt row minted
   BEFORE the first role call (`{ id, taskId, createdAt }`), the canonical
   execution-run authority. Every run row — events, `Confirmation`, crash-recovery
   reset — references THIS id; `failureCount` is a separate counter, NOT the
   identity. On crash, the re-claimed job reads the active attempt authority (a new
   attempt is minted for a fresh run; the prior attempt's id stays on its rows for
   audit). **EPIC 009.1 consumes this same authority unchanged — no re-mint, no
   migration** (revises "carried by Confirmation/failureCount"; debate B12/S3).
   Owns its migration: the `attempts` table (or `Task.activeAttemptId` + history).

3. **Step contracts.** Each step declares its role ref, mutation policy (RED: TE
   writes test-lane only, non-empty diff; GREEN: SE writes non-test only incl.
   toolchain/setup, non-empty diff; SETTLE: engine re-runs the round's witness,
   expects pass, no new lane diff; REVIEW: empty diff + structured findings), the
   witness EXPECTATION (RED fail / GREEN pass), retry policy, output schema. The
   step owns the EXPECTATION; the witness COMMAND is SELECTED from the resolved
   static `ProjectCapacity` — never from the model.

4. **Witness selection + initial-green (deterministic, mixed-capacity-safe).**
   Resolves the "select before RED vs after RED" tension (debate B7/B8): if the
   project has exactly ONE static `TestKind`, it is preselected. With multiple
   static kinds, the kind is selected AFTER RED by the file the TE wrote — **a RED
   must resolve to exactly ONE `TestKind`** (zero-match / multi-kind fails the step
   with a structured code) — and that kind's green baseline is proven against the
   PRE-RED snapshot before the RED result is accepted. Suite-green-at-start runs
   each static `TestKind`'s command once (the baseline). A task whose resolvable
   witness is `requiresRuntime` (all eligible kinds require runtime, OR the RED
   resolves to a runtime kind) parks `blocked{runtime-unsupported}` at THAT point
   — deterministic, durable, never a placeholder branch.

5. **Engine-owned verification: static witness vs acceptance + integrity.** The
   engine runs the commands and captures `CommandEvidence` (command, exit, tail),
   reusing the landed `verification.ts` runner (no second runner). An agent's claim
   is never evidence. `ProjectCapacity = { tests: TestKind[] ({ name; command;
filePatterns; requiresRuntime }); verifyCommands: string[]; runtime? }` — the
   `runtime` block is RECORDED but NEVER executed here (008.3 owns it); it may be
   present for 008.3 without affecting static resolution (debate B15: mixed
   capacity is stored, runtime excluded from execution). **Witness (per round,
   suite-scoped, forward-moving tree, no commits):** suite green once at start;
   each round RED → expect FAIL, GREEN → re-run IDENTICAL command → expect PASS;
   each GREEN is the next round's green-base. **Witness-integrity guard (debate
   B9):** the command-defining files a witness resolves through (e.g.
   `package.json` scripts, test config) are fingerprinted at capacity-confirm time;
   a GREEN that changes a command-defining file INVALIDATES the witness (the SE
   cannot make the identical command pass by moving the goalposts). **Acceptance:**
   `Task.verification[]` runs ONCE after review-complete; `echo
"NO_VERIFICATION_REQUIRED"` is a valid no-op; a failure re-routes through the
   bounded VERIFY→REVIEW synthesis.

6. **Deterministic capacity resolution + `project capacity set`.** Witness capacity
   resolves from `Project.capacity` (declared via `project capacity set <project>
--from-file <capacity.json>`, schema-validated, deterministic — NO agent, NO
   LLM) > one conservative default only when repo evidence supports it (an existing
   non-watch `test` script) > else **no executable static capacity → park
   `unverified` BEFORE any agent run**. Confirmed capacity commands are **trusted
   project configuration** (they run automatically on the host — OS containment is
   EPIC 009). Owns its migration: `Project.capacity` + provenance
   (`confirmedAt`, `evidenceFingerprint`) columns. (The AGENTIC `project onboard`
   loop — inspect / suggest / NL-revise — is EPIC 008.4; 008.2 needs only
   deterministic declaration to run static `tdd@1` on non-conventional repos.)

7. **In-code run + re-run on crash.** The whole cycle executes in code inside ONE
   claimed job — no ledger, no per-step commits. The durable anchor is
   `taskBaseCommit` (repo HEAD at task start) plus the attempt authority (story 2).
   On crash the re-claimed job resets the workspace to `taskBaseCommit` and re-runs
   from the start. Between attempts of a step, the tree is reset to that step's
   snapshot (008.1's workspace service). Exactly ONE commit — the proposal — at the
   end.

8. **`TddExecutor` + executor registration + failure escalation.** `TddExecutor`
   implements the task-executor contract and registers as `tdd@1`. Internally:
   mint attempt → reset-to-`taskBaseCommit` → loop { snapshot tree → pure next-step
   → dispatch via 008.1 `RoleRunner` → engine witness/lane verification → emit
   `executor.step.*` event } → on terminal, make the single proposal commit (if
   any) and park via a `Confirmation`. **Durable `Task.failureCount`** increments
   on each failed pass (`review-cycles-exhausted` / `step-attempt-limit`); at the
   hardcoded `TASK_FAILURE_THRESHOLD` (default 3) the executor parks `escalated`
   instead of auto-routing. A HUMAN `reject`/`retry` grants one attempt (threshold
   re-applies only after it fails); a granted attempt mints a NEW attempt, resets
   per-run counters, re-enters from `taskBaseCommit` with the note as blocker
   input, and the note survives a crash.

9. **Human⇄executor confirmation protocol — COMPLETE SCHEMA, performable subset
   wired.** ONE durable, attempt-bound boundary. Every park writes a durable
   `Confirmation { id, taskId, attemptId, cause: ready|unverified|escalated|
blocked, report{ summary, rounds, verdict, steps[], evidence[], verification },
proposal?{ resourceId, baseRevision, headRevision }, issue?{ summary,
evidence[] }, createdAt, resolvedAt? }`. The SCHEMA is COMPLETE at ship (all
   four causes, incl. `blocked` with its own lifecycle, and the `runtime` capacity
   field) so **008.3 adds NO confirmation/schema migration**. Legal decisions are
   DERIVED from cause + policy + proposal presence: `ready → approve|reject`;
   `unverified → run-anyway|reject`; `escalated → retry|fail`; `blocked → cancel`
   (a non-approvable durable park; runtime-unsupported blocked tasks are RESUMED by
   008.3, not by `retry`/`fail`). A `Resolution { confirmationId, decision, note? }`
   is accepted atomically ONLY if that confirmation is still active
   (`confirmationId`-guarded — stale/duplicate rejected). Every decision may carry
   an OPTIONAL note (reject/retry → next-attempt blocker input; fail → reason;
   approve → audit). `Task.confirmation = auto|manual`: `manual` parks a `ready`
   terminal; `auto` completes it without parking but still records the durable
   report + a completion event (`approval: auto`); `auto` never applies to
   `unverified`/`escalated`/`blocked`. CLI: `task confirmation show <id>`; verbs
   `task approve|reject|retry|fail|cancel|run-anyway <id> --confirmation <cid>
[--note <text> | --note-file <path|->]`; `task list --status
awaiting_confirmation`. Owns its migration: `Confirmation` + `Resolution` tables
   - `Task.confirmation`.

9a. **`run-anyway` — structural-only execution contract (static unverified).** The
escape hatch for an `unverified` task (no static witness capacity): `run-anyway`
re-enters the current unverified attempt and runs the roles STRUCTURALLY — TE
writes, SE writes, RE reviews, LANES ENFORCED (008.1) — but with NO witness gate
(no RED-fail / GREEN-pass evidence, since there is no capacity to run). Defined
mechanics (debate 008.3-S3): it REUSES the current durable attempt authority; on
re-entry the tree is reset to `taskBaseCommit`; the acceptance gate
(`Task.verification[]`) STILL runs if present (structural ≠ unchecked — a task
may carry acceptance commands even with no witness kind), else the proposal is
tagged `verification: unverified`; the resulting proposal resolves via the
task's `confirmation` policy WITHOUT re-asking; `run-anyway` applies ONCE — a
second `unverified` outcome parks again. Moved here from 008.3 (debate 008.3-S2:
it completes 008.2's own static unverified lifecycle and needs no runtime). (The
runtime-un-bringable-up `unverified` case reuses this same contract in 008.3.)

10. **Events surface.** `executor.step.started`/`finished` (role, kind, status,
    witness evidence) through the feed, filterable by `--task`/`--attempt`; a park
    emits a small confirmation event (`taskId`, `confirmationId`, `cause`), an
    `auto` resolution emits a completion event (`approval: auto`). New event-type
    migration. Because re-run-on-crash + retries duplicate events, the `get task`
    step count means **successful logical steps** and consumers tolerate
    duplicates. `get task --id` summary names executor, step count, verdict.

11. **End-to-end smoke + Proof runbook.** Hermetic full static cycle on scripted
    fakes: happy path (RED→GREEN→SETTLE→REVIEW→park `ready`→approve), `auto`-policy
    completing without a park, reviewer `action:yes` re-route once, `reject`-with-
    note re-entry, attempt-limit `escalated` resolved by `retry`-with-note and by
    `fail`, no-static-capacity `unverified`→reject AND `unverified`→run-anyway
    (structural-only, acceptance still run), `requiresRuntime`
    `blocked{runtime-unsupported}`→cancel, stale-`confirmationId` rejection, and
    re-run-after-crash via the same attempt authority. Plus the Proof runbook above
    (with the `scripts/proof/run-cause.sh` helpers + the `greet-cycle.replay`
    fixture it references).

## Decisions

- **Full confirmation SCHEMA ships once, here (all four causes incl. `blocked`).**
  An unused supported cause is cheaper than a later SQLite `CHECK`-constraint
  rebuild. `blocked` is a real cause with cancel/resume lifecycle, NOT `escalated`
  (debate B10/B16/B17). All static-producible verbs are wired here, incl.
  `run-anyway` for the static `unverified` case (story 9a).
- **Capacity is deterministic in 008.2** (explicit `project capacity set` from a
  file + conservative default); the AGENTIC onboarding loop moved to EPIC 008.4
  (debate B14/S1) — this shrinks 008.2 and keeps the correctness path off the LLM.
  `project capacity set` schema-validates any recorded `runtime` block
  (`readiness` requires `start`) even though 008.2 never executes it (debate
  008.3-S5) — the same validation the agentic path will reuse.
- **The engine round-continuation step is `SETTLE`, not `CONFIRM`** — disambiguated
  from the human `Confirmation` gate (debate B18).
- **Witness selection is deterministic and mixed-capacity-safe** (single-kind
  preselect; else post-RED single-kind resolution + per-kind pre-RED green
  baseline); a runtime-only witness parks `blocked` at the resolution point, never
  a placeholder (debate B7/B8).
- **Witness integrity is enforced** — a GREEN editing a command-defining file
  invalidates the witness (debate B9).
- **One durable attempt authority, minted before the first role call**, referenced
  by every run row; 009.1 consumes it unchanged (debate B12/S3).
- **No durable ledger, no per-step commits; one proposal commit at the end.**
- **`tdd@1` reuses 008.1 wholesale** — roles, `RoleRunner`, role-lane policy +
  workspace service, `submit_review`, the `replay` provider. This epic adds
  orchestration, verification, deterministic capacity, attempt authority, and the
  confirmation protocol.

## Non-goals

- **No runtime / app-up witnesses, no `RuntimeController`, no runtime lease / dirty
  marker, no runtime execution** — EPIC 008.3. **No AGENTIC onboarding loop** —
  EPIC 008.4. This epic RECORDS (and schema-validates) a declared `runtime` block
  but never executes it; runtime-witness tasks park `blocked{runtime-unsupported}`.
  (`run-anyway` for the STATIC unverified case IS in scope — story 9a.)
- **No `pr@1` / `answer@1`** — the executor catalog + role registry + artifact
  extension points are the deliverable.
- **No durable step ledger / `workflow_runs`/`workflow_steps` / per-step commits.**
- **No parallel or cross-task steps** — one run, one workspace, sequential.
- **No per-role AIProvider/model** — all roles use the task's `ai_provider`.
- **No mid-run human steering** — touchpoints are the parked confirmation gate and
  attempt-limit parks.
- **No OS sandboxing / ring-1 hardening / SecretRedactor / env allowlist /
  PathPolicy** — EPIC 009.1 (which also generalizes this epic's attempt authority).
