# EPIC 007.12 — Repository publication (land → remote delivery)

> Authored 2026-07-22 from run `e2e-0711` finding F1 (`007.11-e2e-findings.md`)
> and the fix-approach debate (opencode/plan). The E2E built and landed a
> 5-endpoint feature, but the landed commits never left the machine: landing is
> a **local** ref/worktree update, and no CLI command publishes to the remote.
> This epic adds the missing capability — an explicit, human-gated publish — and
> first pins down what "delivered" means.
>
> **Sequencing (2026-07-23 workflow redesign):** runs **after** EPIC 007.14
> (initiative-branch workflow) and EPIC 007.11 (bare storage). Under 007.14 the
> publish unit is the **initiative branch** `kanthord/init/<initId>` (complete in
> the bare home, one commit per objective), and this capability is what the
> deferred **`pr@1`** agent calls to push the branch + open the PR. A bare git
> remote has **no PR API**, so the hermetic Proof asserts a **push** (remote ref
> advance); PR creation needs a concrete hosting adapter and is proven separately
> when one is selected.

## Goal

After 007.12 an operator can publish a landed repository target to its remote
from the CLI, as a step distinct from landing. `publish repository --repository
<id> --branch <b>` pushes the landed branch to the repo's configured remote
using the stored credential (the same GIT_ASKPASS plumbing as clone),
**fast-forward-only by default**: if the remote branch has diverged from the
expected tip, publish **fails loudly** and reports the divergence rather than
force-pushing. Because a local land and a remote push cannot be one atomic
transaction, publication has its **own persisted, queryable state** per
repository target (`unpublished` / `published@<remoteOID>` / `diverged`), so the
daemon can answer "did delivery reach the remote?" — closing the gap where
"completed" tasks silently never shipped.

## Verification Gate

Gates: `npm run verify` (typecheck + test + verify:handoff + lint + db status)

Proof: hermetic, copy-paste-runnable against a **local bare remote** (no
network, no real model). Lands a fake-agent candidate, publishes it, and asserts
the remote ref actually advanced; then proves the divergence guard.

```sh
set -euo pipefail
D=".data/e2e-0712-proof"; rm -rf "$D"; mkdir -p "$D"
git init -q --bare "$D/remote.git"
git clone -q "$D/remote.git" "$D/seed"
mkdir -p "$D/seed/src"; printf 'export const x=1;\n' > "$D/seed/src/base.mjs"
git -C "$D/seed" add -A
git -C "$D/seed" -c user.email=e@e -c user.name=e commit -qm init
git -C "$D/seed" push -q origin HEAD:main
BASE=$(git -C "$D/seed" rev-parse HEAD)
export KANTHORD_DB="$PWD/$D/k.db"
node src/main.ts db migrate >/dev/null
PROJECT=$(node src/main.ts create project --name pub-proof | grep -oE '01[0-9A-HJKMNP-TV-Z]{24}')
REPO=$(node src/main.ts create repository --project "$PROJECT" --name verify \
  --remote-url "file://$PWD/$D/remote.git" --branch main --auth ambient \
  --path "$PWD/$D/home" | grep -oE '01[0-9A-HJKMNP-TV-Z]{24}')
printf 'dummy' > "$D/cred.txt"
CRED=$(node src/main.ts create credential --project "$PROJECT" --name c1 --provider openai-codex --value-file "$D/cred.txt" | grep -oE '01[0-9A-HJKMNP-TV-Z]{24}')
PROV=$(node src/main.ts create ai-provider --project "$PROJECT" --name p1 --provider openai-codex --model gpt-5.6-terra | grep -oE '01[0-9A-HJKMNP-TV-Z]{24}')
scripts/e2e/make-landing-graph.sh "$D/graph" >/dev/null
node src/main.ts import graph "$D/graph" --create --project "$PROJECT" --bind source="$REPO" --bind provider="$PROV" --bind cred="$CRED" >/dev/null
INIT=$(node -e 'console.log(JSON.parse(require("fs").readFileSync(process.argv[1]+"/.kanthord-export.json","utf8")).initiativeId)' "$D/graph")
export KANTHORD_FAKE_AGENT="$PWD/$D/graph/.fake-agent.json"
node src/main.ts run daemon --until-idle --poll-interval 200 >/dev/null
ROOT=$(node src/main.ts list task --initiative "$INIT" --json | node -e "let s='';process.stdin.on('data',d=>s+=d).on('end',()=>{process.stdout.write(JSON.parse(s).find(t=>t.dependencies.length===0).id)})")
node src/main.ts approve task --id "$ROOT" >/dev/null

# ── Story A/B — publish pushes to the remote (fast-forward) ──
REMOTE_BEFORE=$(git --git-dir="$D/remote.git" rev-parse main)
[ "$REMOTE_BEFORE" = "$BASE" ] && echo "PASS pre: remote still at base before publish" || { echo "FAIL pre"; exit 1; }
node src/main.ts publish repository --repository "$REPO" --branch main 2>&1 | tee /tmp/pub.out
REMOTE_AFTER=$(git --git-dir="$D/remote.git" rev-parse main)
LOCAL_HEAD=$(git -C "$D/home" rev-parse main)
[ "$REMOTE_AFTER" = "$LOCAL_HEAD" ] && [ "$REMOTE_AFTER" != "$BASE" ] \
  && echo "PASS A: remote main advanced to the landed local head ($REMOTE_AFTER)" \
  || { echo "FAIL A: remote not updated (before=$REMOTE_BEFORE after=$REMOTE_AFTER local=$LOCAL_HEAD)"; exit 1; }

# ── Story C — publish state is queryable and now reports published ──
node src/main.ts get repository --id "$REPO" --json 2>/dev/null | node -e "let s='';process.stdin.on('data',d=>s+=d).on('end',()=>{const r=JSON.parse(s);process.exit(r.publication&&/published/.test(r.publication.state)&&r.publication.remoteOID==='$REMOTE_AFTER'?0:1)})" \
  && echo "PASS C: get repository reports publication{state=published, remoteOID}" \
  || { echo "FAIL C: publication state not queryable/incorrect"; exit 1; }

# ── Story D — divergence guard: a moved remote makes publish fail loudly (no force) ──
git -C "$D/seed" pull -q origin main
printf 'divergent\n' > "$D/seed/src/other.mjs"; git -C "$D/seed" add -A
git -C "$D/seed" -c user.email=e@e -c user.name=e commit -qm diverge
git -C "$D/seed" push -q origin HEAD:main   # remote now ahead of local's known tip
# land another candidate locally so there is something to publish
node src/main.ts run daemon --until-idle --poll-interval 200 >/dev/null || true
SIB=$(node src/main.ts list task --initiative "$INIT" --json | node -e "let s='';process.stdin.on('data',d=>s+=d).on('end',()=>{const t=JSON.parse(s).find(x=>x.status==='awaiting_confirmation');process.stdout.write(t?t.id:'')})")
[ -n "$SIB" ] && node src/main.ts approve task --id "$SIB" >/dev/null 2>&1 || true
node src/main.ts publish repository --repository "$REPO" --branch main 2>&1 | tee /tmp/pub2.out || true
grep -qiE 'diverg|not a fast-forward|remote moved|rejected' /tmp/pub2.out \
  && echo "PASS D: publish refuses a diverged remote (no force-push)" \
  || { echo "FAIL D: publish did not guard divergence"; cat /tmp/pub2.out; exit 1; }
REMOTE_DIVERGED=$(git --git-dir="$D/remote.git" rev-parse main)
git -C "$D/seed" rev-parse HEAD | grep -q "$REMOTE_DIVERGED" \
  && echo "PASS D2: remote history was NOT overwritten by publish" || { echo "FAIL D2: remote clobbered"; exit 1; }

echo "007.12 PROOF OK"
```

## Stories

- **A — Publish port + git adapter (fast-forward push).** Add a small
  publication capability (`src/publication/port.ts` + a git adapter, or extend
  the landing capability directory) that pushes `refs/heads/<branch>` from the
  managed home checkout to the repo's remote using the stored credential's
  GIT_ASKPASS path (reuse the clone plumbing). **Fast-forward-only**: pass the
  expected remote OID and use `--force-with-lease=<ref>:<expectedOID>` **only**
  as a divergence _guard_ (never a blind `--force`); a non-fast-forward push
  must surface as a typed `PublishDivergedError` carrying the remote OID.
- **B — `publish repository` CLI + use case.** New `publish repository
--repository <id> --branch <b>` command and a use case that resolves the repo,
  reads the landed local head + last-known remote OID, calls the port, and
  persists the result. Human-gated (never automatic). Id-on-stdout /
  friendly-on-stderr output contract.
- **C — Persisted, queryable publication state.** Add per-repository-target
  publication state (`unpublished` / `published@<remoteOID>` / `diverged`) to
  storage and surface it on `get repository` (human + `--json`). This is the
  answer to "did delivery reach the remote?" — a partial publish (local landed,
  push failed) must be distinguishable from a completed one.
- **D — Delivery-contract documentation.** Document, in `README.md` /
  `AGENTS.md`, that task `completed` + `candidate:landed` means **locally
  landed**, and delivery to the remote is the explicit `publish` step — so
  "local development only" is stated as a deliberate boundary, not an accident.

## Non-goals

- No automatic push on approve/land (removes the human control point; publishes
  intermediate per-task commits). Publication stays an explicit operator step.
- No force-push / remote-history rewrite on the normal path; a diverged remote
  requires a separate, explicit reconcile flow (its own future work).
- No support in this epic for non-`https-token` push topologies beyond what the
  existing credential plumbing already provides (ssh-agent / ambient may work
  incidentally, but branch-protection / fork / PR-based delivery are out of
  scope and must fail with a clear message, not silently).
- No change to landing itself (007.11 owns checkout consistency).

## Debate deltas (opencode/plan, 2026-07-22)

- **Separate `publish` command, not `land --push`** — a local land plus a
  network push cannot be atomic; combining them yields an ambiguous "landed but
  not published" failure. Splitting keeps each operation's success crisp.
- **Fast-forward default, not force-with-lease default** — a lease still permits
  intentional remote-history replacement, which is too strong for the normal
  delivery path; use it only as a divergence guard and fail loudly otherwise.
- **Clone credentials ≠ push rights** — the remote may be read-only,
  branch-protected, SSH, or a fork; publish must fail with a clear message.
- **Publication needs its own queryable state** — otherwise the daemon still
  cannot report whether delivery reached the remote (the original F1 gap).
- **Define the delivery contract first** — "completed" is not self-evidently
  "published"; Story D makes the boundary explicit.
