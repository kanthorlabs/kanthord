# 019.16 Deliver the Agent's Work

## Outcome

After Epic 019.15 the driven agent writes **real files** into its task worktree,
but nothing delivers them: the files sit uncommitted, `commitsAhead` reads 0, the
delivery block is skipped, and the task strands at `running`. This Epic closes the
**live delivery path** (never exercised before, because until 019.15 the agent
produced nothing to deliver):

1. after a cleanly completed session that wrote files, the daemon **stages and
   commits** the worktree changes onto the task branch through broker verbs, so
   `commitsAhead(taskBranch, "main") > 0`;
2. the live config constructs the commit / push / create_pr broker adapters with a
   real **`verifySetup` preflight**, so their `submit` actually runs `git` / the
   GitHub REST seam instead of returning `blocked-needs-setup` — the broker pushes
   `taskBranch` and opens a **real PR** on `kanthord-verify`;
3. the delivered task **leaves `running`** and becomes `complete` when the PR is
   merged (the existing observe-merge path).

End state: agent writes → daemon commits → broker opens the PR (human merges) →
task completes. No agent-side git tool; `bash` stays excluded.

## Decision Anchors

- **The delivery block is real but starved.** `run-loop.ts:518-542` auto-delivers
  when `verbAdapters` + `commitsAhead` are wired and `commitsAhead(taskBranch,
  "main") > 0`. Both are wired in the live path (`bootstrap-live-run.ts` spreads
  `buildRealDeps`'s `verbAdapters` + adds `commitsAhead`). The block never fires
  because the agent's files are **uncommitted** → `commitsAhead` = 0. Gap #1 in
  memory `lp-a1-delivery-gap`.
- **No commit step + no stage verb.** Between session end (`run-loop.ts:468`) and
  the delivery block there is no stage/commit. The agent has no git/bash tool (bash
  excluded by design). `git-local.ts` has `git.branch` / `git.commit` / `git.clone`
  / `git.fetch` but **no `git.add`** — and the agent's files are *new/untracked*, so
  `git commit -am` would not stage them. A stage step is required.
- **Adapters wired but dead (`verifySetup` absent).** `run-deps.ts:324-336` builds
  `makeBranchAdapter` / `makeCommitAdapter` / `makePushAdapter` / `makeCreatePrAdapter`
  with **no `verifySetup`**. All four hard-gate on it: `if (opts.verifySetup ===
  undefined) return { status: "blocked-needs-setup", ... }`
  (`git-local.ts:177,291,372`; `git-push.ts:87`; `github-create-pr.ts:126`). So even
  once `commitsAhead > 0`, the delivery submits are blocked. The live create_pr uses
  the **GitHub REST http seam** (`makeGithubHttpSeam`), not the `gh` CLI — so the
  preflight must gate on what the git+REST delivery path actually needs (git + a
  valid identity token), per memory `git-platform-integration-approach` (CLI-first,
  REST for gaps).
- **Task stuck `running`.** The status-transition block (`run-loop.ts:473`) is
  gated on `deps.workflow !== undefined`; the live path wires no `workflow`, so a
  cleanly completed task is never moved off `running`. Gap #3 in `lp-a1-delivery-gap`.
- **Observe-merge already exists.** `run-loop.ts:549-556` marks a task `complete`
  when its create_pr op reaches `merged` in `broker_completion`, keyed via
  `prOpTaskMap` (populated by `deliverSession` when `taskId` is supplied). No change
  needed — S003 only needs to move the task off `running` before merge.

## Stories

- `001-commit-the-worktree-changes.md` - after a clean session that wrote files,
  the daemon stages (new `git.add` broker verb) and commits the worktree changes
  onto `taskBranch` via broker verbs, so `commitsAhead(taskBranch, "main") > 0`; a
  session that wrote nothing makes no commit. (fix, hermetic)
- `002-wire-delivery-adapters-verifysetup.md` - the live config constructs the
  commit / push / create_pr adapters with a `verifySetup` preflight scoped to the
  git + GitHub-REST delivery path, so `submit` runs instead of returning
  `blocked-needs-setup`; the broker pushes `taskBranch` and opens a real PR on
  `kanthord-verify`. (live wiring)
- `003-task-leaves-running.md` - the delivered task transitions off `running` in
  the no-workflow live path and becomes `complete` when the PR is observed merged.
  (lifecycle)

## Verification Gate

- `npm run typecheck` exits 0; `npm test` green — a hermetic test asserts that
  after a clean session with worktree changes the daemon stages + commits onto
  `taskBranch` so `commitsAhead > 0` (and makes no commit when nothing changed); a
  test asserts a delivery adapter built with a passing `verifySetup` submits
  (not `blocked-needs-setup`); a test asserts the delivered task's status is not
  `running` and becomes `complete` on observed merge; existing run-loop / broker /
  run-deps tests pass; guard green.
- **Live proof (LP-A1, in `kanthord-core-dev`):** agent writes `slugify.mjs` + its
  test in the worktree → daemon commits (`commitsAhead > 0`) → broker pushes
  `taskBranch` and opens a **real PR** on `kanthord-verify` (base `main`) → task is
  not stuck at `running` → after the human merges, the task is `complete`.

## Non-Goals

- **No agent-side git tool** — `bash` stays permanently excluded (Epic 015); the
  agent never commits/pushes. The daemon delivers via broker verbs.
- **No change to Epic 019.15 tools or ring-1** — the six file tools and the
  write-scope gate are untouched.
- **No new merge behavior** — the daemon never merges the PR; it only observes the
  existing `broker_completion` merged state (`run-loop.ts:549-556`).
- **No full workflow/gate engine for the live path** — S003 only moves the task off
  `running`; it does not wire a real `Workflow` gate-check.
