# 019.2 kanthord run — Live Daemon Launcher (Phase-2A proof-runnable)

## Outcome

`kanthord run` exists: a single launchable Core daemon that drives a signed-off
one-repo feature end-to-end on a real slot — the missing "real caller" that
Epics 011–018 built components for but never assembled (Epic 019.1 Non-Goals
note it explicitly: *"the real caller does not exist yet — only tests call
`spawnPiSession` today"*). The launcher wires the already-built seams — slot
loader (`src/slots/repo-slot.ts`), worktree strategy (`src/slots/worktree.ts`),
keyring PAT custody (`src/git/keyring.ts`), the ring-1 write-scope hook
(`makeRing1HookAdapter`) and budget breaker (`makeBudgetBreaker`), the pi session
(`spawnPiSession`/`respawnPiSession`), the broker verb adapters
(`makePushAdapter`, `makeCreatePrAdapter`) + poller/reconcile, the scheduler
dispatch (`pollOnce`/`park`/`resume`), and the Epic 017 HTTP/JSON surface
(`src/daemon/status-server.ts`) — into one live run-loop with a thin CLI shell.

The heart is a **dependency-injected run-loop** (`src/daemon/run-loop.ts`): the
same seams the Epic 019 harness drives with doubles, now driven for real. The run
-loop is hermetically TDD-tested with injected doubles (pi-session double, broker
adapters on a temp bare remote, github double, fake clock/keyring); the thin CLI
shell (`src/cli/run.ts`) injects the real adapters and is proven only by the
maintainer's live LP1–LP4 runs — it carries no logic the run-loop tests cannot
already exercise.

Shipping this makes the Phase-2A **live proof executable**: `.agent/plan/feedback/
019-phase2a-single-repo-proof/LP-runbook.md` LP1–LP4 have no launch command until
this epic lands. This epic does **not** run LP1–LP5 (those need real credentials,
a real repo, and a human — maintainer-executed); it delivers the thing they run.

## Decision Anchors

- **Epic 019 `LP-runbook.md` "Prerequisites → BLOCKER"** (2026-07-10) — LP1–LP4
  have no launcher; only `src/cli/bootstrap.ts` and `src/cli/verify.ts` exist.
  This epic is that launcher; it is the prerequisite the runbook names.
- **Epic 019.1 Non-Goals** — "No live daemon wiring of the default manifest into a
  real spawn … the real caller does not exist yet." This epic is that real caller;
  it consumes the corrected pi tool manifest 019.1 produces (hard order: 019.1
  first, so the live session classifies pi's real tools correctly).
- **`src/daemon/boot.ts` DI pattern (Epic 009)** — "All collaborators are injected
  so the lifecycle harness can drive kill/restart deterministically (PRD §7.7)."
  The run-loop follows the same rule: every external effect (pi surface, broker
  adapters, keyring, clock, slot) is injected, so the loop is hermetically
  testable and the CLI shell is the only place real adapters are constructed.
- **PRD §4/§5** — agents have no direct effect; every git/PR effect goes through
  a typed broker verb. The launcher wires the session's produced commits into
  broker `push` then `github.create_pr`; it never lets the agent push directly.
- **Epic 019 harness (`src/harness/golden.ts`, `lifecycle.ts`)** — the hermetic
  twin of this run-loop (scheduler + broker + deploy assembled with doubles and a
  scripted workflow). The run-loop is the live equivalent; its tests reuse the
  same doubles, so this epic adds no new orchestration mechanism the harness has
  not already shaped — it replaces the scripted workflow with a real spawn.

## Stories

- `001-run-launch-and-serve.md` — `kanthord run` boots and serves: loads the slot
  + resolves the per-identity PAT from the keyring, runs `initSchema` + the
  bootDaemon ledger recovery, starts the Epic 017 status/respond HTTP surface on
  the configured loopback port, exposes the broker **hold-point** as a config flag
  (the LP4 cutpoint), installs SIGINT/SIGTERM graceful shutdown, and idles when no
  task is dispatchable. Thin CLI shell over the DI'd run-loop.
- `002-live-agent-run.md` — one run-loop tick drives dispatch → real pi session:
  the scheduler selects the signed-off dispatchable task, `spawnPiSession` runs in
  the slot worktree with the ring-1 write-scope hook and the budget breaker
  **active**, and the session's commits are collected. Live ring-1 enforcement: an
  out-of-scope write is blocked + escalated (LP2), and a budget breach halts
  **before** the breaching model call (LP3).
- `003-live-broker-delivery.md` — the session's commits are delivered only through
  the broker: `push` then `github.create_pr`, the op chain recorded in the ledger,
  the poller driving each op to a terminal state. Kill mid-`create_pr` at the
  hold-point + restart reconciles via head-branch lookup with **no duplicate PR**
  (LP4).
- `004-escalation-and-completion.md` — escalations (diff / re-planning / budget)
  surface as inbox items on the Epic 017 surface; a response through the respond
  RPC resumes or halts the parked task (`park`/`resume`); the feature reaches
  `complete` after the human merges the PR (the loop observes the terminal PR
  state); a budget halt survives a daemon restart (LP3 respawn clause).

## Verification Gate

- `npm run typecheck` exits 0; `npm test` green — the new `src/daemon/run-loop`
  and `src/cli/run` suites plus all pre-existing suites (704 baseline), no
  regression.
- **Run-loop end-to-end (hermetic, driven with doubles):** one assembled run
  drives dispatch → session (ring-1 hook + budget active) → broker `push` →
  `github.create_pr` → escalation → response-resume → `complete`, and the ledger
  shows the full op chain. Asserted on observable state (task status, ledger op
  chain, inbox items), not internals.
- **The three live-safety behaviors are proven on the run-loop with doubles:**
  out-of-scope write blocked + escalated; budget breach halts before the call and
  the halt survives a simulated restart; kill mid-`create_pr` + restart reconciles
  with no duplicate PR.
- **Launchable:** `node src/cli/run.ts --help` (or the wired `npm` script) prints
  usage and exits 0; started against a temp slot with an empty feature dir it
  boots, serves the status endpoint (loopback 200), and shuts down cleanly on
  SIGTERM — no real network, no real model call in the automated gate.
- **Gate closes when** the above are green AND the Epic 019 `LP-runbook.md`
  Prerequisites BLOCKER can be checked off (a launch command exists). LP1–LP5
  execution stays maintainer-run and is **not** part of this automated gate.

## Non-Goals

- **No LP execution.** This epic delivers the launcher; running LP1–LP5 (real
  creds, real repo, human sign-off/merge) stays maintainer-executed per the
  runbook.
- **No multi-slot / multi-repo concurrency.** One slot, `max_concurrent_tasks: 1`;
  multi-slot scheduling is Phase-2B (Epic 030).
- **No new ring-1, broker, scheduler, or pi mechanism.** Pure assembly + the live
  run-loop glue; every effect reuses an existing seam (cite Epics 013–017). A seam
  that turns out missing is a `Findings Out` entry, not new mechanism invented in
  this epic.
- **No web dashboard.** Responses go through the Epic 017 HTTP/JSON surface only
  (Epic 027 owns the SPA).
- **No deploy-chain live observation.** The one-repo feature stops at PR merge;
  deploy-chain live proof is the harness / Phase-2B.
- **No real model call or network in the automated gate.** Hermetic doubles only;
  the real pi + real GitHub path is exercised by the maintainer's LP runs.

## Findings Out

  correction to the depended-on seam (do not edit locked plan files during
  implementation). If none is found, write `none`.
