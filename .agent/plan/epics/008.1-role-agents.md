# EPIC 008.1 — Role agents & role-safe execution

> Phase 1 of the former EPIC 008 (split 2026-07-18 — the single 15-story epic was
> too large). Ships the three kanthord-native role agents and the mechanism that
> makes them safe, provable on their own via a real, supported single-role command.
> Phase 2 (the static TDD executor that orchestrates these roles into a cycle) is
> `008.2-static-tdd-executor.md`; Phase 3 (runtime-backed witnesses) is
> `008.3-runtime-verification.md`.

## Goal

Three kanthord-native role agents — **TestEngineer**, **SoftwareEngineer**,
**ReviewerEngineer** — run on the ONE shared pi-agent-core loop + pi-coding-agent
tool factories from EPIC 006, each as a declarative profile (adapted persona +
per-role tool policy + repo instructions), and each **kept honest by the
mechanism, not by prompt text**: a per-role tool policy (ReviewerEngineer gets a
read-only tool set; TE/SE get the full coding set) plus a post-run **workspace
delta diff** that DETECTS any out-of-lane write, REJECTS the run, and ROLLS the
workspace back to its pre-run state (rejected attempts leave no residue). The
ReviewerEngineer emits its verdict through an executor-owned `submit_review` tool
whose findings carry a **machine-validatable `source` reference**. A role runs as
**one bounded invocation** through a narrow `RoleRunner` port (`run(spec, ctx) →
RoleRunResult`) that knows nothing about TDD cycles — the cycle vocabulary is EPIC
008.2's. The landed EPIC 006 executor is reframed as `GenericExecutor` and the
shared pi building blocks are extracted so BOTH it and the role agents set the pi
loop up once, never a second bespoke loop. A thin, **permanent, operator-facing**
`daemon run role <role>` command drives one role — through the SAME composition
root, `RoleRunner`, and provider-session builder the 008.2 executor will use —
against a **markdown task file** (parsed by the core graph codec) in a workspace,
making role identity + lane exclusion + structured review observable end to end.

Role-safety here is **detection + rejection + rollback**, NOT OS containment: a
post-run diff catches an out-of-lane write and undoes it, but bash, `.git`
mutations, and symlink/ignored-file escapes are out of its reach — true
confinement (PathPolicy, env allowlist, redaction) is EPIC 009.1, which composes
WITH this epic's workspace service rather than re-owning it. This epic is the
integrity core of the TDD executor: it proves the safety mechanism BEFORE any
orchestration exists, and it owns the role-lane **policy**, so **008.1 ships
before 009.1**.

## Verification Gate

Gates: `npm run verify` (hermetic — no network. The role registry resolves the
three profiles and the per-role tool manifest is asserted: ReviewerEngineer has
only read/grep/find/ls; TestEngineer and SoftwareEngineer have the full EPIC 006
coding set. `RoleRunner` runs one bounded role against EPIC 006's
`FakeSessionFactory` with scripted turns — real pi `Agent`, real tools, real git
in temp dirs. Lane enforcement is proven THROUGH the real `daemon run role`
command against a deterministic replay provider (not only unit tests): a
TestEngineer scripted to write a non-test file is REJECTED (exit non-zero) and the
workspace is ROLLED BACK to the pre-run snapshot; a SoftwareEngineer scripted to
write a test file is REJECTED + rolled back (the SE can never make a test pass by
editing it — the one hard boundary); a ReviewerEngineer scripted to produce any
diff is REJECTED; and each role's lane-compliant scripted run passes with its
delta attributed to that run alone. Workspace-delta attribution is unit-tested:
the snapshot captures the FULL working tree including untracked files
(`git status --porcelain -uall` inventory or stage-all-then-`write-tree`), so a
role's delta excludes a prior role's pre-existing changes. Path classification is
unit-tested state-by-state: renames, deletions, symlinks, untracked, overlapping
globs, submodules, ignored files. `submit_review` schema + `source` are validated:
a finding whose `source` RESOLVES to a real referent (task field / AC index /
instruction path+line / loaded-instruction id) is accepted, a free-form/invalid
`source` is rejected, a missing submission is a structured error;
`verdict:"rejected"` requires ≥1 `action:"yes"` finding. `parseTaskFile` turns a
markdown task into a task value via the core codec. `generic@1`'s EPIC 006 tests
keep passing after the `GenericExecutor` reframe — the regression anchor.)

Proof: two parts. **Part A** is the correctness gate — deterministic (replay
provider), asserts lane REJECTION + rollback at the program level. **Part B** is
secondary live-provider smoke — real model, nondeterministic, so it asserts only
that each role runs and stays in-lane, reading each run's OWN delta from the
command's JSON output (never the cumulative workspace status).

```bash
set -euo pipefail

# throwaway target repo + a single-task markdown file (the core codec parses it;
# no `executor:` field — role-run never creates a task, so the ExecutorCatalog
# gate is not involved and tdd@1 need not be registered yet).
REPO_DIR="$(mktemp -d)/greet"; mkdir -p "$REPO_DIR" && git -C "$REPO_DIR" init -q
: > "$REPO_DIR/AGENTS.md"; git -C "$REPO_DIR" add -A && git -C "$REPO_DIR" commit -qm init
cat > /tmp/greet-task.md <<'MD'
---
kind: task
ref: greet
objective: backend
title: greeting module with tests
---
# Instructions
Create src/greet.js exporting greet(name) returning 'hello <name>', with a node:test unit test.
# Acceptance Criteria
- [ ] a unit test covers greet('world') === 'hello world'
- [ ] node --test passes
MD

# ---- Part A: deterministic lane REJECTION + rollback (the correctness gate) ----
WS_A="$(mktemp -d)/ws-a"; cp -R "$REPO_DIR/." "$WS_A"
# replay script drives the SoftwareEngineer to write a TEST file (out of its lane)
cat > /tmp/se-violation.replay <<'RP'
- tool: write
  path: src/greet.test.js
  content: "test('x', () => {})"
RP
if node src/main.ts daemon run role software-engineer@1 --task-file /tmp/greet-task.md \
     --workspace "$WS_A" --provider replay --script /tmp/se-violation.replay --json > /tmp/a.json; then
  echo "FAIL: out-of-lane write was NOT rejected"; exit 1
fi
node -e 'const r=require("/tmp/a.json");process.exit(r.outcome==="lane-violation"&&r.lane.offendingPaths.includes("src/greet.test.js")?0:1)' \
  || { echo "FAIL: wrong rejection payload"; exit 1; }
test -z "$(git -C "$WS_A" status --porcelain -uall)" || { echo "FAIL: workspace not rolled back"; exit 1; }
echo "Part A ok: SE test-lane write rejected + workspace rolled back"

# ---- Part B: live-provider smoke (secondary; assert per-run delta only) ----
: "${OPENAI_API_KEY:?set your provider key for the live smoke}"
WS_B="$(mktemp -d)/ws-b"; cp -R "$REPO_DIR/." "$WS_B"

node src/main.ts daemon run role test-engineer@1 --task-file /tmp/greet-task.md \
  --workspace "$WS_B" --provider gpt-5.5 --json > /tmp/te.json
node -e 'const r=require("/tmp/te.json");process.exit(r.outcome==="succeeded"&&r.changed.every(p=>/\.(test|spec)\./.test(p))&&r.changed.length>0?0:1)' \
  || { echo "FAIL: TE delta not confined to the test lane"; exit 1; }

node src/main.ts daemon run role software-engineer@1 --task-file /tmp/greet-task.md \
  --workspace "$WS_B" --provider gpt-5.5 --json > /tmp/se.json   # reads TE's test off the shared tree
node -e 'const r=require("/tmp/se.json");process.exit(r.outcome==="succeeded"&&r.changed.every(p=>!/\.(test|spec)\./.test(p))?0:1)' \
  || { echo "FAIL: SE delta touched a test file"; exit 1; }

node src/main.ts daemon run role reviewer-engineer@1 --task-file /tmp/greet-task.md \
  --workspace "$WS_B" --provider gpt-5.5 --json > /tmp/re.json
node -e 'const r=require("/tmp/re.json");const ok=["task-field","ac-index","instruction-loc","instruction-id"];process.exit(r.outcome==="succeeded"&&r.changed.length===0&&["accepted","rejected"].includes(r.review.verdict)&&r.review.findings.every(f=>ok.includes(f.sourceKind))?0:1)' \
  || { echo "FAIL: reviewer not read-only / bad citation / bad verdict"; exit 1; }

echo "008.1 ok: rejection+rollback proven (A); TE/SE/RE in-lane with cited review (B)"
```

## Stories

1. **Shared agent-core reuse (`GenericExecutor`).** Frame the landed executor as
   `GenericExecutor` (a thin wrapper over pi-coding-agent's Generic agent) and
   extract the shared pi building blocks (loop setup, provider session,
   instruction loading, evidence normalization) into an adapter-private core
   consumed by BOTH `GenericExecutor` and the role agents — the pi loop is set up
   once per agent, never a second bespoke loop. `generic@1` behavior is unchanged
   (the regression anchor).

2. **Role profile CONTRACT + registry (the executable authority envelope).** Each
   role — `test-engineer@1`, `software-engineer@1`, `reviewer-engineer@1` — is a
   VERSIONED declarative profile whose **authority is executable code, not prompt
   text**: a fixed tool policy reference + a fixed lane policy reference. The prompt
   is assembled from **separate labelled sections, NOT one override stack** (debate
   B4): (i) ENFORCED policy (code — never in the prompt); (ii) TRUSTED invocation
   facts (role id, task id, workspace, the trusted lane MAP, ACs) that no text can
   replace; (iii) base role guidance (persona, ADAPTED from `.claude/agents/*.md` —
   keep the boundary + authority chain, rewrite the harness lines for the pi
   daemon); (iv) repo instructions via `InstructionLoader` (scoped, source-located);
   (v) task material. **The invariant (debate B1): untrusted context may shape the
   PROPOSED WORK, but can never change executable authority — tools, tool
   arguments, path/lane classification, resource/credential bindings, workspace
   roots, verdict schema, or acceptance identity.** Conflicts are explicit:
   enforced policy always wins; trusted facts cannot be overridden by text; current
   task data supersedes stale advisory context for factual work only; an ambiguous
   instruction conflict yields a structured block / review finding, never silent
   prompt-order precedence. Role refs live in a role registry; they are NOT executor
   refs and NOT task-assignable; adding a role is a new module + a registration line
   (OCP). NO generic memory or project-augmentation framework here (008.5 / 008.4).

3. **TestEngineer (`test-engineer@1`) executable contract.** RED behavior: writes a
   failing test. **Test-lane writes ONLY** (never production code). The contract is
   specified, not "full tool set": allowed tool operations + argument constraints,
   the path classes it may write, and how shell/subprocess/formatter writes are
   treated. A forbidden diff is rejected + rolled back (story 7). Proof case via
   `daemon run role test-engineer@1`.

4. **SoftwareEngineer (`software-engineer@1`) executable contract.** GREEN behavior:
   makes the test pass. **Non-test-lane writes ONLY** (never a test file — the SE
   can never make a test pass by editing it). Explicit, specified treatment of
   toolchain / setup / generated / shared files (the debate flagged "full coding
   set" as too weak): allowed operations + argument constraints, path classes,
   symlinks / generated / renames / indirect writes. Forbidden diff rejected +
   rolled back. Proof case via `daemon run role software-engineer@1`.

5. **ReviewerEngineer (`reviewer-engineer@1`) executable contract + `submit_review`.**
   Read-only operations (read/grep/find/ls only); **empty-diff enforced**. The
   executor-owned `submit_review` tool has a JSON schema (findings: severity
   blocker|suggestion, action yes|no, file, note, **source**; verdict accepted|
   rejected). `source` is a machine-validatable **reference** — one of a defined set
   (task field, AC index, instruction path + line range, loaded-instruction id)
   that must RESOLVE to a real referent; it proves the reference EXISTS, NOT that
   the reasoning is justified (and no injected text may redefine what counts as a
   valid citation). `verdict:"rejected"` requires ≥1 `action:"yes"`. The accepted
   payload IS the typed `ReviewFindings` artifact on `RoleRunResult`; missing/invalid
   submission → structured error. (The verdict-driven TDD re-route is 008.2.) Proof
   case via `daemon run role reviewer-engineer@1`.

6. **`RoleRunner` port (one bounded role invocation).** A narrow internal port for
   ONE role run: `run(spec: RoleRunSpec, ctx): Promise<RoleRunResult>` — no task
   lifecycle, no proposal freezing, no TaskResult, and **no TDD-step vocabulary**
   (that layers on top in 008.2). `RoleRunSpec` carries the role ref, the task
   value (requirement + ACs), the workspace, and the resolved provider context;
   `RoleRunResult` carries the outcome, evidence, the attributed workspace delta
   (`changed[]`), and typed artifacts `{ kind; schemaVersion; payload }` (for RE,
   the `submit_review` payload).

7. **Cross-role lane policy + a reusable workspace service.** TWO clearly separated
   parts. (a) **Policy owned by this epic — from a TRUSTED lane MAP, never from
   prompt text (debate B1/#6):** lanes are positive-match per role, a pure
   unit-tested predicate over trusted policy data — no prompt layer (repo, project,
   memory, task) may classify a path as test / production / toolchain / exempt.
   test-engineer = test files only; software-engineer = everything NOT a test file,
   incl. toolchain/setup; reviewer = empty delta. The one hard boundary is mutual
   exclusion on the test lane. (b) **Mechanism as shared infrastructure:** a
   workspace service takes a pre-run FULL-working-tree snapshot (tracked AND
   untracked via `git status --porcelain -uall` / stage-all-then-`write-tree`),
   computes the run's attributed delta, applies the predicate + tool-argument
   constraints, and on violation REJECTS + ROLLS BACK to the snapshot. Path
   classification is one documented rule (normalized repo-relative paths; renames,
   deletions, symlinks, overlapping globs, submodules, ignored files). Tests cover
   renames, symlinks, generated files, and shell/indirect writes. **Honest limit
   (debate S1):** this is DETECT + reject + rollback of WORKSPACE STATE — it does
   NOT prevent external side effects, secret reads, network calls, or commands that
   run BEFORE rollback; that confinement is EPIC 009.1's PathPolicy + env allowlist,
   which COMPOSE with this one infrastructure owner (009.1 does not re-own it). The
   `beforeToolCall` gate is advisory only.

8. **`daemon run role` command — a supported operator capability (thin CLI over a
   `RunRole` use case).** The **composition root** wires the provider-session
   builder + `RoleRunner` + the workspace service into a `RunRole` use case; the
   CLI only parses input, calls the use case, and formats output (no adapter
   construction, no execution policy in the CLI — hexagonal). `daemon run role
<role> --task-file <path.md> --workspace <path> --provider <model|replay
--script <file>> [--json]`: reads the markdown bytes at the edge, parses them
   into a task value via the **core graph codec** (`parseTaskFile`), IGNORES any
   `executor:` field (it never creates a task), renders the role prompt via EPIC
   006's `renderTaskPrompt`, builds a `ProviderSession` from `--provider` (a real
   model, or a deterministic `replay` provider for tests/Proof) + the env
   credential, runs the role via `RunRole`, and prints the `RoleRunResult`
   (outcome + attributed `changed[]` + lane verdict + any `submit_review` payload)
   — JSON with `--json`. A prior role's output reaches the next role only through
   the shared `--workspace` tree (SE reads TE's test off disk) — the same
   tree-as-handoff 008.2's executor relies on. Reuses the SAME execution service
   and provider-session builder as the future executor; only the INPUT source
   differs (flag/file vs a DB-resolved Resource) — a legitimate operator/debug
   entry point, not a parallel execution path.

9. **End-to-end + hermetic suite** per the Verification Gate: the three roles run
   lane-compliant on scripted fakes and (Part B) against a real provider; a
   scripted out-of-lane write is rejected AND rolled back through the real CLI
   (Part A, deterministic); the reviewer citation resolution is validated; the
   `generic@1` regression holds.

## Decisions

- **`RoleRunner`, not `RoleStepRunner`.** The seam runs one bounded role
  invocation and carries no TDD-cycle ("step") vocabulary — "step" is an 008.2
  concept. DTOs are `RoleRunSpec` / `RoleRunResult`.
- **Single-role execution is an INTENTIONAL, supported operator capability**
  (Ulrich, D5), not a proof scaffold — that is what makes 008.1 a valid standalone
  epic rather than "just contracts consumed by 008.2" (the debate's own validity
  condition). `daemon run role` stays in the product as a role-debugging tool.
- **008.1 owns the role-lane POLICY; the snapshot/diff/rollback/normalization
  MECHANISM is shared infrastructure** with 009.1's PathPolicy (revises the
  earlier "008.1 owns lane diff, 009.1 hardens around" framing — cleaner to name
  one infra owner). This still lets 008.1 ship BEFORE 009.1.
- **Role-safety = detect + reject + rollback, NOT containment.** A post-run diff
  undoes an out-of-lane write; bash / `.git` / symlink / ignored-file escapes are
  EPIC 009.1's PathPolicy. The epic claims only what the mechanism delivers.
- **Role-run reuses production composition + provider-session builder; only the
  input source differs** (markdown file + `--provider` flag + env credential vs a
  DB-resolved Resource). File-driven input is deliberately chosen for testability
  (a fixture + a scripted replay provider, no DB seeding) and is a supported
  operator path, not a divergent execution engine.
- **Parse in core, file I/O at the edge.** The CLI reads the file bytes; the core
  codec parses the string → task value. Consistent with the zero-I/O core rule.
- **`submit_review.source` proves reference resolution, not justified reasoning**
  (debate S4) — the enforceable claim is "the citation resolves to a real
  referent," nothing stronger.
- **A role's authority is a versioned EXECUTABLE ENVELOPE that no text can modify**
  (debate 2026-07-19 B1). "Tools vs knowledge" is too narrow: authority also lives
  in tool arguments, path/LANE classification, resource/credential bindings,
  workspace roots, verdict schema, and citation validity. Untrusted context (repo /
  project / memory / task) shapes PROPOSED WORK only; it never changes the envelope.
  Lane classification comes from a TRUSTED lane map, never from a prompt layer.
- **Prompt = separate labelled sections with explicit conflict rules, NOT an
  override stack** (debate B4). `persona<repo<project<memory<task` as precedence is
  rejected — it would turn untrusted text into a policy hierarchy and give memory
  more authority than repo. Enforced policy always wins; ambiguous conflicts →
  structured block / review finding.
- **Per-role stories are justified by DIFFERING EXECUTABLE CONTRACTS**, not persona
  text (debate S1): TE (RED / test-lane), SE (GREEN / non-test + specified
  toolchain treatment), RE (empty-diff / valid verdict). The shared profile shape +
  registry stay in one foundation story.
- **Role capability LAYERING is split by epic** (debate 2026-07-19): base role
  contracts here (008.1); PROJECT-produced role context lands WITH its consumer in
  onboarding (008.4 — no empty seam here, debate B3); USER-managed role knowledge is
  a new epic **008.5** (untrusted advisory only); agent-ACCUMULATED memory is
  deferred to a later epic until a poisoning/moderation model exists (debate B2/B5).

## Non-goals

- **No TDD cycle, no orchestration, no `TddExecutor`** — no RED→GREEN sequencing,
  no pure transition function, no step contracts with witness expectations. That
  is EPIC 008.2. `RoleRunner` runs ONE role; the harness never sequences roles.
- **No witness / test execution as a pass/fail gate, no `ProjectCapacity`
  resolution, no acceptance verification** — 008.2. (Corollary: the
  SE-may-edit-toolchain lane decision stands, and the risk that a toolchain edit
  falsifies a GREEN is guarded in 008.2 by suite-scoped, RED-must-fail-first
  witness integrity — not by narrowing the 008.1 lane. Debate S3, forwarded.)
- **No proposal commit, no confirmation/park, no human gate, no `TaskResult`
  lifecycle** — the harness is one-shot: run, report, exit. 008.2 owns the
  executor lifecycle and the confirmation protocol.
- **No runtime / app-up witnesses** — EPIC 008.3.
- **No per-role AIProvider/model** — the single `--provider` flag this epic.
- **No OS sandboxing / ring-1 hardening / attempt identity / SecretRedactor /
  env allowlist / PathPolicy** — EPIC 009.1 adds those around this epic's role
  policy and composes with its workspace service.
- **No user-managed role knowledge / memory** — the per-role knowledge store +
  `project`/role-scoped inject/list/remove CLI is EPIC **008.5** (untrusted
  advisory context; never tools/creds/lanes/policy). Role personas here ship with
  only base guidance + repo instructions + task material.
- **No project role-augmentation seam** — an empty hook with no consumer fails the
  "every story serves the Proof" rule (debate B3); the augmentation port lands WITH
  onboarding in EPIC 008.4.
- **No agent-accumulated / self-written memory** — deferred to a later epic until
  its prompt-poisoning + moderation model is designed (debate B5).

## Prerequisites (external maintainer tasks — must land before 008.1 executes)

Both are maintainer prerequisites (mechanical, touch lane-forbidden files —
migration / CLI / wide tests / graph format), executed directly by the human +
assistant in normal sessions, NOT dispatched through `/work`. Neither is an
observable-value story of this epic, so neither inflates its scope (debate S2/B10).

- **`agent → executor` rename (shared by 008.1/2/3).** `Task.agent →
Task.executor` (domain + a new migration column), `AgentCatalog →
ExecutorCatalog` / `UnknownAgentError → UnknownExecutorError`, `AgentRunner →
Executor` / `AgentRunnerResolver → ExecutorResolver` / `PiAgentRunner →
PiExecutor`, the `--agent → --executor` CLI flag, and EPIC 007's `store-graph`
  frontmatter key. No shim for the old `agent:` key (safe — no 007 graph files
  exist yet). The create-time `ExecutorCatalog.has` gate covers every
  task-creation path. No behavior change — `generic@1`'s EPIC 006 tests keep
  passing (renamed).
- **Graph codec moved to core + single-file task parse.** The EPIC 007
  export/import codec is refactored OUT of the CLI adapter into core
  (`app/graph/`), reversing 007's "codec lives only in the CLI adapter" (B8). It
  MUST expose a single-file `parseTaskFile(content) → task value` (the existing
  private per-file `buildTask` already does this work — it just needs to be a
  public core function), not only the whole-package `parseGraphPackage(dir)` which
  requires an initiative file. File reading stays at the CLI edge; parsing is core
  (zero-I/O).
