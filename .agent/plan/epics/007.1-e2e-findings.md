## OPEN findings → EPIC 007.1 backlog

### Group A — Observability / logging (Ulrich's focus)

- **A1 [major] daemon stdout is silent.** `daemon run` logs nothing operational
  (no claimed/started/verifying/completed/failed). All trace lives in the
  `events` table; you must poll it to know anything happened.
  _Suggest:_ structured lifecycle logging to stdout (task claimed, agent
  started, verification running + result, outcome + reason).
- **A2 [major] no CLI to view a task result.** `summary`, verification output,
  `commit_sha`, diff all sit in `task_results`, but `get task` surfaces none of
  it. Inspecting an agent's work needs raw SQLite.
  _Suggest:_ `get task --result` / a `show task` command that prints summary,
  verification evidence, commit, and files changed.
- **A3 [minor] progress throttled.** `agent.progress` capped at 1 per 5s
  (`pi.ts`), so the feed undercounts tool calls; real edits were missed.
- **A4 [minor] verification invisible in the feed.** No event when `npm test`
  runs or its result — only in `task_results.evidence`.
  _Suggest:_ emit a `task.verification` event (command, exitCode, tail).
- **A5 [minor] `get task --json` omits `context`** (shows `undefined` though
  `task_context` rows exist) — can't confirm wiring without the DB.
- **A6 [minor] no turn/token/cost logging** — no visibility into how many turns
  or how much quota a run consumed.
- **A7 [minor] `task_results.base_commit` is null** though `commit_sha` is set.

### Group B — Verification scoping / agent scope discipline

- **B1 [major] whole-suite verification causes out-of-scope edits.** `npm test`
  runs _all_ tests, so a **pre-existing failing test** (`test/truncate.test.mjs`
  was broken on `main`: expected `'naïv...'`, correct is `'naïve...'`) forced
  the agent to edit a file outside its task to get the suite green. Here the fix
  was correct and disclosed in the summary — but an agent silently editing
  unrelated tests to pass verification is a real safety risk.
  _Suggest:_ scope verification to task-relevant tests; and/or record + warn on
  files changed outside the task's declared outputs; and/or forbid edits to
  pre-existing test files unless the task says so.
  **Ruling (Ulrich):** the EPIC 008 onboarding workflow must explicitly ask the
  user (via a command) **how to test a single file**, so a task tests only what
  it changes and fixes only what it touches. A 2-year-old failing test is NOT
  ours to fix — expanding scope to "make the whole suite green" without context
  is too risky. Verification should target the changed files, not the whole
  suite.

### Group C — Work-graph orchestration

- **C1 [major] `import graph` drops task context.** The markdown work-graph
  format has no context field (frontmatter = kind/id/ref/objective/title/agent/
  depends-on; body = Instructions/Acceptance Criteria/Verification). Imported
  tasks have no repository/ai_provider/credential binding → **not runnable** by
  an agent. Only `create task --context type=<id>` sets context. This defeats
  the EPIC 007 import path for real runs.
  _Suggest:_ add context (and its refs) to the graph codec + import/apply.
  **Ruling (Ulrich):** 007.1 must **debate** the right way to bind context on
  import — set a default, ask the user to select, or another mechanism — so that
  `import` produces tasks that are immediately **ready for development** (all
  required context resolved).
- **C2 [major] same-repo task outputs don't chain.** Every task clones `home` at
  `main`; a completed task commits to `kanthord/<taskId>` and is never promoted
  to `home/main`. A dependent same-repo task starts from `main` and cannot see
  the prior task's code. (Ulrich: fix later — the "non-merge branch" issue.)
  _Suggest:_ promote/merge accepted task output into `home` (or base dependent
  workspaces on their dependencies' branches).
  **Ruling (Ulrich):** the remote-git case will be handled later by the **pr@1**
  workflow. For **local**, we need a flow that **merges a task's branch back to
  `main`** (in the home repo) once done, so downstream task dependencies see the
  prior work. This local-merge flow is the C2 fix for 007.1.

### Group D — Resource management / config

- **D1 [minor] no `update-resource` CLI.** Changing a model (gpt-5.5 →
  gpt-5.6-terra) required a direct DB edit; only project/initiative/objective
  have `rename`.
- **D2 [minor] repo clone has no credential plumbing.** `buildRemoteUrl` uses a
  hardcoded `https://github.com/<org>/<name>.git`; private repos need a manual
  pre-clone with a token. No repo-credential binding feeds the clone.
- **D3 [minor] no model validation at `create ai-provider`.** An unknown model
  is stored happily and only fails at daemon run (`UnknownModelError`) — a late,
  confusing failure.
