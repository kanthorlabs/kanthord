## OPEN findings → EPIC 007.1 backlog

### Group A — Observability / logging (Ulrich's focus)

- **A1 [major] daemon stdout is silent.** `daemon run` logs nothing operational
  (no claimed/started/verifying/completed/failed). All trace lives in the
  `events` table; you must poll it to know anything happened.
  _Suggest:_ structured lifecycle logging to stdout (task claimed, agent
  started, verification running + result, outcome + reason).
- **A2 [major] no CLI to view a task result.** `summary`, verification output,
  `commit_sha`, diff all sit in `task_results`, but `get task` surfaces none of
  it. Inspecting an agent's work needs raw SQLite.
  _Suggest:_ `get task --result` / a `show task` command that prints summary,
  verification evidence, commit, and files changed.
- **A3 [minor] progress throttled.** `agent.progress` capped at 1 per 5s
  (`pi.ts`), so the feed undercounts tool calls; real edits were missed.
- **A4 [minor] verification invisible in the feed.** No event when `npm test`
  runs or its result — only in `task_results.evidence`.
  _Suggest:_ emit a `task.verification` event (command, exitCode, tail).
- **A5 [minor] `get task --json` omits `context`** (shows `undefined` though
  `task_context` rows exist) — can't confirm wiring without the DB.
- **A6 [minor] no turn/token/cost logging** — no visibility into how many turns
  or how much quota a run consumed.
- **A7 [minor] `task_results.base_commit` is null** though `commit_sha` is set.

#### Group A — design direction (debate-hardened 2026-07-19; Ulrich rulings)

Principle: **events = authoritative but sensitive (never leaves the machine);
a shareable artifact must carry debugging info only, nothing sensitive.** The
debate's key correction: a typed field allowlist limits _keys, not values_ —
strings like repo/branch/command/toolName/errorClass/commitSha can all carry
secrets. Safety = eliminate uncontrolled strings + runtime schema validation +
treat all human-assigned metadata as private by default.

Three layers:

1. **Private audit journal** = the SQLite `events` table. Authoritative,
   local-only, may hold sensitive detail. **Not throttled at capture** (A3 fix:
   throttling belongs only in the _display_ of the events feed, never in the
   record). Records every lifecycle transition, verification start/end, turn,
   and tool occurrence.
2. **Rich local logging** = the daemon writes readable lifecycle logs to stdout
   (A1 fix) with real names — this is the owner's own machine, so sensitive is
   fine. Local inspection CLIs (A2 `show task` / `--result`, A5 context) read the
   private journal directly.
3. **Support export** = a new `kanthord diagnostics export` command — the
   **single sanitization boundary** (Ulrich: _sanitize at export_, not at write).
   It projects the private journal into a **closed safe-facts schema** and is the
   only shareable artifact:
   - fields: `schemaVersion, sessionRef, taskRef, seq, timestamp, kind,
outcomeCode, reasonCode, toolCategory, exitClass, durationMs, turns,
tokensIn, tokensOut` (A4 verification → `verifierKind/exitClass/durationMs/
timedOut`, never command or output; A6 turns/tokens are numeric → safe).
   - **no** repo, branch, commit, command text, paths, error names/messages,
     provider strings, or free-form text.
   - `taskRef`/`sessionRef` are **opaque random** refs (NOT hashes — low-entropy
     names are guessable); the private journal stores the same refs for local
     correlation. `seq` gaps reveal dropped records (sinks can't be atomic).
   - errors map to a **closed enum**; unknown → `internal_unclassified`.
   - export re-validates every record against the exact schema, **previews**
     record kinds + field names before sharing, writes an owner-only, bounded,
     rotated artifact.
   - enforcement: closed record kinds/enums, no `message`/`metadata`/
     `Record<string,unknown>`, runtime exact-schema validation (reject unknown
     keys), explicit field-by-field serialization (never spread), import
     restriction so only the observability module produces the export, canary
     tests proving prompts/paths/commands/tool-args/errors/credentials never
     reach the safe artifact. `debug` raises frequency, never sensitivity.

### Group B — Verification scoping / agent scope discipline

- **B1 [major] whole-suite verification causes out-of-scope edits.** `npm test`
  runs _all_ tests, so a **pre-existing failing test** (`test/truncate.test.mjs`
  was broken on `main`: expected `'naïv...'`, correct is `'naïve...'`) forced
  the agent to edit a file outside its task to get the suite green. Here the fix
  was correct and disclosed in the summary — but an agent silently editing
  unrelated tests to pass verification is a real safety risk.
  _Suggest:_ scope verification to task-relevant tests; and/or record + warn on
  files changed outside the task's declared outputs; and/or forbid edits to
  pre-existing test files unless the task says so.
  **Ruling (Ulrich):** the EPIC 008 onboarding workflow must explicitly ask the
  user (via a command) **how to test a single file**, so a task tests only what
  it changes and fixes only what it touches. A 2-year-old failing test is NOT
  ours to fix — expanding scope to "make the whole suite green" without context
  is too risky. Verification should target the changed files, not the whole
  suite.

#### Group B — design direction (debate-hardened 2026-07-19; Ulrich decision)

> **FOLDED INTO EPIC 008 (2026-07-19).** Group B is NOT in EPIC 007.1. Its
> direction is folded into `008.1-role-agents.md` (change-authority MECHANISM:
> task-declared path classes, hard write-deny, pre-existing-test-edit boundary)
> and `008.2-static-tdd-executor.md` (GATE: acceptance/verification immutable,
> collateral-damage whole-suite structured comparison, approval-required
> scope-expansion via the confirmation protocol) — each under a "## Group B fold"
> section, ready to reslice into stories when those epics start.

Reframe: scoped/single-file testing is a **feedback primitive, not the
correctness gate**. The decisive control is **independent verification
ownership + enforced change authority** — the executor must not redefine the
gate, edit protected evidence, or widen its file scope.

Four separated concerns:

- **What the executor sees:** focused, task-relevant checks (the onboarding
  single-file command) — fast feedback only.
- **What determines correctness:** acceptance checks owned outside the executor;
  the executor cannot modify them or the verification commands.
- **What detects collateral damage:** the **whole suite runs after a
  write-freeze, outside the executor** (Ulrich decision 2026-07-19 — refines the
  "not the whole suite" ruling). Base-vs-candidate **structured outcome
  comparison** (test IDs + statuses), not exit-code delta: reject NEW failures,
  missing/disabled tests, reduced counts, or changed verification config.
  Pre-existing failures are surfaced to the human, **never handed to the agent
  to fix** and non-blocking for the task. Flaky/uncomparable suites → advisory /
  human review.
- **What changes are allowed:** author declares scope **before** execution as 3
  path classes — _allowed_ (narrow globs), _forbidden_ (creds, CI config,
  fixtures, snapshots, pre-existing/unrelated tests, generated artifacts;
  hard-denied at the write tool), _approval-required_ (everything else). The
  executor may **request** expansion (reason + files) then **stops** for
  approval; it cannot widen scope itself. **New files must also match an allowed
  path** (not auto-in-scope).

Enforcement: any undeclared/forbidden edit **prevents auto-completion** (not a
later advisory diff). Protected artifacts (acceptance tests, runner config,
verification commands, baseline) are outside the executor's writable scope.
Test-edit policy by task type: feature tasks add tests only in allowed paths and
**cannot edit pre-existing tests without approval** (the truncate incident);
a "test-fix" task explicitly names the existing tests it may touch. `tdd@1`
role separation is defense-in-depth — test-author freezes acceptance tests, impl
role gets them read-only, enforced by write permissions (not convention).

Onboarding (008) captures test **primitives** (one file / one package / full
suite) + protected & approval-required paths — capabilities, NOT an inferred
source↔test mapping (which is structurally fragile).

### Group C — Work-graph orchestration

- **C1 [major] `import graph` drops task context.** The markdown work-graph
  format has no context field (frontmatter = kind/id/ref/objective/title/agent/
  depends-on; body = Instructions/Acceptance Criteria/Verification). Imported
  tasks have no repository/ai_provider/credential binding → **not runnable** by
  an agent. Only `create task --context type=<id>` sets context. This defeats
  the EPIC 007 import path for real runs.
  _Suggest:_ add context (and its refs) to the graph codec + import/apply.
  **Ruling (Ulrich):** 007.1 must **debate** the right way to bind context on
  import — set a default, ask the user to select, or another mechanism — so that
  `import` produces tasks that are immediately **ready for development** (all
  required context resolved).
- **C2 [major] same-repo task outputs don't chain.** Every task clones `home` at
  `main`; a completed task commits to `kanthord/<taskId>` and is never promoted
  to `home/main`. A dependent same-repo task starts from `main` and cannot see
  the prior task's code. (Ulrich: fix later — the "non-merge branch" issue.)
  _Suggest:_ promote/merge accepted task output into `home` (or base dependent
  workspaces on their dependencies' branches).
  **Ruling (Ulrich):** the remote-git case will be handled later by the **pr@1**
  workflow. For **local**, we need a flow that **merges a task's branch back to
  `main`** (in the home repo) once done, so downstream task dependencies see the
  prior work. This local-merge flow is the C2 fix for 007.1.

#### Group C — design direction (debate-hardened 2026-07-19)

**C1 — import context binding.** No ambient hierarchical resource defaults (a
project holds multiple providers/accounts; resource NAMES aren't safe ids — no
unique constraint, delete+recreate changes authority; entity-stored defaults
make imports non-reproducible). Instead:

- Package declares **binding aliases** (`source: repository`, `model:
ai_provider`, `model-auth: credential`); objectives assign context to aliases
  (package-local lexical default); tasks override per-key.
- Import maps aliases → concrete resource **ids** explicitly:
  `import graph … --bind source=<repo-id> --bind model=<aip-id> …`. IDs
  authoritative; names only as CLI shorthand, rejected on 0/multiple. Credentials
  by id/alias, **never value**.
- Validate provider↔credential compatibility AND each task's effective context
  against its **executor's required/forbidden binding set** (integration ops
  don't need repo+provider+cred), all before the graph transaction. Fail-fast
  non-interactive default (one complete error report); `--interactive` optional.
  Graph format version bumps (context changes round-trip + apply semantics —
  context is currently outside graph CAS; absent context preserves existing
  bindings, so inheritance must not silently replace live bindings).

**C2 — branch integration + the abstraction question.**

- **Reject** `merge@1`-as-executor: executors get `run(task,context)` only (no
  dep branches), write isolated `kanthord/<taskId>`, never touch home; landing
  mutates the canonical branch and needs lock/ancestry/conflict/recovery; remote
  PR is async — a one-shot awaited Promise is the wrong lifecycle.
- **Reject** an explicit integration _node_ (graph boilerplate encoding storage;
  every edge must remember to target it).
- **Redefine `completed`:** for a same-repo dependency, A `completed` must mean
  A's accepted change is on the canonical branch B clones. Landing is part of
  **approval/completion** (today's bug: `approveTask` only promotes the proposal
  ref in the workspace — never lands to main). Keep `main`/configured branch
  canonical — no branch-basing dependents (breaks on multi-parent/multi-repo/
  rebased branches).
- **Landing protocol:** per-repo+branch lock; durable candidate metadata
  (repoId, baseSHA, candidateSHA, ref, target, state); ancestry → fast-forward
  or merge → typed conflict (conflict ≠ executor failure; dependents blocked);
  crash-idempotent recovery from SHAs. `generic@1`: no-change completes; changed
  → accepted-candidate, human approval before landing by default (optional
  `auto-approve`). `tdd@1` already fits (candidate → awaiting_confirmation →
  approve → land → completed).
- **Abstraction answer:** neither a shared local/remote merger port nor dedicated
  disconnected flows. Share the **integration lifecycle as domain state/artifacts**
  — `Executor → ChangeCandidate → Acceptance → Integration → LandedChange` — not
  an operational interface. Build a narrow **local** port now
  (`RepositoryLanding.land(candidate): LandingResult`, synchronous git); do NOT
  force pr@1 to implement it. pr@1 (resumable workflow/executor) is designed only
  when its real shape exists.

### Group D — Resource management / config

- **D1 [minor] no `update-resource` CLI.** Changing a model (gpt-5.5 →
  gpt-5.6-terra) required a direct DB edit; only project/initiative/objective
  have `rename`.
- **D2 [minor] repo clone has no credential plumbing.** `buildRemoteUrl` uses a
  hardcoded `https://github.com/<org>/<name>.git`; private repos need a manual
  pre-clone with a token. No repo-credential binding feeds the clone.
- **D3 [minor] no model validation at `create ai-provider`.** An unknown model
  is stored happily and only fails at daemon run (`UnknownModelError`) — a late,
  confusing failure.
- **D4 [major, security] `create credential --value <secret>` leaks the secret**
  into shell history, argv, and `ps` output. (Surfaced by the Group D debate.)
- **D5 [major] the home clone is never fetched.** After the first clone, tasks
  reuse `home` forever → they can run against **stale code** indefinitely. D2
  only fixes the _initial_ private clone, not ongoing private fetch. (Related to
  C2 landing; surfaced by the Group D debate.)
- **D6 [major, security] credential values are not structurally omitted from
  export/serialization.** Credentials are generic resources (YAML-importable,
  plaintext attributes); any resource export, error serialization, or diagnostic
  dump can leak them. Exact-string redaction is insufficient for OAuth JSON (a
  nested access token leaks without the full JSON appearing). (Surfaced by the
  D4/D5 debate; ties to Group A.)

#### Group D — design direction (debate-hardened 2026-07-19)

**D1 — typed update commands** (reject generic `--attr k=v`). Immutable: `id`,
`projectId`, `type`, Credential/AIProvider `provider`. Mutable: `name`, `branch`,
`path`, `remoteUrl`, auth, Credential `value`, AIProvider `model`/`effort`/
`baseUrl`. Optional fields cleared via explicit `--clear-*` flags (omission ≠
clear). A repo update must NOT silently rewrite a cached clone's origin (require
cache absent or explicit reclone). No mandatory interactive confirm; define
late-binding (running tasks keep loaded values, queued/retried get current,
completed are historical). Credential `value` must NOT come via `--value`
(D4) — read via stdin / protected fd.

**D2 — repo transport identity (no longer minor).** Ownership is
**repository-scoped, not per-task** (git auth is transport config; per-task
collides with the shared home clone — confused deputy; don't overload the task
`credential` context). New Repository shape: clean `remoteUrl` (rejects embedded
userinfo/tokens; supports SSH/self-hosted) + `auth` union `{ambient} |
{https-token, credentialId} | {ssh-agent}`; migrate org/name → derived GitHub URL
but that isn't the long-term model. Secure injection: `GIT_ASKPASS` + static
non-secret askpass program, `execFile` (no shell), `GIT_TERMINAL_PROMPT=0`,
override `credential.helper` empty, strip `GIT_TRACE*`/`GIT_CURL_VERBOSE` from
child env, token never in URL/argv/config/script, redact stderr before any
log/event. **Never** token-in-URL or `-c http.extraHeader`. Cache: resolve
credential fresh per network op; cache identity = normalized `remoteUrl`, never
the credential; rotation can't revoke cloned source → explicit purge/reclone.
SSH = clean remote + ambient agent only this epic (managed keys deferred);
HTTPS+token is the deterministic default. Migrate the E2E pre-clone hack by
audit + **prefer reclone** over origin-scrub.

**D3 — validate via an app-owned `ModelCatalog` port** (pi adapter) at create
AND update, keep authoritative validation at session open (one source, no pi in
app/domain). Reject unknown `(provider, model)` pairs strictly, point to
`get models`. Drop the `--allow-unknown-model`/`--base-url` escape hatch (baseUrl
only applies after the pair resolves, so it would store a guaranteed-fail
config). Model validity is a runtime capability; future-planned config → an
explicit disabled/unresolved state, not a broken-resource flag. Dynamic models =
a separate capability with real metadata + catalog refresh.

#### D4 / D5 / D6 — fix design (debate-hardened 2026-07-19)

**D4 — secret input.** Remove `--value`. Interactive TTY → tested hidden-TTY
reader (raw mode, restore in finally). Automation → `--value-file <path>`
(`--value-file -` = stdin); sources mutually exclusive. No implicit stdin, no
`--value-env`. **`--value-timeout <duration>` (default 3m)** bounds the wait for
the value (TTY prompt or stdin/file read); on timeout the CLI **fails** with a
clear error and non-zero exit — it must never hang forever. Newline contract:
read to EOF, strip ONE trailing LF/CRLF, reject empty, preserve other bytes.
Don't over-check file perms. Value never in
results/errors/logs/exports. DB-at-rest + OAuth-at-rest = separate decision
(key outside SQLite), same non-disclosure rules for all credential types.

**D6 — structural omission.** Credential values must be **structurally omitted**
from any resource export / error serialization / diagnostic (not string-redacted;
OAuth JSON nesting defeats exact-string redaction). Ties to Group A export.

**D5 — authenticated fetch + reconciliation (shares C2 machinery).** home `main`
= local canonical; `origin/main` = fetched external. Every online prepare:
acquire **cross-process** repo+branch lock, resolve D2 auth fresh, fetch, classify
ancestry, update local main via **compare-and-swap** (ff when behind / keep when
ahead / typed divergence otherwise), clone workspace from the **recorded canonical
SHA** before releasing the lock. Explicit `sync` op reconciles divergence with a
**merge commit** (clean) or typed human resolution (content conflict) — NO
auto-rebase, NO separate integration branch. **Durable SHA protocol** (record
base/candidate/new SHAs before ref changes; also fixes A7 null base_commit);
landing reacquires the same lock and reconciles against the current canonical SHA.
Lock covers the initial clone too. Fetch every online prepare (no TTL initially);
offline/auth-fail → fail unless an explicit cached policy records cached-mode +
last-fetched-origin-SHA + fetch-time + base SHA. "Fresh" = base SHA pinned under
the lock, not that the network stopped. **Design D5 together with C2** (shared
per-repo lock + SHA/compare-and-swap protocol).
