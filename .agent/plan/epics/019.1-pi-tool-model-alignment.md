# 019.1 pi Tool Model Alignment (real built-in tool names)

## Outcome

The ring-1 tool model is aligned to **pi's real built-in tool surface** instead of
the invented generic names it was authored against. pi ships exactly seven
built-in tools (`pi -h`): `read`, `bash`, `edit`, `write`, `grep`, `find`, `ls`.
A single **canonical pi tool taxonomy** (source of truth) declares each tool's
class — read-only (`read`, `grep`, `find`, `ls`), file-mutating (`edit`, `write`),
and exec (`bash`) — plus the default allowed manifest (the six non-exec tools;
`grep`/`find`/`ls` are enabled even though pi defaults them off). Every ring-1
consumer reads that one source: the `beforeToolCall` **read/write classifier**
(`hook-binding.ts`), the **network/exec denial + trusted-effectful set**
(`network-denial.ts`), and the **pi-session manifest filter** (`pi-session.ts`,
which today keeps its own inline copy of the blocked set). `bash` stays
**permanently blocked** — exec/shell-class tools remain un-allowlistable (Epic 015
decision; PRD §4 — agents have no direct effect, all git/test/deploy go through
the broker or the chain executor). This closes a real correctness bug (below) and
removes the name-drift between the two blocked-set copies, so the model matches the
tool surface the real pi sessions actually emit before Phase 2B builds on it.

The correctness bug this fixes: `classifyOperation` in `hook-binding.ts` decides
read-vs-write by an **underscore-suffixed prefix heuristic** (`read_`, `get_`,
`list_`, …). pi's read-only tools `read`, `ls`, `grep`, `find` have no underscore,
so they fall through to **write** and a read on a path outside the task's
`write_scope` is wrongly **blocked and escalated as a re-planning signal**. After
this epic those four classify as read and pass; `edit`/`write` still classify as
write and still enforce write-scope unchanged.

## Decision Anchors

- **`pi -h` built-in tool list (2026-07-10, Ulrich)** — the authoritative surface:
  `read`, `bash`, `edit`, `write`, `grep` (read-only, default off), `find`
  (read-only, default off), `ls` (read-only, default off). This epic pins the model
  to these names; `grep`/`find`/`ls` are enabled for the agent (Ulrich 2026-07-10).
- **bash stays blocked (Ulrich 2026-07-10, confirming Epic 015).** Epic 015 Outcome
  — "exec/shell-class tools are permanently un-allowlistable"; PRD §4 — no direct
  network access for agents, all external I/O through the broker; PRD §5 — agents
  submit typed broker verbs, never run raw commands. So the real pi agent gets no
  `bash`; git/test/deploy are broker / chain-executor work, not agent tool calls.
- **Epic 015 Story 002 (`002-real-hook-write-scope.md`) + Epic 007** own the
  `beforeToolCall` write-scope binding and the read/write classification the bug
  lives in. This epic corrects the classifier's input mapping; it does not redesign
  write-scope or the escalation contract.
- **Epic 015 Story 003 (`003-agent-network-denial.md`)** owns the network/exec
  denial set, the allowlist registry, and the trusted-effectful set. This epic
  aligns those to pi's real names (bash still blocked; `edit`/`write` recognised as
  the real file-mutating tools).
- **Epic 016 Story 002 (`002-pi-session-lifecycle.md`)** owns `spawnPiSession` /
  `respawnPiSession` and the manifest filter. This epic replaces the inline
  `BLOCKED_TOOL_NAMES` copy in `pi-session.ts` with the one shared source of truth.

## Stories

- `001-pi-tool-classification.md` — a canonical pi tool taxonomy is the single
  source of truth for tool class; the `beforeToolCall` classifier reads it so pi's
  read-only tools (`read`, `ls`, `grep`, `find`) classify as **read** and a read on
  an out-of-scope path is no longer blocked, while `edit`/`write` still classify as
  write and still enforce write-scope. Closes the misclassification bug.
- `002-canonical-manifest-and-denial.md` — the taxonomy exports the default allowed
  manifest (the six non-exec tools) and the network/exec denial + trusted-effectful
  set read from it: a registry allowlisting `edit`/`write` (file-mutating) loads,
  one allowlisting `bash` still rejects, and `filterToolManifest` keeps the six real
  tools and drops `bash`.
- `003-pi-session-manifest-dedup.md` — `spawnPiSession` / `respawnPiSession` filter
  the manifest through the one shared exec/blocked source of truth instead of an
  inline duplicate: `bash` is stripped, the six real tools pass through, and every
  other spawn behaviour (ring-1 invariant, sanitized env, journal events) is
  unchanged.

## Verification Gate

- `npm run typecheck` exits 0; `npm test` green for all Story suites and for the
  pre-existing `src/ring1/*` and `src/agent/*` suites (no regression).
- **Bug closed (non-vacuous):** driven through the **real SU3 `beforeToolCall`
  shape**, a `read` (and an `ls`/`grep`/`find`) call whose `path` is outside the
  task `write_scope` returns pass-through (no block, no escalation); the same path
  under an `edit`/`write` call is still blocked and still escalates with the
  re-planning tag (Epic 007 semantics unchanged). Asserted against the classifier
  output, not a name string.
- **bash stays blocked:** a network-denial registry that allowlists `bash` rejects
  with `NetworkDenialError`; `spawnPiSession`/`respawnPiSession` given a manifest
  containing `bash` produce a filtered manifest with `bash` removed; the six real
  tools survive both the registry filter and the session filter.
- **Real file-mutating tools accepted:** a registry allowlisting `edit`/`write`
  with `pure: false` loads without error (they are in the trusted-effectful set);
  a read-only tool (`read`/`grep`/`find`/`ls`) with `pure: true` needs no trusted
  entry.
- **Single source of truth:** `pi-session.ts` no longer defines its own blocked-set
  literal; the exec/blocked names come from the shared taxonomy module (asserted by
  grep-for-absence in the story verify, or by importing the same constant).
- **Regression:** the pre-existing generic-name behaviour still holds via the
  fallback heuristic (`read_file` classifies read, `write_file`/`fetch` classify
  write/blocked) so already-green tests are not broken by the alignment.

## Non-Goals

- **No reversal of the bash-block decision.** `bash` stays permanently blocked; no
  sandboxed-bash allowance is introduced (that would be a separate decision record).
- **No new tools and no tool renames in pi.** This epic maps ring-1 to the names pi
  already ships; it does not add, remove, or rename a pi built-in.
- **No live daemon wiring of the default manifest into a real spawn.** The canonical
  manifest is exported and its filter behaviour asserted; wiring it into a live
  dispatch path (the real caller does not exist yet — only tests call
  `spawnPiSession` today) stays with Epic 016 / the Phase-2B spawn work.
- **No removal of the generic network-capable/exec-shell fallback sets.** They stay
  as harmless defense-in-depth against unknown names; the epic only adds pi's real
  names as the authoritative layer (debate-safe: keeping them cannot weaken ring 1).
- **No live model call, no network** in the default hermetic suite.
