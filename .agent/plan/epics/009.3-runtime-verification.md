# EPIC 008.3 — Runtime-backed verification

> Phase 3 of the former EPIC 008 (split 2026-07-18; runtime-only after the
> 2026-07-18 debate — agentic onboarding is EPIC 008.4, `run-anyway` moved to
> 008.2). Extends the static `tdd@1` executor (008.2) to witnesses that need the
> app + deps UP. Phase 1 (roles) is `008.1-role-agents.md`; Phase 2 (static
> executor) is `008.2-static-tdd-executor.md`; Phase 4 (agentic onboarding) is
> `008.4-agentic-onboarding.md`.

## Goal

`tdd@1` can verify against a `TestKind` that sets `requiresRuntime` — the engine
brings the app + deps UP around EACH witness run (the once-at-start base run AND
each round's RED and GREEN each get a FRESH runtime; a persistent runtime would run
stale code / DB state and forge a GREEN), managed by a `RuntimeController` behind a
global lease. `RuntimeController` owns a **runtime environment contract**: it
allocates a unique runtime directory and endpoint (a per-run Unix socket — no host
port collision), and passes ONE sanitized environment (daemon + provider secrets
absent) to every lifecycle command (`setup` / `start` / `readiness` / the witness /
`teardown`), all with a fixed working directory and engine-owned process group. The
tasks EPIC 008.2 parked as `blocked{runtime-unsupported}` are woken by an atomic,
idempotent operation once the project has confirmed runtime capacity. Because these
lifecycle commands are durable, agent-proposed shell with destructive host access,
008.3 ships a **minimum host guard** (explicit per-project runtime trust, command
preview + first-run approval, sanitized env, fixed cwd, process-group ownership,
timeouts, bounded output) — OS containment (containers/sandbox) is still EPIC 009,
but the common-case guard cannot wait for it.

This epic adds NO confirmation/schema migration (008.2 shipped the complete schema
incl. the `runtime` field and the `blocked` cause) and NO new attempt authority
(008.2's is reused) — it only makes the runtime causes/paths PRODUCIBLE. The
correctness Proof is fully DETERMINISTIC — roles run through 008.1's `replay`
provider, so no LLM is in the gate.

## Verification Gate

Gates: `npm run verify` (hermetic — no real Docker/ports; `FakeSessionFactory`
records the lifecycle command order. The suite covers: `RuntimeController.run`
lifecycle command order — acquire lease → persist a dirty marker carrying cleanup
context (process-group id, runtime dir, endpoint) → preflight `teardown`
(reconciliation) → `setup` → spawn `start` as an engine-owned process group → poll
`readiness` → run exactly ONE witness → kill the group + `teardown` in `finally` →
clear the marker only after cleanup; the runtime ENV contract — each lifecycle
command receives the SAME allocated dir + endpoint + sanitized env (canary daemon
secret absent); a FRESH runtime per witness proven by a per-`setup` generation
token the witness reads back — a stale/reused runtime is DETECTABLE and fails;
exactly THREE distinct successful lifecycles for a one-round cycle (base + RED +
GREEN), each event carrying `lifecycleId`, `phase` (base|red|green), `attemptId`,
`seq`, `outcome`; dirty-restart recovery — a seeded leaked runtime (dir + a live
process group) is torn down on restart, teardown success asserted, reconciliation
completing BEFORE any queued runtime witness starts; lease recovery with OWNER
identity + fencing (an overlapping daemon / surviving child group is handled — not
a bare "reclaim on restart"); the global lease SERIALIZES runtime work; the
`blocked{runtime-unsupported}` WAKE operation — atomic + idempotent, binding the
exact blocked task + attempt + selected `TestKind` + confirmed capacity revision,
with duplicate suppression + crash recovery; and edge outcomes — `teardown` failure
blocks further runtime work until reconciled, `start` early-exit fails the witness,
`readiness`-without-`start` is rejected by capacity validation (008.2), unreleased
lease reclaimed via fencing.)

Proof: fully deterministic (roles via `replay`; a minimal real runtime — a tiny
HTTP app over a controller-allocated Unix socket + a file "DB"). All temp files
live under one proof dir; every assertion is mechanical.

```bash
set -euo pipefail
PD="$(mktemp -d)"; jf() { node -e 'const t=JSON.parse(require("fs").readFileSync(process.argv[2],"utf8"));const v=process.argv[1].split(".").reduce((o,k)=>o==null?o:o[k],t);process.stdout.write(v==null?"":String(v));' "$1" "$2"; }
export KANTHORD_DB="$PD/kanthord.db"; node src/main.ts db migrate

# target repo: app reads a per-setup generation token over a Unix socket ($SOCK, $RUNTIME_DIR from the controller)
REPO_DIR="$PD/svc"; mkdir -p "$REPO_DIR/bin"; git -C "$REPO_DIR" init -q; : > "$REPO_DIR/AGENTS.md"
cat > "$REPO_DIR/bin/app.js" <<'JS'
const http=require("http"),fs=require("fs"),p=require("path");
const gen=fs.readFileSync(p.join(process.env.RUNTIME_DIR,"gen"),"utf8").trim();
http.createServer((_,res)=>res.end(gen)).listen(process.env.SOCK);
JS
git -C "$REPO_DIR" add -A && git -C "$REPO_DIR" commit -qm init; BASE=$(git -C "$REPO_DIR" rev-parse HEAD)

node src/main.ts create project --name svc --json > "$PD/p.json"; PROJECT=$(jf id "$PD/p.json")
REPO=$(node src/main.ts create resource --project "$PROJECT" --type repository --name svc --path "$REPO_DIR" --json | jf id /dev/stdin)
AIPROV=$(node src/main.ts create resource --project "$PROJECT" --type ai_provider --name openai --model gpt-5.5 --json | jf id /dev/stdin)
CRED=$(node src/main.ts create resource --project "$PROJECT" --type credential --name key --value replay --json | jf id /dev/stdin)
INIT=$(node src/main.ts create initiative --project "$PROJECT" --name svc --json | jf id /dev/stdin)
OBJECTIVE=$(node src/main.ts create objective --initiative "$INIT" --name backend --json | jf id /dev/stdin)

# (B6) runtime task created BEFORE capacity → must park blocked{runtime-unsupported}
BT=$(node src/main.ts create task --objective "$OBJECTIVE" --title early --instructions x --ac x \
  --executor tdd@1 --provider replay --script test/fixtures/svc-cycle.replay \
  --context repository=$REPO --context ai_provider=$AIPROV --context credential=$CRED --json | jf id /dev/stdin)
node src/main.ts daemon run --until-idle >/dev/null
node src/main.ts get task --id "$BT" --json > "$PD/bt.json"
test "$(jf confirmation.cause "$PD/bt.json")" = "blocked" || { echo FAIL-not-blocked; exit 1; }

# runtime capacity: unique generation per setup makes stale reuse observable;
# a per-project runtime-trust opt-in is required (host guard) before any lifecycle runs.
cat > "$PD/cap.json" <<'JSON'
{ "tests": [ { "name":"integration","command":"npm run itest","filePatterns":["**/*.itest.js"],"requiresRuntime":true } ],
  "verifyCommands": [],
  "runtime": { "setup":["date +%s%N > $RUNTIME_DIR/gen","echo 1 > $RUNTIME_DIR/db"],
               "start":"node bin/app.js","readiness":"curl -sf --unix-socket $SOCK http://x/",
               "teardown":["true"] } }
JSON
node src/main.ts project capacity set "$PROJECT" --from-file "$PD/cap.json"
node src/main.ts project runtime trust "$PROJECT" --approve   # explicit first-run approval (host guard)

# blocked task wakes now that capacity exists (atomic, idempotent)
CUR=0
node src/main.ts daemon run --until-idle; echo "exit=$?"
node src/main.ts get task --id "$BT" --json > "$PD/bt2.json"
test "$(jf confirmation.cause "$PD/bt2.json")" = "ready" || { echo FAIL-not-woken; exit 1; }
ATT=$(jf confirmation.attemptId "$PD/bt2.json")

# exactly THREE distinct successful lifecycles (base+RED+GREEN), each phase seen once, fresh gen each
node src/main.ts events --after "$CUR" --task "$BT" --attempt "$ATT" --json > "$PD/ev.json"
node -e 'const e=JSON.parse(require("fs").readFileSync(process.argv[1],"utf8")).events.filter(x=>x.type==="runtime.lifecycle");
 const ok=e.filter(x=>x.outcome==="ok"); const ids=new Set(ok.map(x=>x.lifecycleId)); const ph=ok.map(x=>x.phase).sort().join(",");
 process.exit(ok.length===3&&ids.size===3&&ph==="base,green,red"?0:1);' "$PD/ev.json" \
 || { echo "FAIL: not three distinct fresh lifecycles"; exit 1; }

# dirty-restart recovery: seed a real leaked runtime (dir + a live sleeper as its process group)
RDIR="$PD/leak"; mkdir -p "$RDIR"; setsid sleep 300 & PG=$!
sqlite3 "$KANTHORD_DB" "INSERT INTO runtime_dirty(project_id,lifecycle_id,runtime_dir,pgid,created_at) VALUES('$PROJECT','leak','$RDIR',$PG,strftime('%s','now'));"
node src/main.ts daemon run --until-idle >/dev/null
kill -0 "$PG" 2>/dev/null && { echo "FAIL: leaked process group not killed"; exit 1; }
test -z "$(sqlite3 "$KANTHORD_DB" "SELECT 1 FROM runtime_dirty WHERE lifecycle_id='leak';")" || { echo "FAIL: marker not cleared after cleanup"; exit 1; }

echo "008.3 ok: blocked→woken, 3 fresh lifecycles, dirty runtime reconciled"
```

## Stories

Depends on EPIC 008.2 (static `tdd@1` executor, confirmation schema incl.
`blocked` + `runtime` field, attempt authority, `project capacity set` with runtime
schema validation) and 008.1 (roles, `RoleRunner`, `replay` provider).

1. **Core-owned runtime port + `RuntimeController` adapter.** Define a
   core-owned `RuntimePort` (the use case depends on it); shell/process, lease, and
   SQLite live in the adapter — the import direction is explicit, not "existing
   command/session infra" (hexagonal). `RuntimeController.run(project, witness)`
   owns the per-witness lifecycle: acquire the global runtime lease → persist a
   dirty marker carrying cleanup context → preflight `teardown` (reconciliation) →
   `setup` → spawn `start` as an engine-owned process group → poll `readiness`
   (fixed default timeout) → run exactly ONE witness → kill the group + `teardown`
   in `finally` → clear the marker after cleanup. Not SIGKILL-proof — the guarantee
   is "dirty runtime is reconciled before the next runtime witness runs." Hermetic
   via `FakeSessionFactory` recording the lifecycle command order.

2. **Runtime environment contract.** `RuntimeController` allocates a unique runtime
   directory and endpoint — a per-run **Unix socket** (`$SOCK`), never a fixed host
   port (dodges host-process collisions the lease can't) — and exports ONE
   sanitized environment (`$RUNTIME_DIR`, `$SOCK`, DB path; daemon + provider
   secrets ABSENT) to every lifecycle command AND the witness, all with a fixed
   working directory. Freshness is observable: the fixture/test writes a unique
   generation token per `setup` that the witness reads back, so a reused/stale
   runtime is DETECTABLE.

3. **Runtime witness execution + lifecycle events.** The 008.2 witness runner gains
   a runtime branch: a `requiresRuntime` `TestKind` runs its witness THROUGH
   `RuntimeController` (fresh runtime per base / RED / GREEN). Events
   `runtime.lifecycle` carry `lifecycleId`, `phase` (base|red|green), `attemptId`,
   `seq`, `outcome` — so freshness, ordering, and cleanup are mechanically
   assertable (not inferable from setup/teardown counts).

4. **`blocked{runtime-unsupported}` wake operation (atomic + idempotent).** A task
   EPIC 008.2 parked `blocked{runtime-unsupported}` becomes claimable once the
   project has confirmed runtime capacity. The wake is a defined atomic transition
   binding the EXACT blocked task + attempt + the selected `TestKind` + the
   confirmed capacity REVISION; it is idempotent (duplicate wakes suppressed) and
   crash-safe (a wake interrupted mid-transition re-derives the same result). The
   block clears from capacity presence, NOT from a human `retry`/`fail` (008.2's
   `blocked` lifecycle, completed here).

5. **Lease + dirty recovery (owner-identified, fenced).** The global lease records
   an OWNER identity and a fencing token; recovery handles overlapping daemon
   processes and a surviving child process group — the dirty marker persists the
   process-group id + runtime dir so restart can kill the leaked group and tear
   down its state, and reconciliation completes BEFORE any lease reassignment or
   queued runtime witness. Edge outcomes: `teardown` failure marks dirty + blocks
   further runtime work until reconciled; `start` early exit fails the witness;
   unreleased lease reclaimed via the fencing token. Owns its migrations: lease
   storage + the dirty-marker table (with cleanup context) + wake-job idempotency
   keys.

6. **Minimum runtime host guard (does NOT wait for EPIC 009).** Runtime lifecycle
   commands are durable, agent/onboarding-proposed shell with destructive host
   access, so 008.3 ships the common-case guard: explicit per-project **runtime
   trust** opt-in (`project runtime trust <id> --approve`) with an exact command
   PREVIEW and NO silent non-interactive FIRST approval; sanitized env (story 2);
   fixed working directory; engine-owned process group; per-command timeouts;
   bounded captured output. This is NOT OS containment (that is EPIC 009 —
   container/sandbox/egress bounding) — it is the minimum that must not be deferred.

7. **End-to-end + Proof runbook.** Hermetic: the full lifecycle command order +
   env contract + fresh-runtime-per-witness + all edge outcomes; owner-fenced lease
   recovery; dirty-restart reconciliation (leaked dir + process group); the
   `blocked` wake (atomic/idempotent/crash-safe); the host guard (trust gate,
   sanitized env canary). Plus the deterministic Proof runbook above (the
   `svc-cycle.replay` fixture + the minimal Unix-socket HTTP-app runtime).

## Decisions

- **Fresh runtime per witness** (base + each RED + each GREEN) — a persistent
  runtime forges GREEN. `RuntimeController` owns the lifecycle behind a global
  lease; no Docker abstraction (plain shell).
- **Runtime-only epic** (debate 2026-07-18): agentic onboarding is a separate
  control plane → EPIC 008.4 (depends on this controller); `run-anyway` completes
  the STATIC unverified lifecycle → moved to EPIC 008.2. Runtime execution is a
  complete, high-risk walking skeleton on its own (process ownership, crash
  recovery, host security, durable wake) — bundling either would hide missing
  lifecycle definitions and weaken the only correctness Proof.
- **Runtime env contract is explicit** — controller-allocated dir + Unix-socket
  endpoint + one sanitized env to every lifecycle command; no fixed port (debate
  B1/B9).
- **`blocked` wake is atomic + idempotent + crash-safe**, binding task + attempt +
  `TestKind` + capacity revision (debate B11).
- **Lease/dirty recovery is owner-identified + fenced**, with persisted
  process-group cleanup context — not a bare "reclaim on restart" (debate B12).
- **A minimum host guard ships here** — trust opt-in, command preview, sanitized
  env, fixed cwd, process-group ownership, timeouts, bounded output; OS containment
  remains EPIC 009 (debate B14).
- **No confirmation/schema migration, no new attempt authority** — 008.2's are
  reused; new migrations are ONLY lease + dirty-marker + wake idempotency (debate
  B13). Capacity provenance for the agentic path is EPIC 008.4's.
- **The correctness Proof is fully deterministic** (roles via `replay`, real
  runtime over a Unix socket) — no LLM in the gate; no optional live smoke needed.

## Non-goals

- **No agentic onboarding loop** — EPIC 008.4 (it depends on this controller for
  runtime trials).
- **No `run-anyway`** — moved to EPIC 008.2 (static unverified lifecycle).
- **No parallel runtime work** — the global lease serializes it; per-project
  isolation / port-allocation is a later epic.
- **No Docker/container abstraction, no OS sandboxing / ring-1 hardening, no
  secret-manager** — EPIC 009; 008.3 ships only the minimum host guard.
- **No `pr@1` / `answer@1`, no per-role model, no mid-run steering, no parallel /
  cross-task steps.**
