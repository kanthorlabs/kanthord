## Findings

### Setup phase

- **minor — no `list credential` / `list credential`-family command.**
  Repro: `node src/main.ts list credential` → `error: unknown command
'credential'`. `list` only knows task/initiative/objective/event/model. After
  a `login provider` or `create credential`, the only way to recover the id is
  to re-read the command's stdout; there is no way to enumerate credentials,
  ai-providers, or repositories that already exist in a project. Suggestion: add
  `list credential|ai-provider|repository --project <id>`.

- **minor — inconsistent create-command output prefixes.**
  `create project` → `project created: <name>`; `create credential` →
  `credential resource added: <name>`; `create ai-provider` and
  `create repository` print a bare ULID with no prefix. Scripting has to
  `grep -oE '01[0-9A-HJKMNP-TV-Z]{24}'` to be safe. Suggestion: one consistent
  line format, e.g. `<kind> created: <id>`.

### Build phase

- **major — nested empty checkout dir misclassified as a git repo →
  misleading "wrong origin" failure (root cause group: WORKSPACE-PREP).**
  `src/workspace/local.ts` `isGitRepo()` runs `git rev-parse --git-dir` with
  `cwd = homePath` and treats success as "homePath is a git repo". But
  `--git-dir` succeeds for **any** directory nested inside a git worktree — it
  resolves to the _enclosing_ repo. When the checkout dir exists but is empty
  (e.g. pre-created with `mkdir -p`) and lives inside another git repo, prep
  takes the "home exists" branch, `getRemoteUrl()` reads the _parent_ repo's
  origin, and it throws
  `Home path <home> has origin git@github.com:kanthorlabs/kanthord.git but
expected https://github.com/kanthorlabs/kanthord-verify.git`.
  - Repro: `mkdir -p .data/e2e-0079/home` (inside the kanthord repo), point a
    repository resource's `--path` at it, run the daemon → root task fails
    `turns=0` with the mismatch error above.
  - Impact: blocked the entire run at task 1. The error actively misleads —
    it blames a wrong remote URL when the real state is "this dir is not its
    own clone at all". Easy to hit: the playbook's own default path
    (`.data/e2e-<tag>/home`) is nested in the kanthord repo, and pre-creating
    checkout dirs / re-running after a partial setup is common.
  - Suggestion: `isGitRepo` (or the caller) must verify the path is the git
    _root_, not merely inside a worktree — compare
    `git rev-parse --show-toplevel` (realpath) to `homePath`. When they differ,
    treat the dir as "not a clone here": clone fresh if empty, else error with
    a message that names the real problem.

- **major — transient provider error fails the task outright, no retry/backoff
  (root cause group: PROVIDER-RESILIENCE).**
  During the List rebuild the agent ran 8 turns then the final model call
  returned a transient upstream error:
  `Codex error: An error occurred while processing your request. You can retry
your request... request ID bff90d84-...`. kanthord marked the task `failed`
  immediately — no automatic retry, no backoff — discarding 8 turns / ~20k
  input tokens of work. The human must `retry task` and pay for a full rebuild.
  - Impact: real-model runs against a live API hit transient 5xx / "processing
    error" responses routinely; every one currently aborts a task and wastes
    the work already done. On a multi-task initiative this makes long runs
    flaky and expensive.
  - Suggestion: classify provider errors; on transient/5xx/rate-limit, retry
    the model call with bounded exponential backoff (e.g. 3 attempts) before
    failing the task. Keep hard failures (auth, invalid request) fail-fast.

- **minor — `get conflict` dumps the candidate's file only, no merge view
  (root cause group: CONFLICT-UX).**
  `get conflict --id` prints `target main@<sha>`, `candidate <id>@<sha>`, then
  the candidate's full copy of the conflicting file — with no conflict markers
  and no side-by-side of the target's version. To actually understand what
  clashes you must diff by hand. Suggestion: show the 3-way conflict (or at
  least the target-vs-candidate hunks) so the `retry --note` can be written
  from real information.

## Run outcome — 5/5, feature built from scratch, full CRUD proof PASSED

- **5/5 endpoints landed** (POST /tasks, GET /tasks, GET /tasks/:id,
  PUT /tasks/:id, DELETE /tasks/:id) through the full pipeline, from an empty
  repo. The run paused mid-way only because of a sustained OpenAI overload
  outage (see PROVIDER-RESILIENCE — 5 consecutive transient failures over
  ~11 min); once the outage cleared, the remaining 3 sibling conflicts were
  recovered via `retry → daemon rebuild → approve` and all landed. Not a
  kanthord defect.
- **Full CRUD program proof PASSED** (`TODO API SMOKE OK`), against the final
  landed `src/todo.mjs` on main:
  - `POST /tasks` → `201` (created task + generated `id`)
  - `GET /tasks` → `200`
  - `GET /tasks/:id` (known) → `200`
  - `PUT /tasks/:id` → `200`
  - `GET /tasks/:id` (unknown) → `404`
  - `DELETE /tasks/:id` → `204`
  - `GET /tasks/:id` (after delete) → `404`
  - Command:
    `git -C .data/e2e-0079/home show HEAD:src/todo.mjs > "$PWD/.data/e2e-0079/todo-final.mjs"`
    then `scripts/e2e/e2e-smoke-todo.sh "$PWD/.data/e2e-0079/todo-final.mjs" 39200`.
  - **minor — `e2e-smoke-todo.sh` breaks when given a `/tmp` path (root cause
    group: SMOKE-SCRIPT).** macOS `/tmp` is a symlink to `/private/tmp`, so
    `process.argv[1]` (`/tmp/todo.mjs`) ≠ `fileURLToPath(import.meta.url)`
    (`/private/tmp/todo.mjs`), and the agent-generated file's standard
    "run-directly" guard (`import.meta.url === process.argv[1]`) never fires —
    the server exits silently without listening and the smoke reports HTTP 000.
    The script's own doc example uses `/tmp/todo.mjs`. Suggestion: have the
    smoke script `realpath` its argument (or copy to / run from a real path, or
    `import { createServer }` instead of running the file), so the documented
    invocation works. Not a product bug in the agent's code (the guard is the
    correct idiom); a test-harness papercut.

## Pipeline behavior CONFIRMED WORKING (no defect)

- OAuth `login provider` (ChatGPT Plus / openai-codex) → live model calls.
- Resource creation (project, ai-provider, credential, repository) + graph
  import with alias bindings.
- Fresh clone via https-token, real-model build, in-workspace verification,
  candidate gate at `awaiting_confirmation`.
- `approve` lands the candidate, promotes main, unblocks dependents, and
  **persists** candidate `state` = `landed` (EPIC 007.8 fix verified).
- Sibling conflict on a moved base is detected, reported with actionable
  guidance, and **persisted** as candidate `state` = `conflict` (007.8).
- `retry task --note` re-queues a conflicted/failed task to `pending`.

## Findings grouped by root cause

- **WORKSPACE-PREP** (major): nested empty checkout dir misclassified as a git
  repo → misleading "wrong origin". One code fix in `src/workspace/local.ts`.
- **PROVIDER-RESILIENCE** (major): transient provider errors fail a task with
  no retry/backoff; wastes work and makes real-model runs flaky. One code
  change in the agent-runner call path + error classification.
- **CONFLICT-UX** (minor): `get conflict` shows candidate file only, no merge
  view. UI/formatting change in the `get conflict` query + CLI printer.
- **CLI-ERGONOMICS** (minor, x2): no `list credential|ai-provider|repository`;
  inconsistent create-command output prefixes. Both in `apps/cli`.
- **SMOKE-SCRIPT** (minor): `e2e-smoke-todo.sh` fails on a `/tmp` path (its own
  documented example) because the macOS `/tmp`→`/private/tmp` symlink breaks the
  file's `import.meta.url === process.argv[1]` run-guard. Fix in `scripts/e2e`.

The two **majors** (WORKSPACE-PREP, PROVIDER-RESILIENCE) are the fix-EPIC
targets; the minors (CONFLICT-UX, CLI-ERGONOMICS x2, SMOKE-SCRIPT) are bundled
as a small ergonomics story.
