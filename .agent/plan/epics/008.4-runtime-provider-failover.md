# EPIC 008.4 — Runtime provider failover on provider errors

## Goal

When the daemon runs a task and the chosen provider fails with a **provider-level
error**, the runner **fails over** to the next provider in the resolved chain
(008.2) and re-runs the task on a **clean attempt**, walking the chain until one
provider succeeds or the chain is exhausted. A provider-level error is defined by
**origin and type**, not by string matching: it is a **typed error raised by the
AI session adapter** — session creation (`CredentialError`, `UnknownModelError`)
and the adapter's classified stream errors (rate-limit `429`, unavailable `503`).
Errors from git, workspace, database, or verification — and task-level outcomes
(bad agent work, verify fail, budget, escalation) — are **not** provider errors
and never fail over. Each failover emits an observable, credential-redacted
`provider.failover` event (new event type + migration 19); chain exhaustion fails
the task with a typed, redacted aggregated reason.

## Verification Gate

Gates: `npm run verify` (typecheck + test + verify:handoff + lint + db status)

Proof: hermetic real daemon runs with the fake agent. Proves (A/B) a typed
provider error on `bad` fails over to `good` and lands, emitting a redacted
`provider.failover` event on the real event feed; (C) a verification failure does
**not** fail over (task-isolated, cursor-fenced); (D/exhaustion) an all-bad chain
fails with a typed aggregated reason and no secret anywhere.

## Stories

- **A — Typed provider-error provenance + fake seam.** Provider errors are
  identified by **origin/type** — typed errors raised by the AI session adapter
  (`ProviderSessionFactory.for` throwing `CredentialError` / `UnknownModelError`,
  and the adapter's classified stream errors `429`/`503`) — never by matching
  "network"/"timeout" strings, which git/workspace/db/verify can also produce.
  The walked chain is already **logged-out-filtered** at resolution (008.2), so
  failover only ever ranges over `active` providers and operates on their
  **record ids**; classification distinguishes `invalid-model` / `429` / `503` /
  auth as the reason codes it emits. Add the hermetic
  `KANTHORD_FAKE_FAIL_PROVIDERS` seam: the fake session factory raises a typed
  provider error at session creation for any provider whose name is listed (the
  provider's `model` stays a valid pinned-catalog id so registration still
  passes; the fake distinguishes providers by name, not model).
- **B — Failover loop with a clean-attempt boundary.** The runner walks the
  resolved chain; on a provider error it advances to the next provider and starts
  a **clean attempt**: re-prepare the workspace from the task's baseline commit,
  discard any files/partial commits from the failed attempt, and reset the
  in-flight task state so the next provider gets a pristine run. A
  **session-creation** error (the common case, which precedes workspace prep)
  needs no cleanup and is what the shell Proof exercises; the **mid-stream**
  `429`/`503` case (error after the workspace was modified) is where the
  clean-attempt reset is load-bearing and is covered by a **unit test** with a
  second fake failure mode that dirties the workspace before raising — the shell
  Proof cannot observe workspace cleanup. Emit a `provider.failover` event
  `{from, to, reasonCode}`.
- **C — Task failures never fail over.** A task-level outcome (verify fail, bad
  work, budget, escalation) or a non-provider error returns immediately on the
  current provider — no chain advance, no failover event. Add the
  `scripts/e2e/make-verify-fail-graph.sh` fixture (root task's fake turn writes a
  file that fails its lightweight verification) used by the Proof.
- **D — `provider.failover` event type + redacted observability + exhaustion.**
  Add `provider.failover` to `EVENT_TYPES` (`src/domain/event.ts`) — it is a new
  type, distinct from the existing `provider.retry` (007.9 same-provider transient
  retry) — and add **migration 19** rebuilding the `events` table so its `type`
  CHECK admits `provider.failover` (SQLite can't `ALTER` a CHECK; follow the
  existing `events_new*` rebuild pattern). Failover events and the final
  exhaustion failure use **typed reason codes + safe messages** — never raw
  exception text, response bodies, headers, URLs, or credentials. Chain exhaustion
  emits a `task.failed` event whose payload carries a **structured**
  `reasonCode='provider_chain_exhausted'` plus a `providerReasons` list of each
  attempted provider's typed code (not free prose), so consumers match on a code,
  not a phrase. The `provider.failover` event carries `{from, to, reasonCode}` and
  is queryable via `list event`; the daemon summary reports failover counts.

## Non-goals

- No provider health tracking / circuit-breaking / cooldown across tasks — each
  task starts the chain from the top.
- No automatic re-ordering of the chain by success rate (order stays operator-set,
  008.2).
- No partial-progress resume across providers — failover re-runs the task from a
  clean attempt.
- No change to the chain resolution order (008.2) or the default convention
  (008.1).
