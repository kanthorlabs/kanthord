# EPIC 007.6 — guided conflict resolution (predict → explain → guide → rebuild-clean)

> **Design direction (Ulrich, 2026-07-21).** Supersedes the earlier
> `007.6-sibling-integration` stub. Grew out of the post-007.4 TODO-API E2E
> (findings: `007.5-conflict-recovery.md (Appendix A)`) and a design conversation: _don't_ merge or
> rebase into a conflict and then fix dirty markers — instead **predict** the
> conflict before touching anything, **explain** exactly what clashes, let the
> user **guide** the fix with a free-text note, and **rebuild** the task on a
> clean latest base so the agent never edits conflict markers. Builds on 007.5
> (which makes `retry task` recover a conflict-marked candidate). Recommend
> re-running the read-only debate on this epic before `/work`.
>
> **Why this shape (grounded in the code, verified 2026-07-21):**
>
> - `git` is 2.48.1 → `git merge-tree --write-tree <main> <candidate>` performs a
>   full 3-way merge **in memory**: exit≠0 on conflict, lists the conflicting
>   paths (stage 1/2/3 blobs) and returns a tree whose file carries the
>   `<<<<<<< ours … ======= … theirs >>>>>>>` markers — **no commit, no index, no
>   working tree, no lock, no dirty state**. Proven on the real E2E conflict
>   (main `6e7edf5` vs List candidate `29218221` → `CONFLICT (content) in
src/todo.mjs`, hunks at lines 72–121 where `Get` and `List` both inserted a
>   route handler).
> - The **user-note → agent** loop already exists: `reject --reason` persists
>   `rejectionReason`, and `composition.ts:303` `getPriorRejection` injects it
>   into the re-run prompt as a `## Prior Rejection` block (`pi.ts:451-457`). We
>   generalize this channel rather than invent one.
> - Re-run already branches from **latest `main`** (`workspace/local.ts prepare()`
>   does `git fetch origin`; the E2E's Get re-ran on the updated base and landed
>   ff). So "rebuild on a clean base" needs no new mechanism — the agent simply
>   never sees markers.

## Goal

After 007.6, approving a candidate whose landing would conflict does **not**
attempt a dirty merge. The approve/land path first runs a pure `merge-tree`
**preview**: a fast-forward or cleanly-mergeable candidate lands as today; a
conflicting one is **not merged** — the task stays `awaiting_confirmation` and
the system records that a preview conflict occurred. The user can read a
**conflict overview** (`kanthord get conflict --id <task>`: the conflicting
files, the `<<<<<<< / >>>>>>>` hunks, and human labels — "current main (after
task X)" vs "this candidate (task Y)"), recomputed on demand from `merge-tree`
(no durable overview stored). The user then **guides** the fix:
`kanthord retry task --id <task> --note "<guideline>"` re-queues the task (007.5
recovery) and persists the note. On the re-run the agent rebuilds the task on a
clean latest base with **both** the user's note **and** the conflict overview
(what the other task changed) injected into its prompt — it never edits conflict
markers. The rebuilt candidate is itself preview-checked before it can land. No
marker-resolving, no auto-serialization, no new durable task lifecycle state.

## Verification Gate

Gates: `npm run verify` (typecheck + test + verify:handoff + lint + db status).
Hermetic — no network. Fakes + real-git-in-temp:

- **S1 (predict):** a `RepositoryLanding.preview(candidate)` port method, real git
  in temp, returns `fast-forward` / `mergeable` / `conflict{ files, perFile:
[{ path, hunks }] }` computed via `merge-tree --write-tree` — no commit/index/
  worktree mutation (assert the repo is byte-identical before/after). Conflict
  case lists the file and non-empty hunks; clean cases report their kind.
- **S2 (explain):** CLI test — `get conflict --id <task>` prints the conflicting
  files, the `<<<<<<<`/`>>>>>>>` hunks, and the ours/theirs task labels; the
  `approve` conflict summary names the file(s), points to `get conflict`, and
  shows the `retry … --note` hint. Sanitized; no raw stack, no temp paths.
- **S3 (guide):** `retry task --id <t> --note <text>` persists the note; a unit
  test on the runner's prompt builder shows BOTH the note and the conflict
  overview injected (the `getPriorRejection` hook generalized to a
  `getPriorFeedback` carrying `{ note?, conflictOverview?, priorSummary? }`).
- **S4 (predict-before-mutate):** an ApproveTask test — a predicted conflict
  returns the `conflict` outcome WITHOUT ever running `git merge`/`--abort` on the
  canonical branch (assert the target HEAD is untouched and no merge/abort ran);
  a predicted-clean candidate proceeds to the real land (ff or merge).

Proof: deterministic — pure CLI + real git in temp dirs, NO model — reproduce the
conflict via the fake-agent seam + `make-todo-graph.sh` on a **relative** DB,
then drive predict → explain → guide. `set -euo pipefail`; expected failures use
`if ! …; then`.

```bash
set -euo pipefail
WORK="$(mktemp -d)"; cd "$WORK"
export KANTHORD_DB=".data/kanthord.db"
REPO_ROOT="$OLDPWD"
node "$REPO_ROOT/src/main.ts" db migrate >/dev/null
PROJECT=$(node "$REPO_ROOT/src/main.ts" create project --name gcr | head -1)

HOME_REMOTE="$WORK/home.git"; git init -q --bare -b main "$HOME_REMOTE"
SEED="$WORK/seed"; git clone -q "$HOME_REMOTE" "$SEED"
git -C "$SEED" -c user.email=a@b.c -c user.name=t commit -q --allow-empty -m init
git -C "$SEED" push -q origin main
MIRROR="$WORK/mirror"
REPO=$(node "$REPO_ROOT/src/main.ts" create repository --project "$PROJECT" --name home \
        --remote-url "file://$HOME_REMOTE" --branch main --auth ambient --path "$MIRROR" | head -1)
CREDVAL="$(mktemp)"; printf 'dummy' > "$CREDVAL"
CRED=$(node "$REPO_ROOT/src/main.ts" create credential --project "$PROJECT" --name c1 --provider openai-codex --value-file "$CREDVAL" | head -1)
PROV=$(node "$REPO_ROOT/src/main.ts" create ai-provider --project "$PROJECT" --name p1 --provider openai-codex --model gpt-5.6-terra | head -1)

GRAPH="$WORK/g"; "$REPO_ROOT/scripts/e2e/make-todo-graph.sh" "$GRAPH" >/dev/null
node "$REPO_ROOT/src/main.ts" import graph "$GRAPH" --create --project "$PROJECT" \
        --bind source="$REPO" --bind provider="$PROV" --bind cred="$CRED" >/dev/null
INIT=$(node -e 'console.log(JSON.parse(require("fs").readFileSync(process.argv[1]+"/.kanthord-export.json","utf8")).initiativeId)' "$GRAPH")
ROOT=$(node "$REPO_ROOT/src/main.ts" list task --initiative "$INIT" --json \
        | node -e 'let s="";process.stdin.on("data",d=>s+=d).on("end",()=>console.log(JSON.parse(s).find(t=>/Create Task/.test(t.title)).id))')
export KANTHORD_FAKE_AGENT="$GRAPH/.fake-agent.json"

node "$REPO_ROOT/src/main.ts" run daemon --until-idle --poll-interval 200
node "$REPO_ROOT/src/main.ts" approve task --id "$ROOT"
node "$REPO_ROOT/src/main.ts" run daemon --until-idle --poll-interval 200
SIB=$(node "$REPO_ROOT/src/main.ts" list task --initiative "$INIT" --json \
        | node -e 'let s="";process.stdin.on("data",d=>s+=d).on("end",()=>console.log(JSON.parse(s).find(t=>/List Tasks/.test(t.title)).id))')

# --- predict + explain: approve predicts a conflict; canonical main is NOT mutated ---
BEFORE=$(git -C "$MIRROR" rev-parse main)
node "$REPO_ROOT/src/main.ts" approve task --id "$SIB" >/dev/null 2>&1     # exit 0, predicted conflict, no dirty merge
test "$(git -C "$MIRROR" rev-parse main)" = "$BEFORE"                       # canonical branch untouched by the failed approve
OVERVIEW=$(node "$REPO_ROOT/src/main.ts" get conflict --id "$SIB")
echo "$OVERVIEW" | grep -q "todo.mjs"                                       # names the conflicting file
echo "$OVERVIEW" | grep -q "<<<<<<<"                                        # shows the actual hunk markers

# --- guide: retry carries a free-text note, which is persisted ---
node "$REPO_ROOT/src/main.ts" retry task --id "$SIB" --note "keep both handlers; list before delete"
test "$(node "$REPO_ROOT/src/main.ts" get task --id "$SIB" --json | node -e 'process.stdin.once("data",d=>console.log(JSON.parse(d).status))')" = "pending"
node "$REPO_ROOT/src/main.ts" get task --id "$SIB" --json | grep -q "keep both handlers"   # note persisted + surfaced

echo "007.6 PROOF OK"
```

(That the injected note+overview actually steer the rebuild is proven by the S3
prompt-builder unit test — the fake agent does not read prompts. Full
recovery-to-`completed` with a real model was demonstrated live in the E2E.)

## Stories

- **S1 — Landing conflict preview (`merge-tree`).** Add a read-only
  `RepositoryLanding.preview(candidate)` (git mechanics in `GitRepositoryLanding`)
  returning `fast-forward | mergeable | conflict{files, perFile:[{path,hunks}]}`
  via `merge-tree --write-tree`. Pure — asserts zero repo mutation.
- **S2 — Conflict overview surface.** `get conflict --id <task>` (a query use
  case) recomputes the overview from the candidate SHA + current target and
  prints files + hunks + ours/theirs task labels. Enrich the 007.5 `approve`
  conflict summary to point at it + the `retry --note` hint.
- **S3 — Guided recovery note.** `retry task --id <t> --note <text>` persists the
  note; generalize the prompt-feedback hook (`getPriorRejection` →
  `getPriorFeedback` carrying `{ note?, conflictOverview?, priorSummary? }`) so
  the re-run agent receives the user's guideline AND what the other task changed.
- **S4 — Predict-before-mutate landing gate.** `ApproveTask` calls `preview`
  before the mutating land; a predicted conflict returns the typed `conflict`
  outcome without running `git merge`/`--abort` on the canonical branch; a clean
  predict proceeds to the real land. The rebuilt candidate re-enters the same
  gate.

## Non-goals

- **Marker-resolving agent** — the agent never edits `<<<<<<<`/`>>>>>>>` markers;
  we rebuild on a clean base with the overview+note as context. (Explicitly the
  approach Ulrich rejected: "don't fix dirty conflict".)
- **Cheap targeted "resolve only the hunks" run** — the rebuild is a full task
  re-run on a clean base (accepted cost; reuses existing machinery). A targeted
  resolver that reuses more of the prior candidate is a possible later
  optimization.
- **Auto-serialization of repo-bound siblings** (the old D1) — a different
  philosophy (avoid conflicts by ordering, at the cost of parallelism). Possible
  complementary optimization; out of scope here.
- **Pre-run prediction** — prediction needs a produced candidate, so it happens
  at approve time; pre-run only a coarse "same file targeted" heuristic is
  possible (not built here).
- **N4 latent** — whether a clean _merge_ (non-ff) landing should need a second
  confirmation / post-merge re-verify (debate B5). Separate product decision.
- No new durable task lifecycle state; the overview is recomputed on demand, not
  stored; no worker scheduler.
