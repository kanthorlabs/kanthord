# EPIC 007.6 — guided conflict resolution (predict → explain → guide → rebuild-clean)

> **Design direction (Ulrich, 2026-07-21), hardened by two read-only debates.**
> Supersedes the earlier `007.6-sibling-integration` stub. Grew out of the
> post-007.4/007.5 TODO-API E2Es (findings: `007.5-conflict-recovery.md`
> Appendix A; this epic's Appendix B): _don't_ merge or rebase into a conflict
> and fix dirty markers — **predict** the conflict before touching anything,
> **explain** exactly what clashes, let the user **guide** the fix with a
> free-text note, and **rebuild** the task on a clean latest base so the agent
> never edits conflict markers. Builds on 007.5 (`retry task` recovers a
> conflict-marked candidate).
>
> **Debate history (both read-only, opencode plan, 2026-07-21):**
>
> - Debate 1 drove: predict/explain/guide/rebuild; fixed the purity claim (B1),
>   required a pinned target OID (B2), resolved a durable-state contradiction
>   (B4), snapshot per retry (S1), test merge parity (S2).
> - Debate 2 **removed the "escalation ladder" / Tier A targeted resolver**
>   (Ulrich's ruling): on the append workload that motivates this epic the
>   candidate adds one route handler in one hunk at the same insertion point, so
>   there are **zero non-conflicting hunks to preserve** and Tier A collapses to a
>   full rebuild — while its `git apply --reject` mechanism did not even preserve
>   git's 3-way merge semantics. A targeted resolver may return later as its own
>   epic, designed with real-model cost data. Debate 2 also replaced "run
>   `git merge` twice and test parity" with **land the previewed tree** (below).
>
> **Why this shape (grounded in the code, verified 2026-07-21):**
>
> - `git` is 2.48.1 → `git merge-tree --write-tree <target> <candidate>` performs
>   a full 3-way merge **in memory**: exit≠0 on conflict (lists conflicting paths
>   - stage 1/2/3 blobs), and on success **writes and prints a merged tree OID**.
>     It writes blob/tree objects into the object DB but leaves **refs, HEAD, index,
>     and worktree unchanged** (debate B1 — that is the real guarantee, _not_ a
>     byte-identical repo). Proven on the real E2E conflict (main `6e7edf5` vs List
>     candidate `29218221` → `CONFLICT (content) in src/todo.mjs`, hunks 72–121).
> - **Land the previewed tree, never re-merge (debate 2).** Because
>   `merge-tree --write-tree` already returns the exact merged tree OID, a clean
>   land builds the merge commit directly from _that_ tree with parents
>   `{targetOID, candidateOID}` and advances the branch with an **atomic**
>   `git update-ref refs/heads/<b> <newCommit> <expectedOldOID>`. This makes
>   preview↔land parity a **construction** (the landed tree _is_ the previewed
>   tree) instead of a tested assumption, and gives compare-and-swap a real atomic
>   boundary. No second `git merge`, no worktree merge state to abort.
> - The **user-note → agent** loop already exists: `reject --reason` persists
>   `rejectionReason`; `composition.ts:303` `getPriorRejection` injects it into the
>   re-run prompt (`pi.ts:451-457`). We generalize this channel.
> - Re-run already branches from **latest `main`** (`workspace/local.ts prepare()`
>   does `git fetch origin`; the E2E's Get/List/Update re-ran on the updated base
>   and landed). So "rebuild on a clean base" needs no new fetch mechanism.

## Goal

After 007.6, approving a candidate whose landing would conflict does **not**
attempt a dirty merge and **never mutates the canonical branch on failure**. The
approve/land path runs a pure `merge-tree` **preview** against a **pinned target
OID** (the exact commit previewed):

1. **fast-forward** → advance the branch to the candidate OID via an atomic
   `update-ref … <expectedOld=targetOID>`.
2. **cleanly-mergeable** → build the merge commit from the **previewed merged
   tree OID** (parents `{targetOID, candidateOID}`) and advance the branch via the
   same atomic compare-and-swap. The landed tree equals the previewed tree by
   construction.
3. **conflict** → **no mutation**; the task stays `awaiting_confirmation` with its
   007.5 durable conflict candidate (`state="conflict"`) retained.
4. **target moved** (CAS `expectedOld` mismatch — an external push or a
   concurrent land moved the branch between preview and land) → **re-preview**
   against the new OID, up to a bounded retry count, then a typed `target_moved`
   outcome. Never a blind merge onto an unexpected base.

On conflict the user reads a **conflict overview** — `kanthord get conflict --id
<task>`: the conflicting files, the `<<<<<<< / >>>>>>>` hunks, and **honest
labels** ("target `<branch>@<OID>`" and "candidate task `<id>`@`<OID>`", not
"main after task X" which can be false when several commits contributed) —
**recomputed on demand** from the retained candidate OID + current target
(nothing new stored).

The user then **guides**: `kanthord retry task --id <task> --note "<guideline>"`
re-queues the task (007.5 recovery) and persists the note. The re-run **rebuilds**
the task on the clean latest base with the note **and** a **marker-free
structured conflict context** injected into the prompt (labeled target-side /
candidate-side / base slices — **not** raw `<<<<<<<` markers, so the agent never
sees a marked file; debate B3). The rebuilt candidate **re-enters the same
preview gate** before it can land. No marker-resolving, no targeted resolver, no
auto-serialization, no new durable task lifecycle state.

## Verification Gate

Gates: `npm run verify` (typecheck + test + verify:handoff + lint + db status).
Hermetic — no network. Fakes + real-git-in-temp:

- **S1 (predict, pinned, non-mutating, returns the merged tree):** a
  `RepositoryLanding.preview(candidate, targetOID)` port method, real git in
  temp, returns `fast-forward{candidateOID}` / `mergeable{treeOID}` /
  `conflict{ files, perFile:[{ path, hunks }] }` via `merge-tree --write-tree`.
  Assert **refs, HEAD, index, and worktree unchanged** before/after (NOT
  byte-identical; debate B1). Conflict lists the file + non-empty hunks; mergeable
  returns the concrete merged tree OID; ff returns the candidate OID.
- **S2 (explain, honest labels):** CLI test — `get conflict --id <task>` prints
  the conflicting files, the `<<<<<<<`/`>>>>>>>` hunks, and honest OID labels
  (`target <branch>@<OID>` / `candidate <taskId>@<OID>`; debate S4). The 007.5
  `approve` conflict summary names the file(s), points to `get conflict`, and
  shows the `retry … --note` hint. `get conflict` binds to and prints the
  candidate + target OIDs it was computed against (debate S2). Sanitized; no raw
  stack, no temp paths.
- **S3 (guide + marker-free agent payload + snapshot):** `retry task --id <t>
--note <text>` persists the note; a unit test on the runner's prompt builder
  shows the note AND a **structured, marker-free** conflict context injected (the
  `getPriorRejection` hook generalized to `getPriorFeedback` carrying `{ note?,
conflictContext?, priorSummary? }`) — assert the injected payload contains **no**
  `<<<<<<<`/`>>>>>>>` sequence (debate B3). The retry durably **snapshots**
  `{ candidateOID, targetOID, conflictContext }` onto the recovery attempt so the
  rebuild prompt is deterministic/auditable and does not depend on a git object
  surviving GC (debate B4/B5/S1).
- **S4 (land the previewed tree via atomic CAS):** an ApproveTask + landing test —
  a predicted **conflict** returns the typed `conflict` outcome with **zero**
  mutation (assert branch OID, HEAD, index, worktree all unchanged; no
  `git merge`/`--abort` ran). A predicted **mergeable** candidate lands a commit
  whose **tree equals the previewed `treeOID`**, with parents `{target,
candidate}`, via `update-ref … <expectedOld=targetOID>`; **ff** lands via
  `update-ref` to the candidate OID. If the branch moved since preview, the CAS
  update fails and the path **re-previews** (bounded; typed `target_moved` after
  the cap) — assert it never lands onto the wrong base (debate B6/B7).
- **S5 (rebuild re-enters the gate):** a recovery-run test — the clean-base
  rebuild's candidate is itself run through `preview` before landing; a rebuild
  that now merges/ff-lands cleanly does so through the same S4 path (assert the
  recovery candidate is previewed, not landed blind).

then drive predict → explain → guide and prove the atomic land wiring on the

## Stories

- **S1 — Landing conflict preview (`merge-tree`, pinned, returns merged tree).**
  Add a read-only `RepositoryLanding.preview(candidate, targetOID)` in
  `GitRepositoryLanding` returning `fast-forward{candidateOID} | mergeable{treeOID}
| conflict{files, perFile:[{path,hunks}]}` via `merge-tree --write-tree`. Assert
  refs/HEAD/index/worktree unchanged (debate B1).
- **S2 — Conflict overview surface (honest labels, version-bound).** `get conflict
--id <task>` (query use case) recomputes the overview from the retained
  candidate OID + current target and prints files + hunks + `target <branch>@<OID>`
  / `candidate <taskId>@<OID>` labels; prints the OIDs it was computed against.
  Enrich the 007.5 `approve` conflict summary to point at it + the `retry --note`
  hint.
- **S3 — Guided note + marker-free agent context + snapshot.** `retry task --id
<t> --note <text>` persists the note; generalize the prompt hook
  (`getPriorRejection` → `getPriorFeedback` with `{ note?, conflictContext?,
priorSummary? }`) injecting the note AND a **marker-free** structured conflict
  context (target/candidate/base slices). Durably snapshot `{candidateOID,
targetOID, conflictContext}` onto the recovery attempt (deterministic prompt;
  debate B3/B4/B5/S1).
- **S4 — Predict-before-mutate + land-the-previewed-tree via atomic CAS.**
  `ApproveTask` calls `preview` (pinned OID); a predicted conflict returns the
  typed `conflict` outcome with zero mutation; a clean predict lands the
  **previewed tree** (ff → candidate OID; mergeable → commit from the previewed
  `treeOID`, parents `{target,candidate}`) via `update-ref … <expectedOld>`.
  Branch moved → bounded re-preview, then typed `target_moved` (debate B6/B7/S3).
- **S5 — Recovery candidate re-enters the gate.** The clean-base rebuild's
  candidate is previewed before it can land (no blind land of the recovery
  output); it lands through the same S4 path when clean.

## Non-goals

- **Targeted / hunk-preserving resolver ("Tier A")** — removed after debate 2:
  on the append workload it collapses to a full rebuild (the one conflicting hunk
  is the whole change) and its `git apply --reject` mechanism did not preserve
  git's 3-way merge semantics. A narrow text-`modify/modify` resolver with a
  restricted prompt/tools may return as **its own epic**, designed with real
  measured cost — not asserted here.
- **Marker-resolving agent** — the agent never edits `<<<<<<<`/`>>>>>>>` markers;
  it rebuilds on a clean base with a marker-free structured context. (Explicitly
  the approach Ulrich rejected: "don't fix dirty conflict".)
- **Auto-serialization of repo-bound siblings** (old D1) — needs reliable
  write-set/path-affinity metadata the system lacks today (sibling ≠ same-file
  writer). Deferred as a complementary track to add later where affinity is
  declared and measured conflict rates justify the lost parallelism.
- **Pre-run prediction** — prediction needs a produced candidate; it happens at
  approve time. Pre-run only a coarse "same file targeted" heuristic is possible
  (not built here).
- **N4 latent** — whether a clean _merge_ (non-ff) landing should need a second
  confirmation / post-merge re-verify (debate B5, run 1). Separate product
  decision.
- No new durable task **lifecycle** state; the overview is recomputed on demand;
  the recovery attempt snapshot is per-attempt metadata, not a lifecycle state;
  no worker scheduler.

## Appendix A — real merge-tree evidence (E2E, 2026-07-21, first run)

`git merge-tree --write-tree 6e7edf5 29218221` (main-after-Get vs List candidate)
→ exit 1; stage 1/2/3 entries for `src/todo.mjs`; result tree `ebfe8fa` whose
`src/todo.mjs` carries the conflict markers (`<<<<<<< 6e7edf5 … ======= …
29218221`, lines 72–121 — both tasks inserted a route handler at the same
point). On a clean merge the same command exits 0 and prints the merged tree OID
— the tree S4 lands. refs/HEAD/index/worktree untouched (it does write result
blob/tree objects; debate B1).

## Appendix B — post-007.5 real-model E2E (2026-07-21, this epic's evidence)

Second full E2E after 007.5 (gpt-5.6-terra, ChatGPT Plus OAuth via kanthord's own
CLI; repo kanthord-verify https-token; isolated relative DB `.data/e2e-076/`).
The 5-endpoint TODO API built **from scratch to completion 5/5** on `main`
(POST/GET-list/GET-id/PUT/DELETE coexist, `node --check` OK).

- **007.5 validated live:** conflict outcomes named `retry task --id`, listed
  `src/todo.mjs`, stated the next step (exit 0, `awaiting_confirmation`); `retry
task` recovered to `pending` and emitted `task.ready`, never `task.rejected`.
- **Cost (the finding):** 4 siblings all edit `src/todo.mjs`; each approval moves
  the base and re-conflicts the next. Delete landed ff (0 re-runs); Get needed 1
  recovery run; List and Update each **re-conflicted after a later sibling landed**
  and needed 2 recovery runs apiece — **5 recovery agent re-runs ≈ 61k input
  tokens** (~= the 68.5k initial 5-task build) + 3 manual `run daemon` cycles.
  This is **worse than N-1** re-runs (debate 1 S4). The plain 007.5 recovery
  (retry → re-run on fresh base) already landed each sibling cleanly on the second
  attempt — 007.6's value over 007.5 is the **no-dirty-mutation predict gate**,
  the **explain** surface, the **guided note**, and the **atomic land-the-tree**
  correctness, not fewer re-runs. (Fewer re-runs is the deferred
  auto-serialization / targeted-resolver space.)
- **Also found (→ EPIC 007.7):** `list event` silently caps at its default page
  (~100 rows) with no truncation/next-cursor signal; `list task --json` omits
  `dependencies` (BUG-4).
