# 019.14 Write-Scope Accepts Worktree Writes

## Outcome

The driven pi agent can save files in its task worktree: ring-1's write-scope
check now authorises writes whose path is inside the worktree, and a read of the
bare worktree root (an `ls` of the workspace) is allowed. Found by the LP-A1
container run after Epic 019.13: the agent made real model calls and targeted the
correct worktree paths, but every write was blocked — the write-scope check
receives the **absolute** worktree path (`.../worktrees/implement-slugify/slugify.mjs`)
while `write_scope` entries are **repo-relative** (`["**"]`), and the check used a
literal-prefix match, so `["**"]` matched nothing and every write escalated
`re-planning-signal` → task parked, no commit, no PR. Separately the agent's `ls`
of the worktree root (`<wt>` with no trailing slash) was blocked by the role read
allowlist `<wt>/**`, which does not match the bare directory.

## Decision Anchors

- Live seam is `ring1/hook-binding.ts:115 isPathInScope` (via `makeWriteScopeCallback`),
  NOT `write-scope.ts:47` — that copy's `makeWriteScopeHook` has no production
  caller (legacy Epic 007 fake-tool path). The LP-A1 memo cited the dead copy.
- `role-path-policy.ts:428 ring1PolicyChain` calls `writeScopeCheck(effectivePath)`
  with the **canonicalized absolute** worktree path; `call.worktree` is in scope
  right there. `write_scope` entries come from the feature doc frontmatter
  (`run-loop.ts:359-360`) and are repo-relative directory prefixes (same frame as
  the parallel-lane lease namespaces, `scheduler/leases.ts normalizeScope`).
- Role read allow is `(sessionWorktreePath) + "/**"` (`run-loop.ts:421-427`);
  `globToRegex("<wt>/**")` compiles to `^<wt>/.*$`, which does not match the bare
  `<wt>` — so a read of the worktree root itself escalates `not-in-allowlist`.

## Stories

- `001-write-scope-accepts-worktree-writes.md` - the write-scope check compares
  in the repo-relative frame (relativize the absolute path against the worktree)
  and treats `**` as the whole-repo sentinel, so a worktree write with
  `write_scope: ["**"]` — and a scoped write_scope — is authorised. (B1)
- `002-worktree-root-read-allowed.md` - the role read/write allowlist includes the
  bare worktree root, so an `ls`/read of the worktree directory itself passes. (S1)

## Verification Gate

- `npm run typecheck` exits 0; `npm test` green — a test asserts a write to a path
  inside the worktree with `write_scope: ["**"]` is allowed (not escalated); a test
  asserts a scoped `write_scope` still matches the relativized in-worktree path and
  still blocks an out-of-scope path; a test asserts a read of the bare worktree
  root is allowed; existing ring-1 / run-loop tests pass; guard green.
- **Live proof (LP-A1):** the in-container run writes `slugify.mjs` + test in the
  worktree, commits, and the broker opens a real PR on `kanthord-verify`.

## Dependencies

- Epic 019.13 (ring-1 bound to the task worktree + brief names it), 019.8
  (per-task worktree), 019.12 (agent is driven).

## Non-Goals

- **No change to the role-policy glob engine or the tool surface** (015/019.1 own
  them) — only how the write-scope sub-check frames the path, and which allow
  entries the run-loop registry declares.
- **No dedupe of the two `isPathInScope` copies.** `write-scope.ts` stays as-is
  (its `EscalationEvent` type is still imported); its unused `makeWriteScopeHook`
  is out of scope.
- **No full glob syntax for `write_scope`.** Entries stay repo-relative directory
  prefixes; `**` is the only wildcard, meaning whole-repo.
