# 019.18 PR-Merge Escalation and Review-State Polling

## Outcome

When the daemon opens a PR for a delivered task, it **actively raises a review
request** through a pluggable `ReviewRouter` (default route = a user inbox
escalation: "PR #N ready — review & merge"), instead of silently waiting. A
**poller** then polls the PR's review/merge state (polling, not webhooks) so that a
merged PR drives the task to `complete` and a closed-unmerged PR escalates; the
review escalation is resolved when the PR reaches a terminal state.

This is the **2A logic**: the `ReviewRouter` seam + the default **user** route + the
PR-state poller. External review routes (LLM reviewer, colleague reviewer, …) are
later implementations against the same seam, and the operator UI is Epic 027 (2B/3) —
both non-goals here.

## Decision Anchors

- **Live gap (LP-A1, 2026-07-13):** delivery opens the PR and sets the task
  `delivering`, but raises **no** signal — `inbox_items` was empty, so an operator has
  no way to know a PR is waiting. The `git.push` op even reconciled to
  `escalation_needed` without producing an inbox item. And nothing polls the git
  platform to move the create_pr op to `merged`, so the observe-merge path
  (`src/daemon/run-loop.ts:549-556`) never fires in live. Memory `lp-a1-delivery-gap`.
- **Ulrich decision (2026-07-13):** a PR-merge **escalation mechanism** — default
  escalate to the **user**, with a **pluggable** route for external review (LLM
  review, colleague review, …). 2A builds the seam + user default; external routes are
  2B/3.
- **Ulrich decision (2026-07-13):** **poll**, do not use webhooks — polling gives the
  user more control and more flexibility over the review process, accepting the
  resource cost. The poll must be **bounded** (only while a task is awaiting review)
  and its interval configurable, to contain that cost.
- **Durable external-state tracking (AGENTS.md rule, Ulrich decision 2026-07-13):**
  the PR→task link and the "awaiting review" state MUST be **durable local rows**, not
  the in-memory `prOpTaskMap` (`run-loop.ts:275`). The poller's worklist and the
  task-transition both read those rows, so tracking **survives crash/restart**. On
  boot the rows are rehydrated/upserted via the existing `recoverFromLedger` →
  `needs_reconciliation` → `reconcileOp` seam. This is the durability spine for every
  future external integration, not a PR-only fix — hence a **generic**
  `external_tracking` projection that PRs specialize (see Story 000).
- **PR lifecycle is observed, not "op completion" (design decision, S1 2026-07-13):**
  `broker_completion` stays **immutable evidence that `create_pr` succeeded** (its
  natural terminal is "PR opened"). Later merge/close/review state is *observation of
  an external artifact* — the poller writes it to the `external_tracking` row and
  drives the task transition (`complete` / escalation) **from that row**, rather than
  mutating the create_pr completion. Retire `prOpTaskMap` as source of truth (keep at
  most a boot-rebuilt read-cache). Platform access is CLI-first (`gh`/`glab`), REST
  for gaps — memory `git-platform-integration-approach`.
- Escalations are inbox items (`createEscalationItem` → `inbox_items`,
  `src/inbox/inbox.ts`), surfaced by the approval-inbox RPC (Epic 017) over the
  Connect API (Epic 026) to a client (Epic 027).

## Stories

- `000-durable-external-tracking-projection.md` - introduce a durable, **generic**
  `external_tracking` projection (surrogate `id`; `local_kind`/`local_id`;
  `external_kind`/`external_provider`/`external_id`/`external_url`; `created_by_op_id`
  provenance; `idempotency_key`; `tracking_status` = local lifecycle;
  `observed_state_json` = external state; `next_poll_at`/`attempt_count`/
  `last_error_json` poll metadata; `created_at`/`updated_at`). On delivery **and on
  reconciliation** a PR row is upserted (`reconcileOp` MUST upsert the row before/with
  completing reconciliation — closes the crash-before-row window). Retire
  `prOpTaskMap` as source of truth; the observe-merge loop reads the projection. Keep
  the 2A column set lean — this is the shape, not a mandate to fill every field.
  (durable spine)
- `001-review-router-seam-user-route.md` - define a `ReviewRouter` seam and its
  default user route; on a successful delivery the run-loop routes a "PR ready" review
  request, which the user route records as an inbox escalation, and marks the task's
  `external_tracking` row `awaiting_review`. (seam + default route)
- `002-poll-pr-review-state.md` - a bounded poller selects **due, active** PR rows from
  `external_tracking` (`tracking_status='active' AND next_poll_at <= now`), queries the
  PR's state via the git platform, and writes the observed state back to the row;
  reaching a terminal state (`merged`/`closed`) drives the task transition **from the
  row** (`merged` → task `complete`, `closed` → escalation). `broker_completion` is not
  mutated post-delivery. Bounded by durable `next_poll_at`/`attempt_count` so a restart
  neither thunder-polls nor double-escalates. (polling)
- `003-resolve-review-escalation-on-terminal.md` - when the PR reaches a terminal
  state (merged or closed), set the `external_tracking` row `tracking_status='terminal'`
  and resolve the open review escalation so the inbox does not leak stale "PR ready"
  items (idempotent — resolve once even across restarts). (lifecycle)

## Verification Gate

- `npm run typecheck` exits 0; `npm test` green — a test proves that after a
  successful delivery the default `ReviewRouter` records a review-request inbox item
  carrying the task id + PR number/url; a test proves the seam accepts an alternate
  router (so external routes are pluggable); a test proves the poller, given a PR that
  reports merged, drives the task to `complete`, and a closed-unmerged PR escalates; a
  test proves the poller does not poll when no task is awaiting review; a test proves
  the review escalation is resolved on terminal.
- **Durability / failure-boundary tests (the crash-survival gate) — all required:**
  1. PR opened, crash **before** the `external_tracking` row is written → boot
     reconciliation (`reconcileOp`) inserts the row → the poller then finds it.
  2. Tracking row exists, in-memory `prOpTaskMap` empty (restart) → the poller still
     observes the terminal state and completes the task from the row.
  3. Poller git-platform failure persists `last_error_json` + advances `next_poll_at`
     (backoff) **without losing the tracking row**.
  4. A closed-unmerged PR escalates **exactly once**, not repeatedly after restart.
  5. A merged PR completes the task **exactly once**, idempotently after restart.
  Existing run-loop / broker / inbox tests pass; guard green.
- **Live proof (LP-A1):** after the daemon opens the PR, an inbox review-request item
  exists for the task; while the PR stays open the task stays `delivering`; **kill and
  restart the daemon while the PR is open — the poller resumes from the durable row**;
  when the operator merges the PR, the poller observes it and the task becomes
  `complete` and the review escalation is resolved.

## Dependencies

- Epic 019.16 (delivery opens the PR, task → `delivering`, observe-merge path,
  `prOpTaskMap`). Epic 017 (approval/escalation inbox) for the inbox surface.

## Non-Goals

- **No external review routes** — LLM reviewer, colleague reviewer, or any non-user
  route is a later implementation against the `ReviewRouter` seam (2B/3).
- **No webhooks** — the git platform is polled, not subscribed (Ulrich decision).
- **No operator UI** — surfacing the review escalation / merge action in a dashboard
  is Epic 027 (2B/3).
- **The daemon never merges the PR** — it only requests review and observes the
  terminal state (unchanged from Epic 019.16).
