# 019.18 PR-Merge Escalation and Review-State Polling

## Outcome

When the daemon opens a PR for a delivered task, it **actively raises a review
request** through a pluggable `ReviewRouter` (default route = a user inbox
escalation: "PR #N ready ‚Äî review & merge"), instead of silently waiting. A
**poller** then polls the PR's review/merge state (polling, not webhooks) so that a
merged PR drives the task to `complete` and a closed-unmerged PR escalates; the
review escalation is resolved when the PR reaches a terminal state.

This is the **2A logic**: the `ReviewRouter` seam + the default **user** route + the
PR-state poller. External review routes (LLM reviewer, colleague reviewer, ‚Ä¶) are
later implementations against the same seam, and the operator UI is Epic 027 (2B/3) ‚Äî
both non-goals here.

## Decision Anchors

- **Live gap (LP-A1, 2026-07-13):** delivery opens the PR and sets the task
  `delivering`, but raises **no** signal ‚Äî `inbox_items` was empty, so an operator has
  no way to know a PR is waiting. The `git.push` op even reconciled to
  `escalation_needed` without producing an inbox item. And nothing polls the git
  platform to move the create_pr op to `merged`, so the observe-merge path
  (`src/daemon/run-loop.ts:549-556`) never fires in live. Memory `lp-a1-delivery-gap`.
- **Ulrich decision (2026-07-13):** a PR-merge **escalation mechanism** ‚Äî default
  escalate to the **user**, with a **pluggable** route for external review (LLM
  review, colleague review, ‚Ä¶). 2A builds the seam + user default; external routes are
  2B/3.
- **Ulrich decision (2026-07-13):** **poll**, do not use webhooks ‚Äî polling gives the
  user more control and more flexibility over the review process, accepting the
  resource cost. The poll must be **bounded** (only while a task is awaiting review)
  and its interval configurable, to contain that cost.
- Existing seam to reuse: the create_pr op ‚Üí `broker_completion` ‚Üí observe-merge path
  already turns `merged`‚Üí task `complete` and `closed`‚Üí escalation. The poller feeds
  that path; no new terminal-transition logic is needed. Platform access is CLI-first
  (`gh`/`glab`), REST for gaps ‚Äî memory `git-platform-integration-approach`.
- Escalations are inbox items (`createEscalationItem` ‚Üí `inbox_items`,
  `src/inbox/inbox.ts`), surfaced by the approval-inbox RPC (Epic 017) over the
  Connect API (Epic 026) to a client (Epic 027).

## Stories

- `001-review-router-seam-user-route.md` - define a `ReviewRouter` seam and its
  default user route; on a successful delivery the run-loop routes a "PR ready" review
  request, which the user route records as an inbox escalation. (seam + default route)
- `002-poll-pr-review-state.md` - a bounded poller queries the PR's state via the git
  platform for tasks awaiting review and records the terminal outcome
  (`merged`/`closed`) into the create_pr op's completion, feeding the existing
  observe-merge path. (polling)
- `003-resolve-review-escalation-on-terminal.md` - when the PR reaches a terminal
  state (merged or closed), the open review escalation is resolved so the inbox does
  not leak stale "PR ready" items. (lifecycle)

## Verification Gate

- `npm run typecheck` exits 0; `npm test` green ‚Äî a test proves that after a
  successful delivery the default `ReviewRouter` records a review-request inbox item
  carrying the task id + PR number/url; a test proves the seam accepts an alternate
  router (so external routes are pluggable); a test proves the poller, given a PR that
  reports merged, records a `merged` completion that drives the task to `complete`,
  and a closed-unmerged PR escalates; a test proves the poller does not poll when no
  task is awaiting review; a test proves the review escalation is resolved on terminal.
  Existing run-loop / broker / inbox tests pass; guard green.
- **Live proof (LP-A1):** after the daemon opens the PR, an inbox review-request item
  exists for the task; while the PR stays open the task stays `delivering`; when the
  operator merges the PR, the poller observes it and the task becomes `complete` and
  the review escalation is resolved.

## Dependencies

- Epic 019.16 (delivery opens the PR, task ‚Üí `delivering`, observe-merge path,
  `prOpTaskMap`). Epic 017 (approval/escalation inbox) for the inbox surface.

## Non-Goals

- **No external review routes** ‚Äî LLM reviewer, colleague reviewer, or any non-user
  route is a later implementation against the `ReviewRouter` seam (2B/3).
- **No webhooks** ‚Äî the git platform is polled, not subscribed (Ulrich decision).
- **No operator UI** ‚Äî surfacing the review escalation / merge action in a dashboard
  is Epic 027 (2B/3).
- **The daemon never merges the PR** ‚Äî it only requests review and observes the
  terminal state (unchanged from Epic 019.16).
