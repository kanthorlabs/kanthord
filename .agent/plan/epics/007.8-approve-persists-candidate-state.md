# EPIC 007.8 — conflict-recovery loop works end-to-end via `approve`

> **Authored 2026-07-21** from the post-007.7 TODO-API real-model E2E
> (full run evidence + all findings in **Appendix A** below), then hardened
> through an adversarial debate (opencode/plan). The E2E built a working
> 5-endpoint TODO API from scratch (real CRUD smoke passed), but reached
> completion only after **manual DB surgery**: the guided conflict-recovery loop
> the CLI advertises is a dead end via the `approve` path.
>
> **Root cause (verified against code 2026-07-21):** `approve task` never
> persists the `landing_candidates.state`. On a merge conflict
> (`src/app/task/approve-task.ts:265`) it appends a `task.conflict` event and
> returns `{kind:"conflict"}` but never calls `updateCandidateState`; and on a
> clean land it never writes `state='landed'` either (create + delete landed to
> `main` yet their rows stayed `state='pending'`). The approve preview/CAS path
> diverged from the `integrate()` path (`src/landing/git.ts:223`) which _does_
> persist state. Because `get conflict` (`get-conflict.ts:81`) and `retry`
> (`retry-task.ts:87`) both gate on `state==='conflict'`, the whole loop is
> unreachable. **Proven fix:** manually setting `state='conflict'` made
> `get conflict → retry → daemon rebuild → approve` complete all 3 stranded
> tasks → 5/5, with no other code change.
>
> Secondary: the conflict message advertises `retry task --id X --note "…"` but
> the commander `retry task` command exposes only `--id`
> (`src/apps/cli/commands/retry/task.ts:12`); the use case already accepts
> `note` (`retry-task.ts:70`) — a pure CLI-wiring gap.

## Goal

After 007.8, the guided conflict-recovery loop works through the **documented
commands only**, with no DB surgery. `approve task` persists the candidate
lifecycle state as the **last observed approve outcome**: `state='landed'` after
the ref actually advances, `state='conflict'` after a preview against the
current target returns a genuine conflict. Because `get conflict` and `retry`
read that state, the loop closes:

`approve` (conflict) → `get conflict --id` renders the overview → `retry task
--id [--note "…"]` recovers the candidate to `pending` and stores the note →
`run daemon` rebuilds on the fresh base → `approve` lands.

A multi-sibling initiative whose siblings edit one file can therefore reach
**full completion via documented commands alone**. `state='conflict'` is the
last persisted approve _decision_, not an eternal git fact — `get conflict`
re-previews live, so a candidate that became mergeable (target moved) reports no
conflict and the user simply runs `approve` again. This epic does **not** reduce
the number of conflicts, only make each one recoverable.

## Verification Gate

Gates: `npm run verify` (typecheck + test + verify:handoff + lint + db status).
Hermetic — fakes + real-git-in-temp; no network, no model.

- **S1 (approve persists state — the keystone).** An `ApproveTask` action/unit
  test asserting the candidate row loaded by approve transitions correctly for
  each of the four CAS-loop outcomes, each write atomic with its matching event
  / task transition (one `UnitOfWork`):
  - preview → **conflict**: candidate `state='conflict'`, written in the SAME
    transaction as the `task.conflict` event; task stays `awaiting_confirmation`.
  - `landPreviewed` succeeds → **landed**: candidate `state='landed'`, written
    inside the existing completion transaction (`approve-task.ts:312`) that also
    marks the task `completed`.
  - CAS mismatch mid-loop → **no state write**; re-preview against the new OID.
  - retries exhausted (`target_moved`) → **no state write**; candidate stays
    `pending` (contention is not a conflict — mislabeling it would push the user
    into a needless rebuild).
    The test must assert the **exact candidate id** approve loaded flips value; a
    fake that records "some state update" is too weak.
- **S2 (retry `--note` wired end-to-end).** A test that drives the **built
  commander command tree** (not just the handler `runRetryTask`, which already
  accepts `note`): `retry task --id X --note "g"` parses, recovers a
  `state='conflict'` candidate to `pending`, and the note surfaces on
  `get task --json`. A handler-only test would pass while the CLI stays broken.

Proof: deterministic, NO model. A self-contained fake-agent fixture produces a
**genuine** sibling merge conflict (distinct same-base edits at one anchor), then
drives the full recovery loop through documented commands to 5/5, asserting the
durable `conflict` and `landed` candidate states directly in SQLite (no CLI
exposes that internal field). `set -euo pipefail`; no `|| true`.

```bash
set -euo pipefail
WORK="$(mktemp -d)"; cd "$WORK"
export KANTHORD_DB=".data/kanthord.db"
REPO_ROOT="$OLDPWD"
K() { node "$REPO_ROOT/src/main.ts" "$@"; }
STATE() { node -e 'const{DatabaseSync}=require("node:sqlite");const db=new DatabaseSync(process.argv[1]);const r=db.prepare("SELECT state FROM landing_candidates WHERE task_id=? ORDER BY id DESC LIMIT 1").get(process.argv[2]);process.stdout.write(r?r.state:"NONE")' "$KANTHORD_DB" "$1"; }
TID() { K list task --initiative "$INIT" --json | node -e 'const t=JSON.parse(require("fs").readFileSync(0,"utf8"));const m=t.find(x=>new RegExp(process.argv[1]).test(x.title));process.stdout.write(m?m.id:"")' "$1"; }

K db migrate >/dev/null
PROJECT=$(K create project --name r | head -1)
HOME_REMOTE="$WORK/home.git"; git init -q --bare -b main "$HOME_REMOTE"
SEED="$WORK/seed"; git clone -q "$HOME_REMOTE" "$SEED"
git -C "$SEED" -c user.email=a@b.c -c user.name=t commit -q --allow-empty -m init
git -C "$SEED" push -q origin main
REPO=$(K create repository --project "$PROJECT" --name home --remote-url "file://$HOME_REMOTE" --branch main --auth ambient --path "$WORK/mirror" | head -1)
CREDVAL="$(mktemp)"; printf 'dummy' > "$CREDVAL"
CRED=$(K create credential --project "$PROJECT" --name c1 --provider openai-codex --value-file "$CREDVAL" | head -1)
PROV=$(K create ai-provider --project "$PROJECT" --name p1 --provider openai-codex --model gpt-5.6-terra | head -1)

# Graph shape (root "Create Task"; 4 siblings depend on it) from the shared authorer.
GRAPH="$WORK/g"; "$REPO_ROOT/scripts/e2e/make-landing-graph.sh" "$GRAPH" >/dev/null

# CONFLICT FIXTURE (the B2 fix — validated 2026-07-21): one fake bash turn,
# task-aware via the branch name (kanthord/<taskId>, unique per task). Every task
# overwrites ONE shared line with its own branch-keyed value. Two siblings built
# on the same base therefore change the SAME line differently → a real 3-way
# merge conflict (distinct *insertions* would auto-merge; a shared-line
# *overwrite* is what forces the conflict). Value is quoted → valid JS, so the
# sibling `node --check src/todo.mjs` verification passes.
cat > "$GRAPH/.fake-agent.json" <<'FAKE'
[
  { "toolCalls": [ { "name": "bash", "arguments": { "command":
    "mkdir -p src && printf \"export const endpoint = '%s';\\n\" \"$(git rev-parse --abbrev-ref HEAD)\" > src/todo.mjs"
  } } ] },
  { "text": "edited src/todo.mjs" }
]
FAKE

K import graph "$GRAPH" --create --project "$PROJECT" --bind source="$REPO" --bind provider="$PROV" --bind cred="$CRED" >/dev/null
INIT=$(node -e 'console.log(JSON.parse(require("fs").readFileSync(process.argv[1]+"/.kanthord-export.json","utf8")).initiativeId)' "$GRAPH")
export KANTHORD_FAKE_AGENT="$GRAPH/.fake-agent.json"

# 1. Build + approve root; assert its candidate row becomes 'landed' (A1 land case).
K run daemon --until-idle --poll-interval 200 >/dev/null
ROOT=$(TID "Create Task")
K approve task --id "$ROOT" >/dev/null
[ "$(STATE "$ROOT")" = "landed" ] || { echo "FAIL: root candidate not landed"; exit 1; }

# 2. Build all siblings on the same (root) base.
K run daemon --until-idle --poll-interval 200 >/dev/null

# 3. Walk the siblings through documented recovery to completion.
for NAME in "List Tasks" "Get Task" "Update Task" "Delete Task"; do
  SID=$(TID "$NAME")
  # conflict exits 0 by 007.5 design AND prints to stderr — capture both streams.
  OUT="$(K approve task --id "$SID" 2>&1)"
  if [ "$(STATE "$SID")" = "conflict" ]; then
    echo "$OUT" | grep -q "conflict" || { echo "FAIL: no conflict message for $NAME"; exit 1; }
    K get conflict --id "$SID" | grep -q "src/todo.mjs" || { echo "FAIL: get conflict empty for $NAME"; exit 1; }
    K retry task --id "$SID" --note "merge at anchor" >/dev/null
    K get task --id "$SID" --json | grep -q "merge at anchor" || { echo "FAIL: --note not surfaced for $NAME"; exit 1; }
    K run daemon --until-idle --poll-interval 200 >/dev/null
    K approve task --id "$SID" >/dev/null
  fi
  [ "$(STATE "$SID")" = "landed" ] || { echo "FAIL: $NAME candidate not landed after recovery"; exit 1; }
done

# 4. All five tasks completed via documented commands only.
DONE=$(K list task --initiative "$INIT" --json | node -e 'const t=JSON.parse(require("fs").readFileSync(0,"utf8"));console.log(t.filter(x=>x.status==="completed").length)')
[ "$DONE" -eq 5 ] || { echo "FAIL: only $DONE/5 completed"; exit 1; }
echo "007.8 PROOF OK"
```

## Stories

- **Story 1 — `ApproveTask` persists candidate lifecycle state (keystone / A1).**
  Give `ApproveTask` explicit ownership of candidate-state persistence on the
  preview/CAS path. Write `state='conflict'` in the conflict-return branch,
  inside the same `UnitOfWork` as the `task.conflict` event; write
  `state='landed'` inside the existing completion transaction that marks the
  task `completed`. Write nothing on CAS mismatch or `target_moved` (see S1 for
  the exact four-outcome table). No new landing path; two placed writes.
- **Story 2 — wire `--note <text>` into the `retry task` command (A4).**
  Add the `--note` option to `src/apps/cli/commands/retry/task.ts` and pass it
  through to `runRetryTask` (which already accepts it). Tested at the command
  tree, per S2.

## Non-goals

- **Auto-serialization of same-file siblings (Group B).** This epic makes each
  conflict _recoverable_; it does not _prevent_ avoidable conflicts. Auto-
  serialization needs a safe write-set signal (siblings do not always touch the
  same files) and a scheduler design — its own later epic with its own proof.
- **Structurally unifying the `approve` CAS path and `integrate()`** into one
  landing implementation. 007.8 makes them consistent on _state persistence_
  only; the `integrate()` path also records an `Integration` that approve does
  not — note as follow-up, do not merge the paths here (that would regress the
  007.6 atomic-CAS safety).
- **Durable conflict snapshots.** `retry-task.ts` reads `targetOID` /
  `conflictContext` from the candidate, but neither is persisted on
  `ChangeCandidate` / the `landing_candidates` row — so the 007.6 rebuild prompt
  currently gets empty context. Making that durable is out of scope; this epic
  must not claim it. (Recovery still works: the daemon rebuilds from the fresh
  base regardless.)
- **C1** (two-level `<cmd> <sub> --help` prints root help), **C2** (approve-on-
  conflict exits 0 — 007.5 by-design), **C3** (`get initiative`/`objective`
  reads), **C4** (`find task`). Unrelated to the critical loop; excluded to keep
  007.8 single-purpose.

## A.3 Group A — Guided conflict recovery is broken end-to-end (CRITICAL)

The 007.5 + 007.6 recovery loop is unreachable when the conflict is surfaced by
`approve`. The conflict message says:

> `conflict: … merge conflict in src/todo.mjs; inspect: get conflict --id <id>;
guide: retry task --id <id> --note "<guideline>", then re-run daemon and approve`

Every command in that guidance fails:

- **A1 — CRITICAL — approve never persists candidate lifecycle state (neither
  `conflict` NOR `landed`).** `approve-task.ts:265` appends a `task.conflict`
  event and returns `{kind:"conflict"}` but never calls `updateCandidateState`;
  the CAS `preview`/`landPreviewed` path diverged from `integrate()`
  (`landing/git.ts:223`) which _does_ persist state.
  - Failure (conflict): after a conflicting `approve`, `SELECT state FROM
landing_candidates` = `pending`, not `conflict` → A2/A3 broken.
  - Failure (landed): create + delete LANDED to `main` yet their candidate rows
    stayed `state='pending'` — so the fix must persist the whole lifecycle.
- **A2 — MAJOR — `get conflict --id` returns "no conflict candidate found".**
  `get-conflict.ts:81` guards on `state !== "conflict"`; A1 makes it always throw.
- **A3 — MAJOR — `retry task --id` returns "not retryable".**
  `retry-task.ts:87` requires `state === "conflict"`; A1 makes it throw
  `TaskNotRetryableError` — the candidate can never be recovered.
- **A4 — MAJOR — `retry task --note` not wired into the CLI.** The message tells
  the user to pass `--note`; the use case (`retry-task.ts:70`) accepts it, but
  the commander command exposes only `--id` → `unknown option '--note'`.

One cluster: 007.6 shipped the guidance _message_ + conflict _prediction_ but
deferred the state persistence + CLI wiring (the 007.6 B1 deferral). The message
writes a cheque the code cannot cash. → **fixed by Story 1 + Story 2.**

## A.4 Group B — Sibling serialization forces avoidable conflicts (MAJOR, design)

All 4 siblings edit one file on the same base; approving the first moves `main`,
so the other 3 conflict. Quantified this run: `get` rebuild 9,414 tok, `list`
23,546 tok, `update` 11,583 tok — three full agent re-runs purely to resolve
avoidable same-file conflicts (confirms the 007.6 ≈61k figure). → **deferred**
(Non-goals: auto-serialization is its own epic).
