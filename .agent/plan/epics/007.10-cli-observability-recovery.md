# EPIC 007.10 — CLI observability & recovery ergonomics

> **Authored 2026-07-22** from the post-007.9 real-model TODO-API E2E (run
> `e2e-0710`; full evidence + all findings in `007.10-e2e-findings.md`), then
> hardened through an adversarial debate (opencode/plan — see "Debate deltas"
> at the bottom). The E2E drove the 5-endpoint TODO API from an empty repo to a
> passing CRUD smoke test with `openai-codex / gpt-5.6-terra`; the pipeline
> completed end to end, but four findings make the CLI hard to drive unaided —
> three of them (F1, F2, F4) leave the operator dependent on a DB-reading helper
> script or a doomed command to make progress.

## Goal

After 007.10 an operator can drive the full build → candidate → approve →
retry → land loop **from the CLI alone**, without the `e2e-status.sh` helper or
raw SQL, and without provoking a deliberate failure:

1. `get task` shows each task's landing-candidate lifecycle (`pending` /
   `conflict` / `landed`) and the base/candidate SHAs, so staleness is visible
   before approving.
2. A stale-based candidate can be rebuilt by explicit intent
   (`retry task --rebuild`) directly from `awaiting_confirmation`, instead of
   running a doomed `approve` just to force a `conflict`.
3. Every `list … --json` invocation emits exactly one parseable JSON document.
4. Nested-subcommand `--help` shows the subcommand's help across all command
   groups, and `login provider` reports its created credential consistently.

## Verification Gate

Gates: `npm run verify` (typecheck + test + verify:handoff + lint + db status)

Proof: one hermetic, copy-paste-runnable block against the built program — no
network, no real model, no credentials. Uses the fake agent runner and a
**local** git remote (like 007.9 Proof A), so it is deterministic. (F6's
`login provider` output line requires OAuth and cannot run hermetically; it is
covered by a unit test and asserted here only via its `--help`.)

```sh
set -euo pipefail
D=".data/e2e-0710-proof"; rm -rf "$D"; mkdir -p "$D"
# local bare "remote" seeded with one commit on main
git init -q --bare "$D/remote.git"
git clone -q "$D/remote.git" "$D/seed"
mkdir -p "$D/seed/src"; printf 'export const x=1;\n' > "$D/seed/src/todo.mjs"
git -C "$D/seed" add -A
git -C "$D/seed" -c user.email=e@e -c user.name=e commit -qm init
git -C "$D/seed" push -q origin HEAD:main
export KANTHORD_DB="$PWD/$D/k.db"
node src/main.ts db migrate >/dev/null
PROJECT=$(node src/main.ts create project --name obs-proof | grep -oE '01[0-9A-HJKMNP-TV-Z]{24}')
REPO=$(node src/main.ts create repository --project "$PROJECT" --name verify \
  --remote-url "file://$PWD/$D/remote.git" --branch main --auth ambient \
  --path "$PWD/$D/home" | grep -oE '01[0-9A-HJKMNP-TV-Z]{24}')
printf 'dummy-token' > "$D/cred.txt"
CRED=$(node src/main.ts create credential --project "$PROJECT" --name c1 \
  --provider openai-codex --value-file "$D/cred.txt" | grep -oE '01[0-9A-HJKMNP-TV-Z]{24}')
PROV=$(node src/main.ts create ai-provider --project "$PROJECT" --name p1 \
  --provider openai-codex --model gpt-5.6-terra | grep -oE '01[0-9A-HJKMNP-TV-Z]{24}')
scripts/e2e/make-landing-graph.sh "$D/graph" >/dev/null   # fake-agent graph (no model)
node src/main.ts import graph "$D/graph" --create --project "$PROJECT" \
  --bind source="$REPO" --bind provider="$PROV" --bind cred="$CRED" >/dev/null
INIT=$(node -e 'console.log(JSON.parse(require("fs").readFileSync(process.argv[1]+"/.kanthord-export.json","utf8")).initiativeId)' "$D/graph")
export KANTHORD_FAKE_AGENT="$PWD/$D/graph/.fake-agent.json"
node src/main.ts run daemon --until-idle --poll-interval 200
ROOT=$(node src/main.ts list task --initiative "$INIT" --json | node -e "let s='';process.stdin.on('data',d=>s+=d).on('end',()=>{process.stdout.write(JSON.parse(s).find(t=>t.dependencies.length===0).id)})")

# ── Story A — get task exposes landingCandidate (state + SHAs) ──
node src/main.ts get task --id "$ROOT" --json \
  | node -e "let s='';process.stdin.on('data',d=>s+=d).on('end',()=>{const t=JSON.parse(s);const c=t.landingCandidate;process.exit(c&&c.state==='pending'&&typeof c.baseSHA==='string'&&typeof c.candidateSHA==='string'&&typeof c.target==='string'?0:1)})" \
  && echo "PASS A: get task --json shows landingCandidate{state=pending,baseSHA,candidateSHA,target}" \
  || { echo "FAIL A"; exit 1; }
# human-readable output shows the state too
node src/main.ts get task --id "$ROOT" | grep -qiE 'landing.?candidate|candidate.*pending' \
  && echo "PASS A2: human get task shows candidate state" || { echo "FAIL A2"; exit 1; }

# ── Story B — retry --rebuild from awaiting_confirmation + pending, no doomed approve ──
node src/main.ts retry task --id "$ROOT" --rebuild 2>&1 \
  && echo "PASS B: retry --rebuild accepted from awaiting_confirmation+pending" \
  || { echo "FAIL B: retry --rebuild rejected"; exit 1; }
# the task is back in the queue as pending (rebuild will run on the current base)
node src/main.ts get task --id "$ROOT" --json \
  | node -e "let s='';process.stdin.on('data',d=>s+=d).on('end',()=>{process.exit(JSON.parse(s).status==='pending'?0:1)})" \
  && echo "PASS B2: task requeued to pending" || { echo "FAIL B2"; exit 1; }
# plain retry (no --rebuild) is still rejected for a non-conflict awaiting candidate:
node src/main.ts run daemon --until-idle --poll-interval 200   # rebuild → awaiting_confirmation again
# capture first (|| true): a guarded retry correctly exits 1, which would abort
# the whole Proof under `set -o pipefail` if piped straight into grep.
B3OUT=$(node src/main.ts retry task --id "$ROOT" 2>&1 || true)
printf '%s\n' "$B3OUT" | grep -qi "not retryable" \
  && echo "PASS B3: plain retry still guarded (explicit --rebuild required)" || { echo "FAIL B3"; exit 1; }
node src/main.ts approve task --id "$ROOT" >/dev/null
node src/main.ts get task --id "$ROOT" --json \
  | node -e "let s='';process.stdin.on('data',d=>s+=d).on('end',()=>{process.exit(JSON.parse(s).landingCandidate.state==='landed'?0:1)})" \
  && echo "PASS B4: landed candidate state visible via get task" || { echo "FAIL B4"; exit 1; }

# ── Story C — list event --json is ONE parseable document with events + nextCursor ──
node src/main.ts list event --after 0 --limit 1000 --json \
  | node -e "let s='';process.stdin.on('data',d=>s+=d).on('end',()=>{const o=JSON.parse(s);process.exit(Array.isArray(o.events)&&o.events.length>0&&typeof o.nextCursor==='string'?0:1)})" \
  && echo "PASS C: list event --json parses as one {events:[...],nextCursor} document" \
  || { echo "FAIL C"; exit 1; }

# ── Story D — nested --help shows subcommand help across all groups ──
for pair in "get conflict" "create ai-provider" "run daemon" "retry task"; do
  first=$(node src/main.ts $pair --help 2>&1 | head -1)
  case "$first" in
    "Usage: kanthord $pair"*) : ;;
    *) echo "FAIL D: '$pair --help' -> '$first'"; exit 1 ;;
  esac
done
echo "PASS D: nested --help shows subcommand help for get/create/run/retry"
node src/main.ts login provider --help 2>&1 | grep -q "kanthord login provider" \
  && echo "PASS D2: login provider --help unchanged (regression guard)" || { echo "FAIL D2"; exit 1; }

echo "007.10 PROOF OK"
```

## Stories

- **A — Landing-candidate visibility in `get task` (F1).** Add a nullable
  `landingCandidate` projection — `{ state, baseSHA, candidateSHA, target }` —
  to the `get task` read model, surfaced in **both** the human-readable and
  `--json` output. `state` is required; the SHAs justify staleness detection
  (Story B) and correlation with landed history. Implement via the read-only
  query path (CQRS-lite permits the one-row `landing_candidates` join); do
  **not** add landing fields to the `Task` domain entity. `null` when the task
  has no candidate.
- **B — Explicit rebuild of a stale candidate (F2).** Add `retry task
--rebuild` that requeues a task in `awaiting_confirmation` whose candidate is
  `pending` (the domain already permits `awaiting_confirmation->pending`; only
  the use-case guard blocks it). Leave the default `retry` path for `failed` /
  `conflict` unchanged, and keep plain `retry` (no flag) **rejected** for a
  non-conflict awaiting candidate so a valid candidate is never silently
  clobbered. Enqueue + candidate-state reset must be idempotent (no two
  competing "current candidates"). Set the candidate back to `pending` and
  reuse the existing rebuild-prompt path. (Optional stretch, only if cheap:
  compare `candidate.baseSHA` to the target head so `--rebuild` can warn/no-op
  when the candidate is **not** actually stale.)
- **C — Uniform `--json` for `list event` (F4).** Binding rule: **each `list …
--json` invocation emits exactly one parseable JSON document.** Change `list
event --json` from JSONL-plus-trailing-`{nextCursor}` to a single envelope
  `{ "events": [...], "nextCursor": "..." }` (the envelope, not a bare array,
  because an empty page has no last-event id from which to derive the cursor).
  Migrate **every** current 007.7 `list event --json` consumer, test, and doc
  in this same story (`e2e-status.sh` reads the DB directly and is unaffected;
  verify). The non-`--json` human format is unchanged.
- **D — Nested `--help` + `login provider` output consistency (F5, F6).** Apply
  the `--help` wiring that already works for `login provider` uniformly to the
  `create` / `get` / `run` / `retry` (and other) command groups so `<group>
<sub> --help` prints the subcommand's help, not root help — proven across the
  full command matrix. Add a `credential created: <id>` line **on stderr** to
  `login provider` (leaving the bare id on stdout, matching the `create *`
  id-on-stdout / friendly-line-on-stderr contract).

## Non-goals

- **F3 — deterministic sibling-conflict fast path is a separate EPIC 007.11**,
  not part of this bug-fix epic. Measured cost this run: ~7 model builds for 5
  endpoints (3 extra rebuilds purely from N siblings editing one file). The fix
  (attempt a deterministic 3-way rebase onto the moved base → run the existing
  verification gate → fall back to a model rebuild only on textual conflict or
  failed verification) needs its own design + proof; it is deferred on scope,
  not dismissed as low value.
- No change to the conflict _detection_ / recovery loop itself (007.8) beyond
  exposing its state (Story A) and the explicit-rebuild entry point (Story B).
- No new `get candidate` / `list candidate` command family (Story A on
  `get task` covers the operator's first reach; a dedicated command is YAGNI
  until a driver needs candidate-only listing).
- No auto-retry-on-approve-conflict (hides the base-moved fact from the
  operator).

## Debate deltas (opencode/plan, 2026-07-22)

The first-draft fix designs were sharpened by an adversarial debate; the
material corrections now baked into the stories above:

- **F2 reversed** from "silently relax the retry guard" to **explicit
  `retry task --rebuild`**: `pending` ≠ stale, so a blanket relaxation could
  clobber a valid candidate, waste a model run, and hide the approve path. The
  real defect is that staleness is discovered only at approve time — hence
  surface `baseSHA` vs target (Story A) and require explicit rebuild intent,
  with idempotent enqueue/candidate replacement.
- **F4 rule reframed** from "make all list commands the same shape" (false — an
  envelope still differs from `list task`'s array) to **"each `--json`
  invocation emits one parseable JSON document"**; envelope chosen over bare
  array because `nextCursor` must survive empty pages; **all 007.7 consumers
  migrate in the same story** and the Proof checks `nextCursor`.
- **F1 completeness**: expose `landingCandidate` in **both** human and `--json`
  output (not `--json` only); `get task` is the right home (one-row join, not a
  hot path); minimal named projection.
- **F3 deferral corrected**: model rebuild also risks semantic error, so the
  honest comparison is deterministic-rebase-plus-verify vs model-rebuild — F3
  becomes its own EPIC 007.11 with measured cost, not a vague backlog note.
- **F6 reframed** (not deferred): our contract is id-on-stdout /
  friendly-on-stderr, so add the stderr line only — preserves script-friendly
  command substitution while closing the consistency gap. Help proof must cover
  the full command matrix, not one example.
