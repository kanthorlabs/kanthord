# EPIC 007.5 — conflict recovery ergonomics (N1)

> **Coupled ergonomics epic (Ulrich, 2026-07-21).** Authored from the post-007.4
> TODO-API E2E — a real 5-endpoint CRUD feature run through the CLI against
> `gpt-5.6-terra` (ChatGPT Plus, effort=medium) on the DEFAULT relative DB. Full
> E2E findings + debate record are in **Appendix A/B** below. 007.4's landing
> fixes (F1/F2/F3) all held:
> absolute workspaces, `task.conflict` schema, and typed approve outcomes worked
> end-to-end. What the E2E exposed is a **recovery UX gap** after a conflict.
> Scope was **debate-hardened** (opencode `plan`, read-only, 2026-07-21) — the
> debate rejected an auto-rebase mechanism as redundant and re-scoped this epic.
>
> **The E2E story:** 5 sibling tasks all edited `src/todo.mjs`. Root + first
> sibling landed (fast-forward). The remaining 3 siblings branched from the same
> base and, on approve, hit a **real content conflict** — `land()` already
> attempts a three-way merge and only returns `conflict` after that merge fails
> (`src/landing/git.ts:180-230`); it durably marks the candidate
> `state="conflict"` and records `conflictFiles`. The task stays
> `awaiting_confirmation` (correct, recoverable). But recovering it is the problem.
>
> **Two findings, one group ("conflict-completion"):**
>
> - **N1 (MAJOR) — recovery is undiscoverable + misleading.** The `approve`
>   conflict line says "re-run after resolving". The intuitive `retry task`
>   command **refuses** (`RetryTask` only accepts `failed` status;
>   `src/app/task/retry-task.ts:67`) with `not retryable (status:
awaiting_confirmation)`. The command that actually works is `reject task
--resolution retry` — semantically backwards (you "reject" to continue) and
>   never named by the message. A user following the guidance dead-ends.
> - **N2 (MAJOR, DEFERRED) — no sibling integration.** A true content conflict is
>   only resolvable by a full agent re-run on the updated base (~10-13k tokens per
>   sibling). The debate showed a mechanical rebase adds no recovery power (the
>   merge already exhausted the same 3-way information), and that a "rebased
>   candidate awaiting a second approve" needs durable state + re-verification +
>   candidate-production semantics — a structural change, not a smallest-change
>   item. **N2 is deferred to its own epic** (`007.6-guided-conflict-resolution.md`).
>
> **Load-bearing principle:** the intuitive command must do the obvious thing. A
> conflicted, recoverable task should be recoverable with `retry task` — keyed
> off the **durable candidate state** (`ChangeCandidate.state === "conflict"`),
> not the append-only `task.conflict` event (debate B6) — and the conflict
> message must name that command and the conflicting files. A git conflict is
> **not** a human rejection, so recovery must NOT emit `task.rejected` (debate
> concession).

## Goal

After 007.5, a task left `awaiting_confirmation` by a landing **conflict** is
recovered with the intuitive `kanthord retry task --id <id>`: it re-queues the
task to `pending` (so the daemon re-runs it on the now-updated base), supersedes
the stale conflict candidate/result so the re-run starts clean, and emits a
recovery event that is **not** `task.rejected`. `retry` accepts such a task
**only** when its latest landing candidate is durably `state="conflict"`; a
fresh, never-approved candidate stays non-retryable (unchanged error). The
`approve` conflict output names the exact recovery command and lists the
conflicting files. No auto-rebase, no re-verification-in-approve, no new durable
task state, no scheduler. `reject task --resolution retry` remains as the
human-driven path; `retry task` is the conflict-recovery path.

## Stories

- **S1 — Honest, actionable conflict message (N1a).** The CLI `approve` action's
  `conflict` formatter (`src/apps/cli/task.ts`) prints one sanitized line that
  names `kanthord retry task --id <id>`, lists the `conflictFiles` from the
  Integration/candidate record, and states the next step. No raw stack; exit 0.
- **S2 — `retry task` recovers a conflicted candidate (N1b).** `RetryTask`
  accepts an `awaiting_confirmation` task **iff** its latest landing candidate is
  durably `state="conflict"` (via `getCandidateByTask`); transitions it to
  `pending`, enqueues, supersedes the stale conflict candidate + result, and
  emits a recovery event (NOT `task.rejected`). Fresh candidates stay
  non-retryable; `failed` retry unchanged. `transitionTask` gains the
  `awaiting_confirmation → pending` edge (guarded to the conflict case in the use
  case). CLI `retry task` on a conflicted task exits 0.

## Non-goals

- **N2 / sibling integration** — auto-rebase, auto-serialization of repo-bound
  siblings, or a targeted conflict-resolution run. The debate showed the
  mechanical rebase is redundant (the 3-way merge already ran) and any
  "resolve + re-land without a full agent re-run" needs durable candidate
  production + re-verification. Deferred to `007.6-guided-conflict-resolution.md`.
- **Re-verification inside `approve`/`land`** — verification lives in the agent
  runner and runs pre-proposal; adding a post-integration verify to the approve
  path is a new capability (debate B4), out of scope.
- **Changing the existing auto-merge-on-approve behavior** — `land()` already
  lands a clean 3-way merge commit the human did not separately review (debate
  B5). Whether that should require a second confirmation / post-merge verify is a
  **separate latent question (N4)**, recorded in Appendix A, not fixed here.
- **N3 — `list task --json` omits `dependencies`** — observability; batched with
  BUG-4/BUG-6 in their own epic.
- No new durable task lifecycle state; no worker scheduler.

## Appendix B — debate deltas (opencode plan, read-only, 2026-07-21)

The debate REJECTED the originally-proposed N2 mechanism (mechanical
rebase-and-reland); all six critiques were verified against the source:

- **Drop the mechanical rebase (B1).** `GitRepositoryLanding.land()` already does
  a 3-way `git merge` for every non-ff candidate and auto-lands it when clean;
  `conflict` is returned only after that merge fails. Cherry-picking the same
  single-parent proposal commit onto the same base reproduces the same conflict —
  zero added recovery power.
- **A "rebased candidate awaiting a 2nd approve" needs durable state (B2/B3) and
  re-verification (B4)** — candidate production, not landing; belongs in N2's
  structural epic, not a smallest-change fix.
- **The N4 latent question (B5)** — approve already lands an unreviewed merge
  commit with no re-verify; singling out rebases for stricter treatment is
  inconsistent → recorded as N4.
- **Retry discriminator is the durable candidate `state`, not the `task.conflict`
  event (B6)** — the event is append-only history and stays true after recovery;
  `getCandidateByTask(id).state === "conflict"` is the direct signal.
- **Concessions accepted:** `retry` may accept a conflict-marked
  `awaiting_confirmation` task; recovery must requeue WITHOUT `task.rejected`.
- **Scope:** N1 now (this epic); N2 as its own structural epic — not here.
