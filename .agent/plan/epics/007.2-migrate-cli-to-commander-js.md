# EPIC 007.2 — Migrate CLI to Commander.js

## Goal

Replace the hand-written `node:util.parseArgs` router in
`src/apps/cli/router.ts` with Commander.js as the single CLI parser and router.
The CLI keeps the existing Ports and Adapters boundary: Commander.js parses and
routes only; each leaf command lives in its own file, calls an injected use case
or existing thin CLI handler, and formats the result. `src/apps/cli/index.ts` is
the composition point for the command tree, not a second command implementation.

The public command grammar follows the Kubernetes-style
`kanthord <verb> <resource> [options]`, using singular resource names consistently.
The only namespace exceptions are `kanthord db <action> [options]` and
`kanthord config <action> [options]`. There are no configuration commands today,
so this epic only documents the `config <action>` naming convention for future
commands; it registers no `config` group and adds no test for it (a group with
zero leaves would be the empty command the Non-goals forbid). Every program, group, and leaf command has a clear description;
every argument and option has help text; and every leaf help page includes a
concrete usage example. Parse errors remain strict and actionable.

The current custom router, production composition, tests, and handler files are
the command source of truth. The partial Commander.js prototype in
`src/apps/cli/index.ts` is stale: it contains only `check graph`, `db migrate`, and
`db status`, and is not called by `src/main.ts`. The migration audits every
registered custom-router command against its use case and handler before adding
it to the Commander.js tree. The audited target command set is:

```text
check graph
db migrate | db status
create project | create initiative | create objective | create repository
create credential | create notification | create ai-provider | create filesystem
create task
rename project | rename initiative | rename objective
pause initiative | resume initiative
add dependency | remove dependency
retry task | approve task | reject task
get task | get project | get resource
find project | find initiative | find objective | find resource
list task | list initiative | list objective | list event | list model
update ai-provider | update credential | update repository
update notification | update filesystem
import resource | import graph | export initiative
login provider
run daemon | export diagnostic | land repository
```

The five changed spellings are deliberate grammar fixes:
`daemon run` → `run daemon`, `events` → `list event`, `get models` →
`list model`, `diagnostics export` → `export diagnostic`, and `repo land` →
`land repository`. `login <provider>` becomes
`login provider --provider <provider>` so the route remains verb + resource and
the provider value is explicit. The old spellings are removed rather than kept as
hidden aliases; this is a pre-release CLI and there is no persisted command
compatibility contract.

The `get models` → `list model` rename has one non-CLI consequence: the
`UnknownModelError` message in `src/model-catalog/port.ts` tells the user to
``Run `get models` ...``, and a locked test (`src/model-catalog/port.test.ts`)
asserts that verbatim substring. This epic must update that message to point at
`list model` and update the assertion to match. This is the one deliberate edit
outside `src/apps/cli/`; it is a route-pointer fix, not a change to any handler
output format, and it is exempt from the "keep formatting unchanged" rule below
precisely because leaving it would point users at a removed command.

## Verification Gate

Gates: `npm run verify` (typecheck + test + verify:handoff + lint + db status).
Hermetic tests use a local `Command`, `.exitOverride()`, and configured output so
Commander.js never terminates the test process. They prove the complete audited
command inventory is registered exactly once; each leaf has a description,
documented arguments/options, and example help; async actions run through
`.parseAsync()`; required, repeatable, positional, boolean, and choice-like inputs
reach existing handlers with the same values; unknown commands/options, missing
values, and excess arguments fail on stderr with non-zero status; handler
stdout/stderr and exit codes are preserved; and the five old spellings are
rejected. Existing handler and end-to-end tests remain green. An architecture
test proves `src/apps/cli/index.ts` only builds the tree and that every leaf
command is registered from a separate command file.

Proof: run the real program against a temporary SQLite database, assert generated
top-level/group/leaf help, reject an old route, execute a real create/list
workflow through Commander.js, and exercise the highest-risk migrations — the two
renamed output routes (`list model`, `list event`) and a positional argument
(`export initiative <id>`).

```bash
set -euo pipefail
export KANTHORD_DB="$(mktemp -d)/kanthord.db"

node src/main.ts --help > /tmp/kanthord-help.txt
grep -q 'Usage: kanthord' /tmp/kanthord-help.txt
grep -q 'create' /tmp/kanthord-help.txt
grep -q 'db' /tmp/kanthord-help.txt

node src/main.ts create project --help > /tmp/create-project-help.txt
grep -q 'Usage: kanthord create project' /tmp/create-project-help.txt
grep -qi 'create' /tmp/create-project-help.txt
grep -q -- '--name <name>' /tmp/create-project-help.txt
grep -qi 'example' /tmp/create-project-help.txt

node src/main.ts db migrate --help > /tmp/db-migrate-help.txt
grep -q 'Usage: kanthord db migrate' /tmp/db-migrate-help.txt

if node src/main.ts daemon run --help >/tmp/old-route.out 2>/tmp/old-route.err; then
  echo 'FAIL: old daemon run route must be removed'; exit 1
fi
grep -qi 'unknown command' /tmp/old-route.err

node src/main.ts db migrate
PROJECT=$(node src/main.ts create project --name commander-proof)
INITIATIVE=$(node src/main.ts create initiative --project "$PROJECT" --name cli)
OBJECTIVE=$(node src/main.ts create objective --initiative "$INITIATIVE" --name routing)
TASK=$(node src/main.ts create task --objective "$OBJECTIVE" \
  --title 'migrate parser' --instructions 'Route the CLI with Commander.js.' \
  --ac 'Commander.js parses every command')

test "$(node src/main.ts find project --name commander-proof)" = "$PROJECT"
node src/main.ts list task --initiative "$INITIATIVE" --json > /tmp/tasks.json
node -e 'const rows=require("/tmp/tasks.json");process.exit(rows.length===1&&rows[0].id===process.argv[1]?0:1)' "$TASK"
node src/main.ts run daemon --help | grep -qi 'run'

# renamed output route (get models → list model): wires end to end, real catalog
node src/main.ts list model --json > /tmp/models.json
node -e 'process.exit(Array.isArray(require("/tmp/models.json"))?0:1)'

# renamed route over real SQLite (events → list event): queries events table, exits 0
node src/main.ts list event --after 0 --json

# positional-argument promotion (export initiative <id>, not --id)
EXPORT_DIR="$(mktemp -d)"
node src/main.ts export initiative "$INITIATIVE" --out "$EXPORT_DIR"
test -f "$EXPORT_DIR/cli/INDEX.md"
echo 'PROOF OK'
```

## Stories

1. **Audit and freeze the public command contract.** Compare `COMMANDS` in the
   custom router, production `RouterDeps`, composition wiring, handler tests, and
   the stale `src/apps/cli/index.ts`. Record every valid leaf command, input,
   requiredness rule, repeatable option, output stream, and exit behavior. Remove
   prototype-only assumptions and apply the target names listed in this epic.

2. **Commander.js program shell and result boundary.** Build a local `Command`
   in `src/apps/cli/index.ts`, set program name/description/version/help behavior,
   configure strict actionable parse errors, and use `.parseAsync()` from
   `src/main.ts`. Add one shared action adapter that writes the existing
   `{ exitCode, stdout, stderr }` result without moving business logic into the
   router. Tests configure output and override exits without global Commander.js
   state.

3. **One file per leaf command.** Create a grep-able command module for each
   audited leaf, grouped only by directories if useful. Each module constructs
   and returns one configured `Command`, declares its arguments/options, and owns
   its thin action callback. Parent verb/namespace builders only compose children
   with `.addCommand()`; `index.ts` only assembles the full tree. Existing handler
   functions may stay in their current files until a command is migrated, but no
   leaf registration or action logic remains in one central table.

4. **Kubernetes-style routes and namespace exceptions.** Register the canonical
   `<verb> <resource>` routes, migrate the five reversed/single-word routes and
   `login provider`, keep `db migrate|status`, and document `config <action>` as
   the only other action namespace (naming only — no `config` group is
   registered). Reject the old names with no aliases. Update only **live**
   repository-owned callers of changed routes: `docs/flowchart/*.md`,
   `README.md`, runnable `scripts/`, and current test/verification commands. Do
   **not** rewrite historical records — `.agent/tdd/history/**`,
   `.agent/tdd/memory/**`, and the Proof/story text of already-closed epics
   (e.g. `007.1`) — they document what was run at the time and must stay verbatim.
   This step also includes the `UnknownModelError` message + locked test edit
   from the Goal, since that string is a live pointer to the renamed route.

5. **Complete generated help.** Give the program, every parent group, and every
   leaf a user-focused description. Describe every argument and option, including
   valid values, defaults, requiredness, repeatability, and side effects where
   relevant. Add at least one copy-paste example to each leaf with
   `.addHelpText("after", ...)`; examples must use the canonical route and must
   never place credential values in argv. Test help structurally from the built
   command tree and snapshot a small representative set rather than snapshotting
   all Commander.js whitespace.

6. **Behavior parity and removal of the custom parser.** Migrate all commands,
   including positional graph import/export inputs, repeated task/import options,
   credential stdin/TTY handling, event follow cancellation, and daemon options.
   Keep use-case calls, formatting, security rules, and exit semantics unchanged.
   Delete `src/apps/cli/router.ts` and obsolete router-only tests only after the
   full Commander.js inventory and existing end-to-end suite pass.

7. **Program-level help and workflow proof.** Add a hermetic spawned-process test
   for top-level, group, and leaf help plus one real SQLite workflow. Run the exact
   Proof block above and keep `npm run verify` green.

## Non-goals

- No new domain capability, use case, persistence schema, resource type, or
  configuration action. `config <action>` defines naming for future commands; it
  does not justify an empty or fake command today.
- No changes to business rules, handler output formats, database behavior,
  credential handling, daemon behavior, graph format, or event semantics. The one
  exception is the `UnknownModelError` route pointer noted in the Goal (`get
models` → `list model`), which is a wiring fix, not an output redesign.
- No renaming of status/label strings inside handler output to match the new
  routes. The `diagnostics.ts` success line still reads `diagnostics export: …`
  after the `export diagnostic` rename; keeping output byte-stable is the
  deliberate choice, and this cosmetic drift is accepted, not overlooked.
- No stand-alone executable subcommands or child-process-per-command design.
  Command modules are in-process builders added with `.addCommand()`.
- No generic command-definition DSL, reflection, glob auto-registration, or DI
  container. Explicit imports and command construction remain the routing map.
- No compatibility aliases for the five old routes or the old positional login
  syntax. Repository-owned callers are migrated in the same epic.
- No broad CLI visual redesign, shell completion, interactive prompts, colors,
  telemetry, or new global options beyond help/version behavior needed for this
  migration.
