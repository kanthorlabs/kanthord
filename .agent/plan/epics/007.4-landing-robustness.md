# EPIC 007.4 — landing robustness (default-config land path)

> **Coupled bug-fix epic (Ulrich, 2026-07-21).** Three defects surfaced by the
> post-007.3 TODO-API E2E — a real 5-endpoint feature run through the CLI against
> `gpt-5.6-terra` (ChatGPT Plus, effort=medium). Full findings in the Appendix.
> The agent produced high-quality code and the candidate→`awaiting_confirmation`
> gate held, but **no task can land on a default install**, and any landing that
> is not a clean fast-forward **crashes the `approve` command**. Design
> directions are debate-hardened (opencode `plan`, read-only, 2026-07-21).
>
> **Load-bearing principle:** the out-of-the-box CLI (default relative
> `KANTHORD_DB=.data/kanthord.db`) must land, and every landing outcome —
> success, expected conflict, and unexpected failure — must be a **typed result
> the caller can act on**, never an uncaught throw. The 007.3 `landing-proof.sh`
> is green only because it uses an ABSOLUTE `mktemp` DB; it never exercised the
> default relative path, so these shipped. This epic closes that gap.
>
> **Three defects (severity from the E2E):**
>
> - **F1 (BUG-1, CRITICAL)** — landing runs `git fetch <workspace> <sha>` with
>   `cwd=<mirror>`; `candidate.workspace` is RELATIVE (`.data/workspaces/<id>`)
>   because `workspaceRoot = join(dirname(dbPath), "workspaces")` and the default
>   `dbPath` is relative → git resolves it against the mirror → fatal. Landing is
>   dead on a default install.
> - **F2 (BUG-5, CRITICAL)** — `task.conflict` is in domain `EVENT_TYPES`
>   (event.ts) but ABSENT from the events table CHECK constraint
>   (migrations.ts). Any non-fast-forward landing hits `approve-task.ts` →
>   `feed.append(newEvent("task.conflict"))` → `CHECK constraint failed` → raw
>   crash, task stuck. Reproduced: 3 of 5 sibling tasks (they branch from the
>   same base and all edit `src/todo.mjs`).
> - **F3 (BUG-2, MAJOR)** — a non-conflict landing error (e.g. F1's fetch
>   failure) hits `throw err` and escapes as an uncaught exception: a raw Node
>   stack trace + non-zero exit, no actionable message, task left inconsistent.

## Goal

After 007.4, the CLI lands work **on a default install** and reports **every**
landing outcome as a typed, safe result. (1) **Paths** — the workspace-management
boundary produces **absolute** candidate workspace paths; landing **validates**
them (`isAbsolute`) and rejects invariant violations instead of reinterpreting
them against ambient cwd. (2) **Schema** — a forward table-rebuild migration adds
`task.conflict` to the events CHECK, preserving rows/indexes; a bidirectional
schema-contract test forbids future domain↔DB drift. (3) **Outcomes** —
`ApproveTask.execute()` returns a **discriminated result** (`approved` /
`conflict` / `landing_failed`); the driving adapter (CLI) formats it — a clean
redacted `error:`/conflict line, exit code by outcome — while the original cause
is retained in structured logs. A conflict leaves the task `awaiting_confirmation`
(recoverable); no new durable task state is added.

## Verification Gate

Gates: `npm run verify` (typecheck + test + verify:handoff + lint + db status).
Hermetic — no network. The suite proves each fix with fakes / real-git-in-temp:

- **F1:** `LocalWorkspaceManager` emits an absolute candidate `workspace` even
  when constructed with a RELATIVE root (`isAbsolute(candidate.workspace)`);
  `GitRepositoryLanding.land` throws a typed invariant error on a relative
  workspace path rather than shelling a doomed `git fetch`; a landing test runs
  the real adapter with a relative root+DB in a temp cwd and lands successfully.
- **F2:** a migration test asserts (a) every domain `EVENT_TYPES` member inserts
  into the migrated events table, (b) an unknown/misspelled type is REJECTED, (c)
  pre-existing event rows and indexes survive the rebuild. Explicit column lists;
  FK handling decided before `BEGIN` (not a no-op `PRAGMA` inside the txn).
- **F3:** `ApproveTask` returns a discriminated outcome — fake landing that
  fast-forwards → `approved`; that raises `LandingConflictError` → `conflict`
  (task still `awaiting_confirmation`, `task.conflict` event recorded, no throw);
  that raises a generic error → `landing_failed` carrying safe user text + an
  internal cause (no raw stack, task recoverable). `feed.append` failure while
  recording a conflict MUST NOT report `approved`. CLI action test: each outcome
  maps to the right exit code + a single sanitized line.

Proof: deterministic — pure CLI + real git in temp dirs, NO model — proving all
three outcomes on a **relative** DB path (the exact default-config gap). Uses the
fake-agent seam (`KANTHORD_FAKE_AGENT`) and `make-todo-graph.sh`. `set -euo
pipefail`; expected failures use `if ! …; then`.

```bash

# condition 007.3's proof missed (it used an absolute mktemp DB).

MIRROR="$WORK/mirror"
REPO=$(node "$REPO_ROOT/src/main.ts" create repository --project "$PROJECT" --name home \
        --remote-url "file://$HOME_REMOTE" --branch main --auth ambient --path "$MIRROR")

        | node -e 'let s="";process.stdin.on("data",d=>s+=d).on("end",()=>console.log(JSON.parse(s).find(t=>/Create Task/.test(t.title)).id))')

export KANTHORD_FAKE_AGENT="$GRAPH/.fake-agent.json"
node "$REPO_ROOT/src/main.ts" run daemon --until-idle --poll-interval 200

# --- F1: default RELATIVE DB path lands (this is what crashed before) ---
node "$REPO_ROOT/src/main.ts" approve task --id "$ROOT"
test "$(node "$REPO_ROOT/src/main.ts" get task --id "$ROOT" --json | node -e 'process.stdin.once("data",d=>console.log(JSON.parse(d).status))')" = "completed"
git -C "$MIRROR" cat-file -e "main:src/todo.mjs"           # actually landed

# --- F2 + F3(conflict): a second sibling that edits the same file must land as a
#     clean CONFLICT — exit 0, task still awaiting_confirmation, NO crash ---
node "$REPO_ROOT/src/main.ts" run daemon --until-idle --poll-interval 200
SIB=$(node "$REPO_ROOT/src/main.ts" list task --initiative "$INIT" --json \

test "$(node "$REPO_ROOT/src/main.ts" get task --id "$SIB" --json | node -e 'process.stdin.once("data",d=>console.log(JSON.parse(d).status))')" = "awaiting_confirmation"
```

## Stories

- **S1 — Absolute workspace paths + landing validation (F1).** Resolve
  `workspaceRoot` to absolute once in composition; make `LocalWorkspaceManager`
  emit absolute candidate `workspace` paths; `GitRepositoryLanding` validates
  `isAbsolute` and throws a typed invariant error (does NOT `resolve()` against
  ambient cwd). Legacy relative rows: backfill against the explicit
  `workspaceRoot` base in the migration, or declare unsupported.
- **S2 — Add `task.conflict` to events schema + drift guard (F2).**
  Transactional table-rebuild migration (explicit columns; FK decision before
  `BEGIN`; preserve indexes/rows) adding `task.conflict`. Bidirectional
  schema-contract test: all domain types accepted, unknown rejected, rows/indexes
  preserved.
- **S3 — Discriminated approve outcomes (F3).** `ApproveTask.execute()` returns
  `approved | conflict | landing_failed`; conflict keeps `awaiting_confirmation`
  and only reports success if the `task.conflict` append succeeds; generic
  failures return safe user text + retain internal cause in structured logs. CLI
  action maps outcome → exit code + one sanitized line. No raw stack traces.

## Non-goals

- No new durable "conflict" task lifecycle state (S7, debate) — `awaiting_confirmation`
  - typed outcome + event is enough; durable state is a separate product decision.
- No auto-rebase / auto-serialization of sibling tasks that share a repo file
  (cross-task coordination is a design question, deferred).
- No phase-aware landing recovery beyond documenting land re-run safety /
  idempotence (deep partial-failure recovery deferred — see Appendix B8).
- BUG-3 (events CLI) and BUG-4 (`list task` dependencies) — minor observability,
  deferred to their own epic.
- BUG-6 (rename `depends-on`/`dependsOn` → `dependencies`, ~196 sites, NO
  backward-compat — Ulrich 2026-07-21) — naming cleanup, deferred with BUG-4.
- No push to the remote (007.3 remains local-mirror landing).

## Appendix — E2E findings (2026-07-21)

Run: default DB `.data/kanthord.db`; project todo-api-e2e; repo kanthord-verify
(https-token PAT); ai-provider gpt-5.6-terra effort=medium via openai-codex
(ChatGPT Plus, OAuth through our own CLI — independent of the pi/Copilot auth).

**Worked:** OAuth login + credential persisted in kanthord DB; graph import (7
nodes, correct DAG); daemon ran gpt-5.6-terra (root: 6 turns, 11755/1700 tok),
produced a high-quality `src/todo.mjs` (node:http, exported handlers/store for
tests, body-size guard, JSON validation, main-module check); candidate →
`awaiting_confirmation` gate held; root landed via fast-forward after approve
(with a path workaround); first sibling (`delete`) landed ff.

**Bugs:** F1/BUG-1 (CRITICAL, composition.ts:376 + landing/git.ts:103), F2/BUG-5
(CRITICAL, migrations.ts:180 vs event.ts:15 + approve-task.ts:234), F3/BUG-2
(MAJOR, approve-task.ts:237). Minor: BUG-3 (no events CLI), BUG-4 (`list task
--json` omits dependencies). Cross-task observation: siblings depend only on the
root and all edit one file → serial conflicts (expected from the graph shape;
F2 turned the expected conflict into a crash).

**Debate deltas (opencode plan, 2026-07-21):** single source of path truth
(reject, don't ambient-resolve); FK-pragma-inside-txn is a no-op; explicit
migration columns; bidirectional (not one-way) schema-parity test; typed approve
outcomes over a generic catch; retain internal cause; don't add a conflict task
state; proof must cover the non-conflict failure path and stay on a relative DB.
