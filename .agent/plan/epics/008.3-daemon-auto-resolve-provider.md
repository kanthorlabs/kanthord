# EPIC 008.3 — Daemon auto-resolves the provider chain (drop per-task binding)

## Goal

The daemon stops requiring a per-task AI provider. When a task runs, the daemon
resolves the **provider chain** (008.2) for the task's project
(`task → initiative → projectId`) and runs on the **first** provider in the
chain, folding that provider's own credential. `ai_provider` and `credential`
are **removed** from the required task bindings (`repository` stays required),
the project-scoped `ai_provider` resource type is **retired**, and `login` is
**cut over** to register a global provider. Importing a graph no longer needs
`--bind provider` / `--bind cred`. This is one atomic cutover epic — runner
rewire, old-path removal, fixtures, docs, and login all move together so no
boundary is left in a mixed model. (Failover across the chain is 008.4 — this
epic always uses the first provider; the `credential` resource type and
repository `authCredentialId` are preserved for git auth.) It also accepts pi's
builtin **API-key** providers (`deepseek` / `openrouter` / `opencode`) against
the pinned catalog, and makes the OAuth refresh write-back a **credential-version
CAS** so a concurrent `logout`/`remove` (008.1) can never be undone by an
in-flight refresh.

## Verification Gate

Gates: `npm run verify` (typecheck + test + verify:handoff + lint + db status)

Proof: hermetic real daemon runs with the fake agent. Proves (1) an empty
registry fails a task with the exact typed provider-resolution error; (2) a task
runs on the project's **assigned, non-default** provider (proving the chain is
resolved from `task→initiative→projectId`, not "always the default"); (3) import
needs no provider/cred bind; (4) the retired `create ai-provider` command is a
hard unknown-command; (5) the `credential` + `authCredentialId` git-auth path
survives; (6) pi's builtin API-key providers (deepseek/openrouter/opencode) are
accepted against the pinned catalog while a bogus model is rejected.

## Stories

- **A — Daemon resolves the chain; runner takes a resolved provider.**
  `run-next-task` owns the resolution: from the task it reaches
  `initiative → projectId`, runs the 008.2 chain resolver, takes the first
  provider (with its folded credential), and hands it to the runner — the runner
  no longer reads `ai_provider`/`credential` from task context. (Story A defines
  the ownership boundary and data flow; the exact `AgentRunner`/`RunnerResolver`
  signatures are an implementation detail of this story, not fixed here.) Remove
  `ai_provider` and `credential` from `EXECUTOR_BINDING_SPECS` (repository stays
  required). Redaction uses the resolved provider's folded credential value. The
  resolved provider id is emitted on the `agent.started` event (`providerId`) for
  observability and the Proof's selection check.
- **B — No-provider resolution fails loudly.** An empty resolved chain (no
  assignments and no default) fails the task with the exact typed error
  `no AI provider available for project <id>` — never a silent hang. Registering
  or assigning a provider makes the same graph succeed.
- **C — Retire the project-scoped `ai_provider` path (with a data policy).**
  Remove `ai_provider` from `RESOURCE_TYPES` / `buildResource`, delete the
  project-scoped `create ai-provider` / `update ai-provider` commands + use cases
  so `create ai-provider` is a hard unknown-command, and drop the `provider`/
  `cred` binding aliases from `make-todo-graph.sh`. Migration 18 is a full retire:
  it deletes stale `resources` rows with `type='ai_provider'` **and** the stale
  `task_context` rows that referenced them (that column has no FK), **and**
  rebuilds the `resources` table so its `type` CHECK no longer lists
  `'ai_provider'` (SQLite can't `ALTER` a CHECK) — so the literal is gone at the
  schema level, not just inert. The `credential` resource type and repository
  `authCredentialId` are explicitly untouched (regression-gated in the Proof).
- **D — Update the E2E scripts + docs.** Update `scripts/e2e/*` and the E2E skill
  flow to the new setup (`register` + `assign`, no provider/cred bind), and
  document in AGENTS.md that provider selection is daemon-resolved from the
  project chain, not a task binding.
- **E — Cut `login` over to the global registry.** `login` drops `--project` and,
  on a successful OAuth, persists a **global** provider (folding the returned
  secret) applying the first-wins default convention. `LoginProvider` and
  `RegisterAiProvider` (008.1) **share a common registry command / domain
  helper** — neither use case calls the other (the architecture forbids
  use-case-calls-use-case). Login is moved here, not 008.1, so the login→run path
  is only ever cut over together with the daemon.
- **F — Runtime support for pi's builtin API-key providers.** Registry
  validation of `deepseek` / `openrouter` / `opencode` against the pinned catalog
  is already the 008.1 contract; this story adds the **runtime** half —
  `PiProviderSessionFactory` builds a real `ProviderSession` for each via the
  API-key path, with hermetic session-construction tests for one exact pinned
  model per provider, explicitly covering `opencode`'s **mixed per-model API
  flavors** (a model whose `api` has no entry must fail clearly, not silently).
  Support is documented as "the pi-ai catalog pinned by this release." No OAuth
  for these — API-key only until a login+runtime-auth flow is built (deferred).
- **G — Credential-version CAS + running-task contract.** The OAuth refresh
  write-back (`saveCredentialValue`, `composition.ts:327`) becomes a full
  **compare-and-set**, replacing today's unconditional `void` update: at session
  start it reads `(state, credentialVersion)`; the write-back applies **only** in
  a single transaction where `id` matches, `state='active'`, and
  `credentialVersion = <expectedVersion>` — on success it increments the version
  and the session adopts the new expected version; on a mismatch (caused by
  `logout`/`remove`/re-login/another refresh) the write-back is a **no-op that
  does not fail the running attempt**. So an in-flight refresh can never resurrect
  a credential that `logout`/`remove` (008.1) invalidated. Contract for a
  `logout`/`remove` during a run: an **already-started attempt finishes with its
  captured credential** (a secret already in memory cannot be revoked); only
  **new** resolutions see the new state (logged-out excluded, per 008.2). The
  version-bump / stale-refresh-rejection is covered by unit tests (not observable
  from the shell Proof). Same-kind multi-account isolation is preserved by
  construction — each `sessions.for()` builds its own catalog and keys the
  write-back by the provider **record id**, never by provider kind.

## Non-goals

- No runtime failover across the chain — always the first provider (008.4).
- No change to the global registry, default convention (008.1), or the chain
  resolver (008.2).
- No change to notification/repository/filesystem bindings.
- No removal of the `credential` resource type (still used for git auth).
- No **custom / non-builtin** provider type and no OAuth for the builtin API-key
  providers — both deferred out of the 008.x series.
