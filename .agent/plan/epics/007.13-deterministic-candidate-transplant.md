# EPIC 007.13 — Deterministic stale-candidate transplant (no-model rebuild)

> **SUPERSEDED / DEFERRED (2026-07-23 workflow redesign).** EPIC 007.14
> (sequential objective-branch workflow) removes the intra-objective stale-base
> rebuild storm **by construction** — tasks build in order on one branch, so
> siblings never go stale against each other. That eliminates most of this
> epic's motivation. The **surviving remnant** is a _later_ optimization for the
> deferred topological-parallel execution mode (independent tasks built in
> parallel from the same tip, then an ordered integration queue): there, a
> deterministic transplant-then-verify still beats a model rebuild, with the
> approval-reinvalidation invariant below. Do **not** schedule this before 007.14
> ships and the parallel mode is actually pursued. Retained for that future.
>
> Authored 2026-07-22 from run `e2e-0711` finding F3 (`007.11-e2e-findings.md`)
> and the fix-approach debate (opencode/plan). The E2E cost **11 real-model
> builds for 5 endpoints** (6 extra) because every sibling that edited the same
> file was fully regenerated by the model whenever the base moved — even when
> the edits did not textually overlap. This epic makes the cheap, deterministic
> path the default recovery for a stale-but-non-conflicting candidate, with a
> hard safety invariant: a transplanted candidate is a **new** candidate and
> **must be re-approved**.

## Goal

After 007.13, recovering a stale-base candidate first attempts a **deterministic
3-way transplant** of the candidate onto the moved base (Git's merge machinery,
no model). If the transplant applies with **zero textual conflicts** and the
existing **verification gate passes** in an isolated workspace, the task returns
to `awaiting_confirmation` with a **new candidate SHA + new base SHA** and
**requires fresh human approval** — the prior approval, made against a different
base/diff, never carries over. On any textual conflict or gate failure it
**falls back to a full model rebuild** (today's behavior). The path taken is
recorded as an event so a human can audit deterministic-transplant vs
model-rebuild. Measured effect: the TODO-API shape drops from ~11 model builds
to ~5 (one per endpoint), with no loss of the human gate.

## Verification Gate

Gates: `npm run verify` (typecheck + test + verify:handoff + lint + db status)

Proof: hermetic, copy-paste-runnable with the fake agent runner (the "model")
and a local bare remote. Two fake-agent siblings edit **non-overlapping** parts
of one file; the Proof asserts the second is recovered by transplant **without a
model build** (build-count / event evidence), still requires approval, and that
an **overlapping** edit falls back to a model rebuild.

```sh
set -euo pipefail
D=".data/e2e-0713-proof"; rm -rf "$D"; mkdir -p "$D"
git init -q --bare "$D/remote.git"; git clone -q "$D/remote.git" "$D/seed"
mkdir -p "$D/seed/src"
printf 'export const A=1;\n// ---\nexport const Z=1;\n' > "$D/seed/src/f.mjs"
git -C "$D/seed" add -A
git -C "$D/seed" -c user.email=e@e -c user.name=e commit -qm init
git -C "$D/seed" push -q origin HEAD:main
export KANTHORD_DB="$PWD/$D/k.db"
node src/main.ts db migrate >/dev/null
PROJECT=$(node src/main.ts create project --name transplant-proof | grep -oE '01[0-9A-HJKMNP-TV-Z]{24}')
REPO=$(node src/main.ts create repository --project "$PROJECT" --name verify \
  --remote-url "file://$PWD/$D/remote.git" --branch main --auth ambient --path "$PWD/$D/home" | grep -oE '01[0-9A-HJKMNP-TV-Z]{24}')
printf dummy > "$D/cred.txt"
CRED=$(node src/main.ts create credential --project "$PROJECT" --name c1 --provider openai-codex --value-file "$D/cred.txt" | grep -oE '01[0-9A-HJKMNP-TV-Z]{24}')
PROV=$(node src/main.ts create ai-provider --project "$PROJECT" --name p1 --provider openai-codex --model gpt-5.6-terra | grep -oE '01[0-9A-HJKMNP-TV-Z]{24}')
# A dedicated fake-agent graph: two sibling tasks whose scripted candidates edit
# DIFFERENT, non-overlapping regions of src/f.mjs (top vs bottom), plus a third
# whose edit overlaps the first (to force the model-fallback branch).
scripts/e2e/make-transplant-graph.sh "$D/graph" >/dev/null   # NEW helper (Story D)
node src/main.ts import graph "$D/graph" --create --project "$PROJECT" --bind source="$REPO" --bind provider="$PROV" --bind cred="$CRED" >/dev/null
INIT=$(node -e 'console.log(JSON.parse(require("fs").readFileSync(process.argv[1]+"/.kanthord-export.json","utf8")).initiativeId)' "$D/graph")
export KANTHORD_FAKE_AGENT="$PWD/$D/graph/.fake-agent.json"

builds () { node src/main.ts list event --after 0 --limit 1000 --json | node -e "let s='';process.stdin.on('data',d=>s+=d).on('end',()=>{process.stdout.write(String(JSON.parse(s).events.filter(e=>e.type==='agent.finished').length))})"; }

node src/main.ts run daemon --until-idle --poll-interval 200 >/dev/null   # builds root + siblings
ROOT=$(node src/main.ts list task --initiative "$INIT" --json | node -e "let s='';process.stdin.on('data',d=>s+=d).on('end',()=>{process.stdout.write(JSON.parse(s).find(t=>t.dependencies.length===0).id)})")

B1=$(builds)

SIB=$(node src/main.ts list task --initiative "$INIT" --json | node -e "let s='';process.stdin.on('data',d=>s+=d).on('end',()=>{const t=JSON.parse(s).find(x=>/non-overlap/i.test(x.title));process.stdout.write(t.id)})")
node src/main.ts retry task --id "$SIB" --refresh 2>&1 | tee /tmp/ref.out
B2=$(builds)
# ── Story A/B — deterministic transplant ran WITHOUT a model build ──
[ "$B2" = "$B1" ] \
  && echo "PASS A: --refresh transplanted with NO new model build (agent.finished unchanged: $B1)" \
  || { echo "FAIL A: a model build ran (before=$B1 after=$B2)"; exit 1; }
node src/main.ts list event --after 0 --limit 1000 --json | node -e "let s='';process.stdin.on('data',d=>s+=d).on('end',()=>{process.exit(JSON.parse(s).events.some(e=>e.type==='candidate.transplanted')?0:1)})" \
  && echo "PASS A2: candidate.transplanted event recorded (auditable path)" || { echo "FAIL A2"; exit 1; }
# ── Story C — transplanted candidate requires FRESH approval (new SHAs, back to awaiting) ──
node src/main.ts get task --id "$SIB" --json | node -e "let s='';process.stdin.on('data',d=>s+=d).on('end',()=>{const t=JSON.parse(s);const c=t.landingCandidate;process.exit(t.status==='awaiting_confirmation'&&c&&c.state==='pending'?0:1)})" \
  && echo "PASS C: transplanted sibling is awaiting_confirmation+pending (must re-approve)" || { echo "FAIL C"; exit 1; }
node src/main.ts approve task --id "$SIB" >/dev/null \
  && echo "PASS C2: fresh approval lands the transplanted candidate" || { echo "FAIL C2"; exit 1; }

# ── Story A — overlapping stale sibling falls back to a MODEL rebuild ──
B3=$(builds)
OVL=$(node src/main.ts list task --initiative "$INIT" --json | node -e "let s='';process.stdin.on('data',d=>s+=d).on('end',()=>{const t=JSON.parse(s).find(x=>/overlap/i.test(x.title)&&!/non-overlap/i.test(x.title));process.stdout.write(t.id)})")
node src/main.ts retry task --id "$OVL" --refresh 2>&1 | tee /tmp/ref2.out
node src/main.ts run daemon --until-idle --poll-interval 200 >/dev/null
B4=$(builds)
[ "$B4" -gt "$B3" ] \
  && echo "PASS A3: overlapping candidate fell back to a model rebuild (agent.finished $B3 -> $B4)" \
  || { echo "FAIL A3: no model fallback on textual conflict"; exit 1; }

echo "007.13 PROOF OK"
```

## Stories

- **A — Deterministic transplant in the stale-candidate recovery use case.** In
  the recovery path (the use case behind `retry`, **not** the CLI), when a
  candidate is stale-but-`pending`: attempt a 3-way transplant of the candidate
  commit onto the current base with Git's merge machinery in an isolated
  workspace. On **zero conflicts**, run the existing verification gate. On green,
  produce a new candidate commit; append a `candidate.transplanted` event. On
  **any** textual conflict **or** gate failure, fall back to the existing model
  rebuild path (append the existing rebuild/retry events). No model invocation on
  the deterministic path.
- **B — New candidate identity + `retry task --refresh` interface.** The
  transplant persists a **new** candidate SHA **and** new base SHA (never mutate
  the old candidate row silently — preserve provenance). Add `retry task
--refresh`: deterministic-transplant-first, model-fallback. Keep `--rebuild`
  literal (always a model rebuild) so the flags are honest; plain `retry`
  behavior unchanged. Logic lives in the use case; the CLI only forwards the
  flag.
- **C — Approval reinvalidation (hard safety invariant).** A transplanted
  candidate returns the task to `awaiting_confirmation` with `candidate:pending`
  and **requires fresh human approval** — the prior approval, made against a
  different base/diff, must never authorize landing the transplanted result.
  (Optional, opt-in only: an approve-time auto-refresh that returns "candidate
  refreshed; approval still required" — never silent rebase-and-land.)
- **D — `make-transplant-graph.sh` helper + gate/audit wiring.** A fake-agent
  graph whose scripted sibling candidates edit non-overlapping vs overlapping
  regions of one file, so the Proof can exercise both the transplant and the
  model-fallback branches deterministically. Wire the `candidate.transplanted`
  event into the feed (string payload, matching the existing contract).

## Non-goals

- No trust-relaxation of the verification gate: the deterministic path is only
  as safe as the gate, so it must run the **same** gate the candidate was
  approved under, in isolation. If a candidate could weaken its own gate, that
  is a separate concern (flagged as a blocker, out of scope here).
- No pre-execution serialization / dependency-ordered sibling builds (a possible
  future design to avoid the quadratic re-verify cost); this epic only removes
  the _model_ cost of the rebuild, not all re-verification.
- No daemon-automatic transplant-and-land; recovery stays operator-initiated
  (or opt-in per Story C), preserving the human gate.

## Debate deltas (opencode/plan, 2026-07-22)

- **Approval reinvalidation is a hard invariant** (Story C): a human approved a
  candidate against a specific base; a rebase changes the effective diff, so the
  old approval cannot authorize the rebased result. This was the biggest gap in
  the first-draft approach.
- **Interface renamed** to `retry --refresh` (deterministic-first) so `--rebuild`
  can stay literally "run the model" — overloading `--rebuild` to sometimes skip
  the model is dishonest.
- **Logic in the use case, not the CLI** (the CLI is a driving adapter only).
- **New candidate identity** (Story B): transplant must persist new candidate/
  base SHAs and preserve audit history, not mutate the old candidate.
- **Semantic-breakage risk acknowledged but bounded**: a model rebuild can also
  introduce unrelated changes; the honest comparison is empirical
  (diff size, review burden, fallback rate) — Story A's event makes the path
  auditable so that comparison is measurable. Verification cost may stay
  quadratic (re-verify per moved base); that is a separate future optimization
  (Non-goals).
