# EPIC 008.4 — Agentic project onboarding

> Phase 4 of the former EPIC 008 (split 2026-07-18; onboarding carved out
> 2026-07-19 after two debates found it a separable control plane). An agent-driven
> loop that PRODUCES a confirmed `ProjectCapacity` — the discovery-based alternative
> to 008.2's deterministic `project capacity set`. Depends on 008.1 (generic agent +
> read-only tool policy + `replay` provider), 008.2 (`Project.capacity` storage +
> schema), and 008.3 (`RuntimeController` + host guard, for runtime trials). Phases
> 1–3: `008.1-role-agents.md`, `008.2-static-tdd-executor.md`,
> `008.3-runtime-verification.md`.

## Goal

`kanthord project onboard <project-id>` runs 008.1's generic coding agent in the
repo workspace as an iterative natural-language loop that produces a confirmed
`ProjectCapacity` for a repo whose commands are not obvious enough for 008.2's
conservative default. The agent inspects repo files with a READ-ONLY tool set (no
write/edit/mutating bash) confined to the workspace root, and emits a
schema-validated `ProjectCapacity` draft + EVIDENCE (the files justifying each
command); the user revises in natural language and the session re-drafts. The agent
may REQUEST a runtime validation, which runs ONLY after an explicit **per-trial
user approval** AND under 008.3's **durable per-project runtime-trust** — two
DISTINCT gates (neither substitutes for the other) — and ONLY through 008.3's
`RuntimeController` under the lease + host guard. Only an explicit user `OK`
persists `Project.capacity` + provenance (`confirmedAt`, `evidence`,
`evidenceFingerprint`, `runtimeTrial`); abandoning the session persists NOTHING
(session state is transient — no durable session record). Approval and `OK` cross
the confirmation boundary ONLY from the USER input channel — never from
agent-emitted text (an agent that types "OK" authorizes nothing). Confirmed
capacity commands are trusted project configuration thereafter (the 008.3 host
guard bounds them; OS containment is EPIC 009). Onboarding is OPTIONAL and
idempotent. **Drift** is surfaced by `project capacity check` as a
`suspected-capacity-drift` outcome (correlation, not proven causation) when a
confirmed command fails AND the cited-evidence fingerprint changed — an actionable
signal to re-onboard, not a definitive diagnosis and not a timer nag.

## Verification Gate

Gates: `npm run verify` (hermetic — the onboarding loop runs 008.1's agent on EPIC
006's `FakeSessionFactory` with scripted turns; runtime trials run through 008.3's
`RuntimeController`; no network. The suite covers: read-only inspection — the
onboarding tool set excludes write/edit/mutating bash and is workspace-root
confined, and a full-working-tree before/after snapshot (tracked + staged +
untracked, via 008.1's workspace service) is UNCHANGED; the emitted draft is
validated in two layers — STRUCTURAL (008.2's schema: `readiness` requires `start`,
non-empty `command`) and onboarding-SEMANTIC (a `filePatterns` matching nothing in
THIS repo is flagged) — an invalid draft is re-prompted, never persisted; the
AUTHORITY boundary — `OK`/approval are accepted ONLY from the user-input channel, so
a replay agent emitting "OK"/approval text authorizes nothing; the APPROVAL gates —
a runtime trial does NOT run without BOTH a per-trial user approval AND durable
per-project runtime-trust, and a standing host-trust alone never auto-runs a live
trial nor grants standing witness authority; the trial runs ONLY through
`RuntimeController` (sanitized-env canary absent) and its result is bound to the
persisted draft as `runtimeTrial` provenance (tested capacity digest, result,
timestamp) — not merely a `trialRan` flag; the PERSIST gate — only an explicit `OK`
writes capacity + provenance, abandonment writes nothing, session state is
transient; the `evidenceFingerprint` canonicalization (sorted normalized
repo-relative paths, file bytes, missing-file markers, symlink + duplicate handling,
path-escape rejection, named hash) is unit-tested; DRIFT — `suspected-capacity-
drift` fires ONLY when a confirmed command fails AND the fingerprint changed, and an
identical failure with UNCHANGED evidence is an ordinary failure (both branches
tested); idempotent no-change re-onboard leaves the fingerprint stable.)

Proof: deterministic gate (onboarding agent via `replay`, scripted user turns via
`--input-file`, a real Unix-socket runtime trial via 008.3). Every assertion is
mechanical and captures exit status where a command legitimately fails.

```bash
set -euo pipefail
PD="$(mktemp -d)"; jf() { node -e 'const t=JSON.parse(require("fs").readFileSync(process.argv[2],"utf8"));const v=process.argv[1].split(".").reduce((o,k)=>o==null?o:o[k],t);process.stdout.write(v==null?"":String(v));' "$1" "$2"; }
export KANTHORD_DB="$PD/kanthord.db"; node src/main.ts db migrate

# non-conventional repo (no `test` script) with a REAL Unix-socket runtime + integration test
REPO_DIR="$PD/svc"; mkdir -p "$REPO_DIR/bin" "$REPO_DIR/test"; git -C "$REPO_DIR" init -q
git -C "$REPO_DIR" config user.email t@e; git -C "$REPO_DIR" config user.name t   # hermetic identity
: > "$REPO_DIR/AGENTS.md"; printf '{ "scripts": { "verify:it": "node --test test/app.itest.js" } }\n' > "$REPO_DIR/package.json"
cat > "$REPO_DIR/bin/app.js" <<'JS'
const http=require("http"),fs=require("fs"),p=require("path");
const gen=fs.readFileSync(p.join(process.env.RUNTIME_DIR,"gen"),"utf8").trim();
http.createServer((_,res)=>res.end(gen)).listen(process.env.SOCK);
JS
cat > "$REPO_DIR/test/app.itest.js" <<'JS'
const t=require("node:test"),a=require("node:assert"),http=require("http");
t.test("app responds",()=>new Promise(r=>http.get({socketPath:process.env.SOCK,path:"/"},x=>{a.ok(x.statusCode===200);r();})));
JS
git -C "$REPO_DIR" add -A && git -C "$REPO_DIR" commit -qm init

PROJECT=$(node src/main.ts create project --name svc --json | jf id /dev/stdin)
node src/main.ts create resource --project "$PROJECT" --type repository --name svc --path "$REPO_DIR" --json >/dev/null
node src/main.ts create resource --project "$PROJECT" --type ai_provider --name openai --model gpt-5.5 --json >/dev/null
node src/main.ts create resource --project "$PROJECT" --type credential --name key --value replay --json >/dev/null
node src/main.ts project runtime trust "$PROJECT" --approve   # durable host-trust (008.3) — necessary, NOT sufficient

SNAP() { git -C "$REPO_DIR" status --porcelain -uall | sort; }   # full working-tree inventory (B1)
B4=$(SNAP)

# (1) authority + approval boundary: the replay agent EMITS "OK"+approval text, but the user channel DENIES.
#     Trial must NOT run; nothing persisted; repo untouched.
node src/main.ts project onboard "$PROJECT" --provider replay --script test/fixtures/onboard.replay \
  --input-file test/fixtures/onboard-deny.turns --json > "$PD/deny.json"
test "$(jf trialRan "$PD/deny.json")" = "false" || { echo "FAIL: trial ran without user approval (agent text spoofed it)"; exit 1; }
test "$(jf persisted "$PD/deny.json")" = "false" || { echo "FAIL: persisted without user OK"; exit 1; }
test "$(SNAP)" = "$B4" || { echo "FAIL: onboarding mutated the working tree"; exit 1; }
node src/main.ts project capacity show "$PROJECT" --json > "$PD/none.json"   # absent-capacity contract: exit 0, empty
test -z "$(jf tests.0.name "$PD/none.json")" || { echo "FAIL: capacity persisted on a denied session"; exit 1; }

# (2) full loop: user approves the trial (runs via RuntimeController) and OKs → capacity + provenance persisted
node src/main.ts project onboard "$PROJECT" --provider replay --script test/fixtures/onboard.replay \
  --input-file test/fixtures/onboard-approve-ok.turns --json > "$PD/ok.json"
node src/main.ts events --after 0 --project "$PROJECT" --json > "$PD/ev.json"   # independent trial record (B3/S4)
node -e 'const e=JSON.parse(require("fs").readFileSync(process.argv[1],"utf8")).events;
 process.exit(e.some(x=>x.type==="runtime.lifecycle"&&x.outcome==="ok"&&x.sanitizedEnv===true)?0:1);' "$PD/ev.json" \
 || { echo "FAIL: no real RuntimeController trial with sanitized env"; exit 1; }
node src/main.ts project capacity show "$PROJECT" --json > "$PD/cap.json"
test "$(jf tests.0.requiresRuntime "$PD/cap.json")" = "true" || { echo "FAIL: runtime kind not persisted"; exit 1; }
test -n "$(jf provenance.runtimeTrial.result "$PD/cap.json")" || { echo "FAIL: trial not bound to draft"; exit 1; }
FP=$(jf provenance.evidenceFingerprint "$PD/cap.json"); test -n "$FP" || { echo "FAIL: no fingerprint"; exit 1; }

# (3) idempotent stability FIRST (before any evidence change): no-change OK keeps the fingerprint
node src/main.ts project onboard "$PROJECT" --provider replay --script test/fixtures/onboard.replay \
  --input-file test/fixtures/onboard-nochange-ok.turns --json >/dev/null
test "$(node src/main.ts project capacity show "$PROJECT" --json | jf provenance.evidenceFingerprint /dev/stdin)" = "$FP" \
  || { echo "FAIL: no-change re-onboard changed the fingerprint"; exit 1; }

# (4) drift branch A: change a cited evidence file AND make the confirmed command fail → suspected-capacity-drift
node -e 'require("fs").writeFileSync(process.argv[1],"{ \"scripts\": { \"verify:it\": \"node --test test/nope.js\" } }")' "$REPO_DIR/package.json"
git -C "$REPO_DIR" commit -aqm drift
if node src/main.ts project capacity check "$PROJECT" --json > "$PD/driftA.json"; then echo "FAIL: check should exit non-zero on drift"; exit 1; fi
test "$(jf outcome "$PD/driftA.json")" = "suspected-capacity-drift" || { echo "FAIL: not flagged as suspected-capacity-drift"; exit 1; }

# (5) drift branch B: same command failure but evidence UNCHANGED → ordinary failure, NOT drift
git -C "$REPO_DIR" checkout -q -- package.json; git -C "$REPO_DIR" commit -aqm restore --allow-empty
rm -f "$REPO_DIR/test/app.itest.js"   # command fails, but no CITED-evidence (package.json) change
if node src/main.ts project capacity check "$PROJECT" --json > "$PD/driftB.json"; then echo "FAIL: check should exit non-zero on failure"; exit 1; fi
test "$(jf outcome "$PD/driftB.json")" = "command-failed" || { echo "FAIL: unchanged-evidence failure mislabeled as drift"; exit 1; }

echo "008.4 ok: read-only inspect, authority+approval gated trial, OK-gated persist, drift A vs B, idempotent"
```

## Stories

Depends on EPIC 008.1 (generic agent + read-only tool policy + `replay` provider),
008.2 (`Project.capacity` storage + `ProjectCapacity` structural schema +
`project capacity set`), 008.3 (`RuntimeController` + durable per-project
runtime-trust host guard + lease).

1. **Onboarding session loop (read-only inspection → draft → NL revision).**
   `project onboard <project-id>` drives 008.1's generic coding agent in the repo
   workspace with a READ-ONLY tool set (no write/edit/mutating bash), workspace-root
   confined; the repo working tree is provably unchanged (full-tree before/after
   snapshot). Each turn the agent emits a `ProjectCapacity` draft + EVIDENCE (the
   files justifying each command); the user revises in natural language and the
   session re-drafts. **Session state is TRANSIENT** — held in-memory for the run;
   abandonment persists NOTHING (no durable session record). Deterministic in tests
   via `replay` + a scripted `--input-file` of user turns. Two-layer validation
   (debate S3): STRUCTURAL via 008.2's schema (`readiness` requires `start`,
   non-empty `command`); onboarding-SEMANTIC here (a `filePatterns` that matches
   nothing in THIS repo is flagged). An invalid draft is re-prompted, never
   persistable.

2. **Authority + approval boundary.** `OK` and trial-approval cross the confirmation
   boundary ONLY from the USER input channel; agent-emitted text (even literal "OK")
   authorizes NOTHING — a tested authority boundary (debate B12). A runtime trial
   runs only when BOTH gates hold, and they are DISTINCT (debate B5): (a) 008.3's
   durable per-project **runtime-trust** (host commands MAY run at all) and (b) a
   per-trial explicit user **approval** (run THIS proposed trial now). Neither
   substitutes for the other: standing host-trust never auto-runs a live onboarding
   trial and never grants standing witness authority beyond the confirmed capacity;
   a per-trial approval never grants host-trust. Define one API per gate, each with
   its persisted decision, scope, and expiry.

3. **Approval-gated runtime trial (through `RuntimeController`) + trial provenance.**
   When both gates hold, the trial runs ONLY through 008.3's `RuntimeController`
   under the lease + host guard (no bespoke path); an independent record
   (`runtime.lifecycle` event, sanitized env) proves it — not a self-attested flag
   (debate B3/S4). The trial's outcome is bound to the persisted draft as
   `provenance.runtimeTrial { testedCapacityDigest, result, timestamp,
controllerDecision }` (debate S8) — a file-only fingerprint alone does not bind
   an approved trial to the confirmed capacity.

4. **Explicit-`OK` persist + provenance + fingerprint canonicalization.** Only an
   explicit user `OK` writes `Project.capacity` (the confirmed draft) + provenance
   `{ confirmedAt, evidence, evidenceFingerprint, runtimeTrial }`, atomically.
   **Migration ownership is exact** (debate B9): 008.2 owns the `Project.capacity`,
   `confirmedAt`, and `evidenceFingerprint` columns; 008.4 adds the `evidence` blob
   - `runtimeTrial` provenance columns (no session table — session state is
     transient). The `evidenceFingerprint` has a DEFINED canonicalization (debate S6):
     sorted normalized repo-relative paths, file bytes, explicit missing-file markers,
     symlink + duplicate handling, path-escape rejection, a named hash.

5. **Drift detection (`project capacity check`, correlation-honest).** A
   `project capacity check <project>` re-runs the confirmed commands and compares
   the current cited-evidence fingerprint. Outcomes: a confirmed command FAILS
   **and** the fingerprint CHANGED → `suspected-capacity-drift` (an actionable
   re-onboard signal, NOT proven causation — cited evidence is provenance/
   explanation, not full dependency coverage; a command may depend on lockfiles,
   env, tool versions the agent never cited — debate B6/S7); the same failure with
   an UNCHANGED fingerprint → `command-failed` (an ordinary failure, no drift nag).
   **Drift lives ONLY in `project capacity check`** — it does NOT alter any `tdd@1`
   transition or witness classification (debate B7: removes the earlier "inside a
   witness run" contradiction). Idempotent re-onboard: re-running against a
   confirmed capacity re-opens it; a no-change `OK` leaves the fingerprint stable.

6. **Project role-augmentation port (the seam 008.1 deliberately did NOT ship).**
   The hook by which confirmed project context adds PROJECT-SCOPED role CONTEXT to
   008.1's roles — landed HERE, with onboarding as its FIRST real consumer, so it is
   never an empty seam (debate 008-role B3). The port carries only NON-authority
   fields: advisory guidance / conventions per role, provenance-labelled and
   project-scoped. It is **PROHIBITED from carrying authority-bearing fields** — no
   tools, tool-argument widening, path/LANE reclassification, resource/credential
   bindings, or verdict/citation redefinition (008.1's executable authority envelope
   is unmodifiable by any layer). A role reads project augmentation as a distinct
   labelled prompt section (008.1's composition), never merged into policy. Onboarding
   MAY propose role augmentation in its draft; it persists (with the capacity) only
   on the same explicit `OK`, and a draft carrying an authority-bearing augmentation
   field is rejected by validation.

7. **End-to-end + Proof runbook.** Hermetic: read-only inspection (tree unchanged);
   two-layer draft validation; the authority boundary (agent "OK" authorizes
   nothing); both approval gates (denied trial does not run; approved trial runs
   only via `RuntimeController` with sanitized env + bound provenance); OK-gated
   transient persist (abandon persists nothing); fingerprint canonicalization; drift
   branch A (`suspected-capacity-drift`) vs branch B (`command-failed`); a
   role-augmentation draft with an authority-bearing field REJECTED, a valid advisory
   one persisted + surfaced as a labelled prompt section; idempotent re-onboard. Plus
   the deterministic Proof runbook above (`onboard.replay` + `*.turns` fixtures + the
   Unix-socket runtime fixture).

## Decisions

- **Onboarding is a separate epic** (debate 2026-07-18/19; Ulrich 2026-07-19) — an
  agent-driven control plane (agent sessions, read-only inspection, NL revision,
  approval UX, provenance, drift) distinct from runtime EXECUTION (008.3). It
  DEPENDS on 008.3's `RuntimeController` for trials; a dependency is not a merge
  reason.
- **Two distinct approval gates** (debate B5) — durable per-project runtime-trust
  (008.3) and per-trial user approval (008.4); neither substitutes for the other.
  Reverses the earlier "unified, not duplicated" claim.
- **Authority flows only from the user channel** — agent output never authorizes
  `OK`/approval (debate B12).
- **Drift is `suspected-capacity-drift`, correlation-honest, in
  `project capacity check` only** — no `tdd@1`/witness change (debate B6/B7/S7).
- **Session state is transient; only `OK` persists** (debate B8) — no durable
  abandoned-session record.
- **Exact migration split** (debate B9) — 008.2 owns capacity + `confirmedAt` +
  `evidenceFingerprint`; 008.4 adds `evidence` + `runtimeTrial`.
- **Read-only = tool-set + workspace-root confinement, NOT a confidentiality claim**
  (debate S9) — a clean tree proves non-mutation; it does NOT prove the agent read
  no secrets. Secret confidentiality + OS containment are EPIC 009.
- **Fingerprint canonicalization is specified** (debate S6), not implementation-
  defined.
- **The project role-augmentation port lands here, with onboarding as its consumer**
  (debate 2026-07-19 B3) — NOT an empty seam in 008.1. It carries advisory,
  provenance-labelled, project-scoped role CONTEXT only; authority-bearing fields
  (tools / arguments / lanes / bindings / verdict) are rejected — 008.1's executable
  authority envelope stays unmodifiable.

## Non-goals

- **No new executor / cycle / confirmation / witness behavior** — onboarding only
  PRODUCES a `ProjectCapacity` consumed by 008.2/008.3, and its drift check is a
  standalone command; it changes no `tdd@1` transition or witness classification.
- **No runtime EXECUTION machinery** — `RuntimeController`, lease, dirty markers,
  the host guard all belong to 008.3; onboarding only INVOKES them for a trial.
- **No auto-onboarding / no background drift polling** — user-invoked; drift
  surfaces at `project capacity check` time, not on a timer.
- **No confidentiality / secret-read prevention / OS sandboxing / secret-manager** —
  EPIC 009; onboarding's read-only claim is tool-set + workspace confinement only.
- **No multi-repo / cross-project / per-branch capacity** — one confirmed capacity
  per Project this epic.
