# 019.6 Daemon Provider Wiring

## Outcome

The live `kanthord run` daemon spawns real pi agent sessions backed by a
logged-in provider account. Epic 019.4 built the provider engine
(`buildProviderSession` ŌĆö account + model ŌåÆ `{ model, streamFn }`) and a runnable
`kanthord login`, but nothing in the daemon dispatch path consumes them: today
`run-loop` spawns sessions with `model`/`streamFn` undefined, so the real model
call falls through to the `getApiKey` path and cannot run. This epic adds the one
missing wire ŌĆö resolve the accountŌåÆsession at boot and inject it as the default
`model`/`streamFn` for every spawned session ŌĆö so LP-A1ŌĆ”LP-A3 can make a real
call. It adds no provider logic; it assembles the 019.4 core into the run path.

## Stories

- `001-wire-provider-session-into-dispatch.md` - resolve the logged-in account to
  a `{ model, streamFn }` session at boot and make it the default for every
  daemon-spawned pi session; fail closed with a `kanthord login` hint when no
  account resolves.
- `002-ensure-data-root.md` - `KANTHORD_DATA` defaults to `~/.kanthord`; `login`
  and `run` resolve-then-ensure the data root (mkdir `0700`) at start so a fresh
  machine's first login persists instead of silently failing on `ENOENT`.
