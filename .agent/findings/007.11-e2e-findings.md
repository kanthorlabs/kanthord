# EPIC 007.11 — E2E findings (real-model TODO-API build, run `e2e-0711`)

> Recorded 2026-07-22 from a full end-to-end feature build: the 5-endpoint TODO
> API driven from an empty base through the kanthord CLI with
> `openai-codex / gpt-5.6-terra` (effort medium), against the throwaway repo
> `kanthorlabs/kanthord-verify`. This run was the first to exercise the 007.10
> CLI-observability fixes on a real model.

## Outcome — the feature builds and works, but delivery is incomplete

All 5 tasks reached `completed` with `candidate:landed`; the built server passes
the full CRUD smoke test (POST 201, GET-list 200, GET-by-id 200, PUT 200,
unknown 404, DELETE 204, get-after-delete 404 → `TODO API SMOKE OK`). Base
progression on the **local** checkout's `main`:
`6d0b63f → 592d8dff → 0e784fc3 → 2aceb5c7 → eb2f258c → 468e5072`.

What worked cleanly (regression-checked against 007.9 / 007.10):

- **007.10 Story A** — `get task` exposes `landingCandidate {state, baseSHA,
candidateSHA, target}` in both human and `--json`. The entire
  build→approve→retry→land loop was navigable **from the CLI alone** — no
  `e2e-status.sh` SQL needed to read candidate state (though the helper was
  still used for the at-a-glance board).
- **007.10 Story B** — `retry task --rebuild` requeued every stale-base sibling
  directly from `awaiting_confirmation`; the old doomed-`approve` dance was
  never needed.
- **007.10 Story C** — `list event --json` returned one parseable
  `{events,nextCursor}` envelope.
- **007.10 Story D2** — `login provider` printed `credential created: <id>`.
- **007.9** — conflict detection fired on a stale approve; `get conflict`
  rendered the `git merge-tree` result with conflict markers (both sides);
  `retry --note` + rebuild recovered.
- OAuth login to kanthord's own `openai-codex` store, isolated from the company
  Copilot pi CLI; catalog listed `gpt-5.6-terra`.

Two **major** delivery defects and several minor issues surfaced. Grouped by
root cause below.

## Group 1 — Landing does not complete delivery

Root cause: "landing" is implemented as a **local ref update only**
(`git update-ref refs/heads/<target> <candidateOID>` under CAS, `src/landing/
git.ts:364-377`). There is **no `git push`** anywhere in `src/landing`,
`src/workspace`, or `src/scm`, and the land step does not reconcile the target
checkout's working tree/index with the new HEAD.

### F1 — landed work is never published to the remote — **major (missing capability, not a bug)**

> Classification refined by the fix-approach debate (opencode/plan): "landing"
> (a local ref update) and "publishing" (a remote push) are **distinct product
> operations**. Whether "completed" must mean "pushed" is a **delivery-contract
> decision**, not a settled correctness bug — so this is tracked as a missing
> capability (own epic, `007.13`), not folded into the `007.11` bug-fix epic.

After all 5 tasks landed, `git ls-remote` shows `refs/heads/main` still at the
original base `6d0b63f`. The local home checkout's `main` is **5 commits ahead
of `origin/main` and unpushed** (`git status`: "Your branch is ahead of
'origin/main' by 5 commits"). There is no CLI affordance to publish the landed
result — `land repository` also only does the local ref update. For the stated
goal ("develop a feature from scratch to completion"), the deliverable never
leaves the machine.

- Repro: after N/N completed, `git ls-remote <remote> refs/heads/main` → base
  SHA; `git -C .data/e2e-0711/home log --oneline -6` → 5 `kanthord: proposal
for <taskId>` commits on top of the base; `git -C … status` → "ahead by 5".
- Note: AGENTS.md says the project is "local development only", so an automatic
  push may be deliberately out of scope — but there is then **no** documented
  or CLI-exposed step that completes delivery, and the operator is left to push
  by hand from a checkout that is itself in a broken state (F2).
- Suggestion: add an explicit publish step (e.g. `land … --push`, or a
  `publish repository --repository <id> --branch <b>`) that pushes the landed
  target to the remote via the repo's stored credential (the same GIT_ASKPASS
  plumbing already used for clone), with CAS/`--force-with-lease` semantics.

### F2 — the landing-target checkout is left inconsistent with HEAD — **major**

After landing, the home checkout's HEAD (`468e5072`) **contains**
`src/todo.mjs`, but the working tree and index show it **deleted** (`git status`
→ `D  src/todo.mjs`; `cat-file -e HEAD:src/todo.mjs` = present, file on disk =
absent). A human who opens the checkout — or runs the server directly from it —
finds the just-landed file missing, and a `git status` that contradicts the
"landed" report. The land updates the ref but does not `reset`/`checkout` the
worktree to match.

- Repro: `git -C .data/e2e-0711/home rev-parse --short HEAD` → `468e507`;
  `test -f .data/e2e-0711/home/src/todo.mjs` → absent; `git -C … status
--short` → `D  src/todo.mjs`. Extracting `HEAD:src/todo.mjs` yields the
  correct 158-line file (that is what the smoke test ran against).
- Suggestion: after a successful land, reconcile the target checkout
  (`git reset --hard <target>` / `git checkout -- .` on the landed branch, under
  the existing per-repo lock) so the working tree matches HEAD, or land in a way
  that never leaves a staged deletion. At minimum, document that the checkout is
  not a source of truth and the landed ref must be re-checked-out.

## Group 2 — Rebuild efficiency (sibling serialization)

### F3 — N siblings editing one file cost N full real-model rebuilds — **major** (known cost)

The 4 read/update/delete tasks all edit `src/todo.mjs`. Because every sibling
built on the same base and each approve moved that base, the run cost **11
real-model builds for 5 endpoints** (`agent.finished` = 11): Create(1) +
first-pass siblings(4) + rebuilds(3 + 2 + 1 = 6). Six of the eleven builds were
spent re-generating textually non-overlapping edits after the base moved. This
is exactly the F3 cost 007.10 deferred, now measured on a real model.

- Repro: `list event --after 0 --limit 1000 --json` → `agent.finished: 11` for
  5 endpoints; the base advanced once per approve, re-staleing the remaining
  siblings each time.
- Suggestion (007.10 non-goal, now the primary optimization for THIS epic):
  before demanding a model rebuild of a stale-but-`pending` candidate, attempt a
  **deterministic 3-way rebase** of the candidate onto the moved base, run the
  existing verification gate, and fall back to a model rebuild **only** on
  textual conflict or failed verification. `retry --rebuild` (007.10 Story B) is
  the natural entry point; the daemon could also auto-attempt it on
  approve-time staleness.

## Group 3 — CLI output ergonomics

### F4 — `approve task` prints only a bare id, no confirmation line — **minor**

`retry` prints `task re-queued: <id>`, `create *` prints `<kind> created: <id>`,
`login provider` now prints `credential created: <id>` (007.10 D2) — but
`approve task` emits only the bare ULID on stdout with no friendly stderr line
stating what happened (landed vs conflicted). Same output-convention drift
family as 007.10 F6.

- Repro: `node src/main.ts approve task --id <id>` → single line, the bare id.
- Suggestion: add a stderr line, e.g. `task landed: <id>` on a clean land and a
  distinct line on conflict (keep the id on stdout for scripting).

### F5 — daemon `--until-idle` gives no terminal "awaiting approval" summary — **minor**

When the daemon reaches idle with tasks parked at `awaiting_confirmation`, it
simply exits; nothing tells the operator "M task(s) are awaiting your approval,
run `get task`/`approve`". Discovering the gate requires polling `get task` or
`e2e-status.sh`. Minor ergonomics gap in the otherwise CLI-drivable loop.

- Suggestion: on idle exit, print a one-line summary of tasks now waiting on a
  human (count + ids), mirroring the observability intent of 007.10.

## Note on the run harness (not a kanthord bug)

When `run daemon` was launched detached, the launching wrapper reported exit 0
while the daemon process kept running (verified live via `ps`/`pgrep`). This is
a backgrounding artifact of the test harness, not a kanthord defect — status was
tracked by polling the DB (`get task --json`) instead. Recorded so a future run
does not mis-read it as a premature-exit bug.
