## S003-T2 fix — diffProjection projected identity key (2026-07-04)

- Root cause: `diffProjection` built the shadow map and looked up live rows using the raw `rowIdentityKey`, including runtime-only fields like `plan_generation.generation` (0 in shadow, 1 in live) — so rows didn't match and were falsely reported as absent.
- Fix: compute `projectedIdentityKeys = rowIdentityKey.filter(k => !runtimeOnlySet.has(k))` and use it for both the shadow map and live lookups. For `plan_generation` this reduces the identity to `["feature_id"]` only; for all other tables (no runtime-only keys in their identity) it is a no-op.
- `rowIdentity` in divergences now also uses `projectedIdentityKeys` for consistency (no test asserts its exact shape; this is the cleaner semantic).
