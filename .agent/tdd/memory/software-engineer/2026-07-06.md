# Software Engineer Journal — 2026-07-06

## 012-002 B1(8th) — stale-lock takeover absent-file window (writer-lock.ts)

The recurring B1 blocker finally had a clear root cause identified by the TE: `unlink(lockPath)` + `open(lockPath, "wx")` creates a brief absent-file window. A racer whose `open("wx")` libuv syscall is queued and fires in that window succeeds and becomes a second holder.

**Fix**: Replace Step 3 with `writeFile(lockPath, payload)` — overwrite-in-place, never absent. The claim mutex (O_EXCL on claimPath) already ensures only one takeover racer is in Step 3. No unlink = no window.

**Lesson**: When a lock file must always be held by exactly one holder, never unlink it during recovery — overwrite it. The invariant is "exactly one holder", not "file exists then doesn't exist then exists". Keeping the file present at all times prevents any queued syscall from exploiting an absent-file gap.

## 015/003 effectful-config-optional-fail-open (B1 6th review) — fail-closed via DEFAULT_TRUSTED_EFFECTFUL

Reviewer required `trustedEffectful` to be enforced unconditionally. But T1(a) calls `loadNetworkDenialRegistry` without the arg while having `write_file: pure:false` — so a raw "required field" approach would break the existing test. Solution: `DEFAULT_TRUSTED_EFFECTFUL = { names: new Set(["write_file", ...file-ops...]) }` as the default value; signature changes from `trustedEffectful?: TrustedEffectfulConfig` to `trustedEffectful: TrustedEffectfulConfig = DEFAULT_TRUSTED_EFFECTFUL`. The guard becomes `if (e["pure"] === false && !trustedEffectful.names.has(toolName))` — unconditional. T1(a) passes because `write_file` is in the default; `arbitrary_effectful_tool` is not. Lesson: "required with a built-in default" is the TypeScript idiom for "always enforced, but callers may omit when the default is appropriate".
