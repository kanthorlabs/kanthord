## Verification Gate

Gates: `npm run verify` (typecheck + test + verify:handoff + lint + db status).
Hermetic — no network. Fakes + real-git-in-temp:

- **S1 (predict, pinned, non-mutating, returns the merged tree):** a
  `RepositoryLanding.preview(candidate, targetOID)` port method, real git in
  temp, returns `fast-forward{candidateOID}` / `mergeable{treeOID}` /
  `conflict{ files, perFile:[{ path, hunks }] }` via `merge-tree --write-tree`.
  Assert **refs, HEAD, index, and worktree unchanged** before/after (NOT
  byte-identical; debate B1). Conflict lists the file + non-empty hunks; mergeable
  returns the concrete merged tree OID; ff returns the candidate OID.
- **S2 (explain, honest labels):** CLI test — `get conflict --id <task>` prints
  the conflicting files, the `<<<<<<<`/`>>>>>>>` hunks, and honest OID labels
  (`target <branch>@<OID>` / `candidate <taskId>@<OID>`; debate S4). The 007.5
  `approve` conflict summary names the file(s), points to `get conflict`, and
  shows the `retry … --note` hint. `get conflict` binds to and prints the
  candidate + target OIDs it was computed against (debate S2). Sanitized; no raw
  stack, no temp paths.
- **S3 (guide + marker-free agent payload + snapshot):** `retry task --id <t>
--note <text>` persists the note; a unit test on the runner's prompt builder
  shows the note AND a **structured, marker-free** conflict context injected (the
  `getPriorRejection` hook generalized to `getPriorFeedback` carrying `{ note?,
conflictContext?, priorSummary? }`) — assert the injected payload contains **no**
  `<<<<<<<`/`>>>>>>>` sequence (debate B3). The retry durably **snapshots**
  `{ candidateOID, targetOID, conflictContext }` onto the recovery attempt so the
  rebuild prompt is deterministic/auditable and does not depend on a git object
  surviving GC (debate B4/B5/S1).
- **S4 (land the previewed tree via atomic CAS):** an ApproveTask + landing test —
  a predicted **conflict** returns the typed `conflict` outcome with **zero**
  mutation (assert branch OID, HEAD, index, worktree all unchanged; no
  `git merge`/`--abort` ran). A predicted **mergeable** candidate lands a commit
  whose **tree equals the previewed `treeOID`**, with parents `{target,
candidate}`, via `update-ref … <expectedOld=targetOID>`; **ff** lands via
  `update-ref` to the candidate OID. If the branch moved since preview, the CAS
  update fails and the path **re-previews** (bounded; typed `target_moved` after
  the cap) — assert it never lands onto the wrong base (debate B6/B7).
- **S5 (rebuild re-enters the gate):** a recovery-run test — the clean-base
  rebuild's candidate is itself run through `preview` before landing; a rebuild
  that now merges/ff-lands cleanly does so through the same S4 path (assert the
  recovery candidate is previewed, not landed blind).

Proof: deterministic — pure CLI + real git in temp dirs, NO model — reproduce the
conflict via the fake-agent seam + `make-todo-graph.sh` on a **relative** DB,
then drive predict → explain → guide and prove the atomic land wiring on the
fast-forward sibling. (The clean-_merge_ landed-tree==previewed-tree assertion and
the conflict-rebuild-to-`completed` path live in the S1/S4/S5 real-git unit tests
and the real-model E2E of Appendix B — the fake agent is not base-aware, per
007.5.) `set -euo pipefail`; expected failures use `if ! …; then`.
