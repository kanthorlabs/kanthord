# Implementation cycle — 019.3-task-goal-loop

Pulled from EPIC: `.agent/plan/epics/019.3-task-goal-loop.md`.

Verification gate (binding, from the EPIC's `## Verification Gate` section):
> - `npm run typecheck` exits 0; `npm test` green — new suites plus all
>   pre-existing suites (Epic 019.2 baseline), no regression.
> - **Loop end-to-end (hermetic):** a task whose fake exit gate fails twice
>   then passes reaches `complete` in three attempts; each retry's spawn brief
>   contains the prior failure evidence (the `GateResult` summary); the
>   attempt ledger reads 3 dispatched attempts at completion.
> - **Termination end-to-end:** with `max_attempts: 2` and an always-failing
>   gate, the task parks after attempt 2 with an `attempts-exhausted` inbox
>   item and NO third spawn; a `needs_human` outcome parks with its own item
>   and no retry; a mid-attempt lifecycle respawn does not increment the
>   ledger (respawn-equivalence suite stays green); an aborted/errored
>   session is NOT gate-checked and routes to the lifecycle path.
> - **Operator actions:** on an `attempts-exhausted` item, `retry-once`
>   grants exactly one more attempt (next fail re-parks immediately) and
>   `re-arm` resets the counter — both recorded as interactions, asserted
>   distinct.
> - **Precedence:** with the budget ceiling breached, no retry spawn happens
>   regardless of remaining attempts (budget outranks retry); a halted task is
>   not re-dispatched by the loop.
> - **Resolution:** a task frontmatter `max_attempts` overrides the system
>   default; absent frontmatter uses 3; an invalid value produces the typed
>   diagnostic.
